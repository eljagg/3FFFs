## What changed and why

### 1. Opaque mitigated overlay (the bug fix)

**Before:** `mitigatedOverlay` had `background: rgba(0, 0, 0, 0.04)` — effectively transparent. The card itself had `opacity: 0.55` and a `grayscale(0.3)` filter, but the underlying label and MITRE badge remained legible *through* the overlay. The shield icon and "MITIGATED" text overlapped the live label text, so the effect read as "card is broken" rather than "card is disabled."

**After:** the overlay uses `background: var(--success-bg)` — fully opaque against the success color token. The shield and "MITIGATED" label sit centered and are the only thing visible on a mitigated card. The card border switches to a solid success-color stroke so the boundary reads cleanly even when the overlay is present.

Padding reorganised: card padding moved from the outer `<button>` to an inner `nodeContent` div, so the overlay can sit at the card edge (zero inset) without being pushed inward by the padding. This is what made the v25.7.0.2 overlay look like a small label *inside* the card rather than a full-card cover.

### 2. Framer-motion on state transitions

The dependency was already declared. Five places now use it:

- **Mitigated overlay** — fades + scales in/out on toggle (220ms). Mitigation now visibly *happens* rather than snapping.
- **Defense-active badge** ("2 defenses active") — fades + scales on first/last toggle (180ms).
- **Defense hint expansion** — height + opacity (220ms) when the first hint appears, smoothly collapses on last toggle off.
- **Selected node description** — height + opacity (200ms) on click rather than a hard show/hide.
- **Hover** — un-mitigated cards lift 1px with a soft shadow on hover (CSS transition, 180ms). Subtle, but the page now feels like it has body.

Total motion budget is around 200ms per interaction — fast enough not to feel sluggish, slow enough to register as motion. No animations stack.

### 3. Visible SVG edges between connected nodes

This was deferred from v25.7.0.2 (planned for v25.7.1) but the screenshot evidence made it the most-needed polish item. Without visible edges, a "kill chain" reads as three disconnected columns of cards — the user has to *infer* that the columns relate to each other.

Implementation:

- The column grid is wrapped in a `position: relative` container. An absolutely-positioned SVG sits behind the nodes (zIndex 0; nodes are zIndex 1) with `pointer-events: none` so it never intercepts clicks.
- Node refs are collected into a ref object keyed by `node.id`. After mount, `useLayoutEffect` reads each node's `getBoundingClientRect`, computes container-relative coordinates for the source's right-edge midpoint and the target's left-edge midpoint, and stores the edge geometry.
- Recomputation triggers on three events: (1) initial mount, (2) `ResizeObserver` firing on the container (covers window resize and accordion expand/collapse), (3) the React effect deps changing (`edges`, `nodes`, `mitigatedNodeIds`, `activeNodeId` — anything that could shift node positions or change visual state).
- Edges render as cubic Bezier curves with horizontal control points pulled halfway across the gap. Smooth S-curves, never crossing through node bodies.
- Edge state model:
  - **Default** — `--rule-strong` stroke at 0.42 opacity, 1px wide
  - **Highlighted** (both endpoints in the active selection's traced path) — `--accent` stroke at 0.95 opacity, 2px wide
  - **Mitigated** (either endpoint shielded) — dashed (`stroke-dasharray: 4 4`) at 0.18 opacity. Reads as "this attack vector is broken."
  - **Dimmed** (selection active but this edge isn't part of the path) — 0.18 opacity

The grid gap was widened from 10px to 18px to give edges room to land cleanly without ending right at the node border.

### 4. Teller affordance — three-state body

`Framework.jsx` previously used a two-state model:
- *No viz at all* → single-column layout
- *Viz visible to current role* → two-column with viz panel

This conflated "no viz attached" with "viz attached but excluded for your role." Both rendered identically. From the user's perspective, a teller landing on TA0043 saw no signal that anything was different — same as TA0042 (where no viz exists at all).

New three-state model:

- **State 1** (no viz attached anywhere) — single-column layout, unchanged. No empty right pane.
- **State 2** (viz attached but NONE for current role) — two-column with an "Interactive content for other roles" affordance panel on the right. Lists which roles WOULD see something, and notes that a role-tuned version is on the v25.7.1 backlog. Dashed border on the affordance panel signals "placeholder, not real content."
- **State 3** (viz visible to current role) — two-column with the rendered viz, identical to v25.7.0.2 behaviour.

The affordance is the OBS-018 four-lens discipline made *visible to the user*. The deploy notes have always documented "no change for the other three lenses"; now the user can see that documentation in the UI. This is what we should have shipped originally.

## Honest reflections

**What worked here:** the diagnosis was clean. Screenshot evidence showed exactly what was wrong (transparent overlay, no edges) and OBS-016 ("read the body before assuming") meant we didn't escalate by rebuilding the viz. The fix is two files.

**What didn't ship cleanly the first time:** the v25.7.0.2 overlay bug should have been caught in my visual review before delivery. I tested that the overlay rendered, but didn't test that it actually obscured the underlying content. Adding "screenshot or describe each interactive state" to my pre-delivery checklist for any new viz kind. Not formal enough to be an OBS yet — closer to a personal practice — but worth stating.

**What I'm watching:** the SVG edge rendering uses `getBoundingClientRect` reads in a `useLayoutEffect`, which means there's a brief moment between first paint and second paint where edges may not be present. With ResizeObserver wired up the second paint is fast (sub-frame in practice), but on slow devices users could perceive a flicker. If that turns out to be a real issue we'll cache the geometry in a ref and only re-render the SVG when geometry actually changes — minor optimization, not worth doing speculatively.
