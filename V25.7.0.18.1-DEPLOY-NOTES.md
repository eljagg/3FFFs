## Pedagogical insight (OBS-027 — methodological)

**Front-end animation wiring is not the same as framework taxonomy
wiring.** The 3FFFs F3 framework grid is populated from the Neo4j
graph database via API. Adding a technique to ANIMATION_MAP only
controls what happens AFTER the user clicks an existing technique
card — it doesn't make the card appear. For new techniques to be
discoverable, they need a corresponding Technique node in the
graph, linked to a Tactic via :PART_OF.

For Phase 1 going forward, the methodology is:
1. **Verify the technique ID is in the F3 grid** before authoring
   the animation. Open the framework page; check that the technique
   card exists.
2. **If absent, write the additive seed migration FIRST** (using
   `add-t1621-technique.js` as template).
3. Then author the animation; ANIMATION_MAP entry connects the two.

This avoids the v25.7.0.18 → v25.7.0.18.1 split. The next two
animations (F1072 visible in grid; F1018/T1111 to verify) will be
checked before authoring.

## Diagnosis (what went wrong with v25.7.0.18)

The framework grid is rendered by `TechniquesTree.jsx`, which calls
`/api/framework/tactics/:tacticId/techniques-tree`. That API runs:

```cypher
MATCH (tech:Technique)-[:PART_OF]->(:Tactic {id: $tacticId})
RETURN tech.id, tech.name, tech.description, tech.roles, ...
```

For T1621 to appear, a Technique node {id: 'T1621'} must exist with
a :PART_OF relationship to a Tactic. v25.7.0.18 didn't create either.

T1621 is referenced in `sc014.js` as `mitreTechnique: 'T1621'` (a
different field name used by AASE/CBEST scenarios), but the F3
framework reads from `Technique` nodes — not `MitreTechnique`
references. The two paths don't connect.

By contrast, F1018.001 (which DOES appear in the grid) is referenced
in `caribbean-scenarios.js` line 310 as `techniqueId: 'F1018.001'`.
The base seed processes that field and creates the corresponding
Technique node. T1621 didn't get the same treatment because no
caribbean-scenarios.js entry references it.

## What ships in this release

### NEW: server-side migration script

`server/src/seed/add-t1621-technique.js` — 142 lines, idempotent.
Modeled on `add-caribbean.js` pattern.

What it does:
1. Verifies the Tactic TA0001 (Initial Access) exists in the graph
2. Reports whether T1621 already exists (so re-runs are clearly
   informational)
3. MERGEs a Technique node `{id: 'T1621', name: 'MFA Request Generation',
   description: ..., roles: ['teller', 'analyst', 'soc', 'executive']}`
4. MERGEs a `:PART_OF` relationship to TA0001
5. Verifies the API query that powers the framework grid will now
   return T1621
6. Reports success/failure clearly

What it does NOT do:
- Doesn't delete or modify any existing Technique nodes
- Doesn't create scenarios that reference T1621 (the v25.7.0.18
  animation provides the trainee experience)
- Doesn't touch any User, COMPLETED, ATTEMPTED_STAGE, or ANSWERED
  data

## Deploy

This release has TWO deploy steps because it includes a server-side
migration in addition to the front-end deploy:

### Step 1 — frontend deploy (same as usual)

```bash
cd /workspaces/3FFFs
unzip -o v25.7.0.18.1-3FFFs.zip
rm v25.7.0.18.1-3FFFs.zip
git add -A
git status
git commit -m "v25.7.0.18.1: additive migration to surface T1621 in F3 framework grid"

```

Wait for Railway to redeploy.

### Step 2 — run the server-side migration ONCE

Open Railway dashboard, find the `server` service, and use the
"Run a command" feature (or use `railway run --service server`
if you have the Railway CLI installed locally with the project
linked).

Run:

```bash
node src/seed/add-t1621-technique.js
```

You should see output like:

```
🛡️   v25.7.0.18.1: Adding T1621 (MFA Request Generation) as F3 framework technique...

✓   Tactic TA0001 (Initial Access) confirmed in graph
ℹ️   Technique T1621 not present — will be created.
✅  MERGEd Technique T1621 ("MFA Request Generation")
✅  Linked T1621 -[:PART_OF]-> TA0001

✅  Verification passed: T1621 will appear in the F3 framework grid under TA0001 (Initial Access).
    Hard-refresh the browser on the Framework page to see the new card.
    Card name: "MFA Request Generation"
    Roles: teller, analyst, soc, executive

🎉  v25.7.0.18.1 migration complete.
```

If you see `❌  Tactic TA0001 not found in graph` — something is
seriously wrong with the base seed; do not proceed and tell me.

If you see `❌  Verification failed` — the MERGE succeeded but the
PART_OF link didn't form correctly; tell me and I'll diagnose.

The script is idempotent — safe to re-run if anything looks weird.
