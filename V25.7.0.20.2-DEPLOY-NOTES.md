## Pedagogical insight (OBS-027 — methodological)

**Two design assumptions, two same-shape bugs.** Both fixes in this
release follow the same pattern: a code path was written to be
correct under v25.7.0.8-era assumptions that no longer hold, and
v25.7.0.20's introduction of F1018-as-its-own-thing exposed the gap.

The grid card's "expandable parents are toggle-only" assumption
held when expandable parents had no animations of their own.
F1018's animation broke that assumption — the parent now needs
both expand and open-detail navigation paths.

The sidebar's `setInternalTechId(newId)` assumption that "newId is
always different from the previous internalTechId" held when users
navigated linearly through the technique graph. The F1018 link in
F1018.001's sidebar exposed the gap when users navigated to F1018,
closed the sidebar, re-opened F1018.001 from the grid, and clicked
the F1018 link again — same value, no state change, no re-fetch.

The methodological lesson: **state transitions that "always
change a value" should be paired with a counter or timestamp** so
the consumer can react to the act of navigation, not just to the
new value. This is a standard React pattern (sometimes called a
"version" or "tick"). Locked into the methodology going forward.

## Diagnosis

### Finding 1 — F1018 parent card click does nothing

`TechniquesTree.jsx` line 174-180 (pre-fix):

```js
const handleCardClick = () => {
  if (expandable) {
    onToggle()             // expand-only
  } else {
    onTechniqueClick(effectiveTargetId)
  }
}
```

After v25.7.0.20.1 made F1018 expandable, the only thing the F1018
card click did was toggle expansion. The `onTechniqueClick` path
was unreachable for expandable parents. F1018's animation existed
in `ANIMATION_MAP['F1018']` but no UI affordance opened it.

### Finding 2 — F1018 parent-link in sub-tech sidebar inconsistent on repeat

`TechniqueDetailSidebar.jsx` `navigateInternal()` (pre-fix):

```js
function navigateInternal(newId) {
  setInternalTechId(newId)
}
```

React skips state updates when `newId === internalTechId` (reference
equality on strings). The fetch `useEffect([open, internalTechId])`
doesn't re-fire because the dependency didn't change. The user
clicked the F1018 link, nothing happened, sidebar stayed on
F1018.001.

The pattern was reproducible because the prior interaction (open
F1018.001 → click F1018 link → close sidebar) left `internalTechId`
at `'F1018'`. Re-opening F1018.001 from the grid reset
`internalTechId` to null via the `[techniqueId]`-dependency reset
effect, but if the prop value `techniqueId` was already
`'F1018.001'` from before, the reset effect didn't fire either.
So `internalTechId` could remain stuck at `'F1018'` across closes
and re-opens.

## The fix

### Fix A: split affordance for expandable parents with their own content

`TechniquesTree.jsx`:

1. Convert the parent card from `<button>` to `<div role="button">`.
   This is required because the next change adds a real `<button>`
   inside, and button-inside-button is invalid HTML.
2. Add a secondary `<button type="button">` inside the expand hint
   row, labeled "Open <id> detail". This button stops event
   propagation so its click doesn't also fire the card's expand
   toggle.
3. Style the inner button as a subtle bordered chip aligned right
   in the expand hint row.

Result: F1018 card shows the expand chevron AND a separate
"Open F1018 detail" button. Click chevron area → expand reveals
F1018.001 chip. Click "Open F1018 detail" button → opens F1018's
sidebar. Click F1018.001 chip after expansion → opens F1018.001's
sidebar. All three navigations available.

For atomic cards with no sub-techniques (no expand affordance),
the main card click still opens their sidebar directly — no
secondary button needed.

### Fix B: counter pattern for sidebar internal navigation

`TechniqueDetailSidebar.jsx`:

1. Add a paired `internalNavTick` counter alongside `internalTechId`.
2. `navigateInternal(newId)` calls both `setInternalTechId(newId)` and
   `setInternalNavTick(t => t + 1)`.
3. Re-fetch effect dependency becomes `[open, internalTechId,
   internalNavTick]`. The tick guarantees the dependency changes
   even when `newId === internalTechId`, forcing the effect to
   re-run.
4. Reset effect on `[techniqueId]` also resets the tick to 0 so
   subsequent internal navs start from a clean state.

Result: clicking the F1018 parent-link inside F1018.001's sidebar
always re-fetches and re-renders, regardless of whether F1018 was
the previous internalTechId.

## Honest accounting

Three notes:

1. **Two routing bugs from one v25.7.0.20 design choice.** Adding
   F1018's animation triggered both. The first hotfix
   (v25.7.0.20.1) addressed surfacing-as-separate-cards. This
   hotfix addresses the click-routing gaps that the surfacing
   exposed. Locked methodology going forward: when adding an
   animation to a parent canonical that previously had no
   animation, audit BOTH grid card click handling AND sidebar
   internal navigation handling for the same-value-state-update
   class of bug.

2. **The counter pattern is a small but durable fix.** It applies
   beyond this specific case — any time React state is used as
   an effect dependency and the value space might collapse to a
   small set of repeating values, the counter pattern guarantees
   re-fire semantics. Worth documenting separately if a future
   release introduces another path that hits the same shape.

3. **No content changes in this release.** All animations, scene
   files, audio, and seed migrations from v25.7.0.20 + .20.1
   carry forward unchanged. Pure UX routing fixes.
