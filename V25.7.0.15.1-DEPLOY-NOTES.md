# v25.7.0.15.1 — Audio engine bug fixes (no scene changes; engines only)

Diagnostic-driven hotfix. After v25.7.0.15 deployed and Omar tested,
two bugs surfaced:

1. **IVR Discovery (F1073) — no audio at all.**
   Audio fields were correctly authored on stages 2-5, but no speech
   was reaching the trainee.

2. **Gather Victim Information (F1067) — partial audio "in parts".**
   This animation has zero audio fields, so this finding was
   surprising — there shouldn't be any audio to play.

This release diagnoses both as a single root cause (effect-dependency
churn) plus one related issue (leftover speech queue across animation
navigations) and adds a third fix for an unrelated over-aggressive
audio-derivation behavior.

## Pedagogical insight (OBS-027)

**A bug that changes what the trainee hears is a teaching error,
not just a UX bug.** The IVR audio failure meant trainees never heard
the actual IVR menu prompts the attacker heard during reconnaissance —
which is the entire pedagogical point of audio in IVR Discovery. The
phantom OSINT audio was worse: it played fragments of dialogue from
the previous animation (Vishing, most likely) at a context where it
made no sense, potentially disorienting the trainee.

For a training app, audio fidelity belongs in the same correctness
category as the visual diagram. The fix here is a correctness fix,
not just polish.

## Diagnosis

**Root cause #1 (the IVR bug):** the audio side-effect dependency
arrays in ProcessAnimation and MultiActorSequenceAnimation included
`currentStage` (the destructured `stages[currentStageIdx]` reference).
The `stages` array is destructured from `scenes` at the top of every
render. Even if `scenes` itself is referentially stable across renders,
React identity rules make `currentStage` a "new" reference each time
the component re-renders for any reason (stage change, control toggle,
playback speed change, parent re-render).

Each re-render fired the audio effect. The effect's structure was:
1. Cancel any in-flight speech (`stopAllNarration()`)
2. Schedule a new `setTimeout(speakMessage, 350)`

When renders happened faster than 350ms apart — which is normal during
an active animation — the cancel kept firing before the speak ever
landed. Net result: cancellation chain that never produced audio.

ProcessAnimation showed this most clearly because its 350ms delay is
short. MultiActorSequenceAnimation had a longer slot-distribution
schedule that occasionally got partial audio through (which is why
Vishing produced audible speech in v25.7.0.15 testing).

**Root cause #2 (the OSINT phantom audio):** when the trainee navigates
between technique animations, the parent sidebar renders a new engine
component for the new technique. But `window.speechSynthesis` is a
global queue — utterances queued by the previous animation persist
across the navigation. The new engine's mount didn't clear the global
queue, so speech queued during Vishing playback could still be
playing when the trainee opened OSINT.

The cancellation chain from root cause #1 made this worse: each render
in OSINT triggered a `stopAllNarration()` call that interrupted the
leftover speech mid-word, producing the "in parts" symptom.

**Root cause #3 (over-eager auto-narration):** the `deriveAudioFromMessage`
helper was auto-narrating any message of kind 'sms' / 'callback' /
'voice' that lacked an explicit audio field, using the message's
diagram label as the spoken text. SIM Swap stage 5/6 has system-routed
SMS messages with labels like "SMS delivered · to ported SIM" — these
are diagram-annotation language, not actual SMS content. Auto-
narrating these would confuse rather than teach.

## What changed and why

### Engines (3 files modified)

**ProcessAnimation.jsx**
- Audio side-effect dependency array now omits `currentStage`. Reads
  `stages[currentStageIdx]` inside the effect rather than depending
  on the reference. Effect fires only when stage actually changes.
- Added a mount-only `useEffect` that calls `stopAllNarration()` once
  when the engine first mounts, clearing any leftover speech queue
  from a previous animation.

**MultiActorSequenceAnimation.jsx**
- Same dependency-array fix as ProcessAnimation.
- Same mount-cleanup `useEffect`.
- Removed the `deriveAudioFromMessage` auto-narration. Now requires
  explicit `audio: { text, profile }` per message. Affects SIM Swap
  stage 5/6 SMS messages that previously would have spoken
  diagram-label text — now silent (correct behavior).

**TimelineThresholdAnimation.jsx**
- Already had the correct dependency pattern (read inside the effect).
  Just added the mount-cleanup `useEffect` for consistency.
- No behavior change for current Structuring scene (no audio fields
  exist there) — but defensive against any future case-review animation
  that adds audio.

### Scene files

**No changes.** All scene files (vishing, phishing, simSwap, ivrDiscovery,
osint, structuring) are byte-identical to v25.7.0.15.

## Honest accounting

Three notes:

1. **Auto-narration removal is a pedagogical improvement.** The
   over-eager auto-narration was speaking diagram-language rather
   than actual incident-language. This release narrows the audio
   surface to exactly what was intentionally authored. If a future
   scene wants additional audio, the author adds an explicit `audio`
   field — no surprise narrations from kind-based auto-derivation.

2. **The mount-cleanup pattern is defensive against a real React
   nuance.** Same engine component class can be reused by React's
   reconciler when navigating between techniques (same engine, new
   scenes prop). In that case unmount/cleanup doesn't fire, and global
   browser state (like speechSynthesis queue) persists. The
   mount-cleanup `useEffect` runs once per engine instance — but
   since the global state is shared, it correctly handles the
   reconciliation-reuse case.

3. **My v25.7.0.15 testing didn't catch the IVR bug because** I only
   tested Vishing in detail. ProcessAnimation's shorter setTimeout
   delay (350ms) made the dep-array bug fully fatal there;
   MultiActorSequenceAnimation's longer slot-distribution schedule
   produced *some* audio so the bug looked like a quality issue
   rather than a structural one. Future audio releases should test
   IVR Discovery as the canary — its short single-stage audio is
   the most stress-testing for the dep-array correctness.

This release is engines-only — minimal surface area, fast to verify,
no pedagogical risk. Phase 1 of comprehensive coverage continues with
v25.7.0.16 (Account Takeover: Password Reset, F1018.001) once you
confirm IVR audio works.
