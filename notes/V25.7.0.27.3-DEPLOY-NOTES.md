## Pedagogical insight (OBS-027 — methodological)

**Animation releases are scene-file-plus-map-entry releases. Period.**

The flow that worked across v25.7.0.12 through v25.7.0.21
(eight Phase 1 animations, no layout regressions, no sidebar
churn) was: one new scene file, one ANIMATION_MAP entry, one
audio version bump, one set of deploy notes. The slot for new
animations is the technique sidebar — `TechniqueDetailSidebar`
already handles every interaction concern (slide-out, backdrop,
Section components, role chips, animation engine routing,
cross-references). Adding to ANIMATION_MAP gets free interior
treatment.

The mistake in v25.7.0.27.1 was scope creep. Silent Alarm
(F1008.001) shipped correctly in v25.7.0.27 following the
established recipe. The follow-up hotfixes broke discipline by
treating "tactic-level visualizations don't slide out the same
way technique animations do" as a problem to solve as part of
animation work. It isn't. The tactic-page rendering is its own
surface, with its own design history (v25.7.0.2.2, v25.7.0.3,
v25.7.0.4), and any change to it is a separate scoped release
requiring explicit greenlight before authoring.

Generalized: when something works, the cost of "while we're
here, let's also unify..." is paid in the next several releases'
worth of regressions. The right answer to "this could be more
consistent" inside an animation release is "noted; let's queue
a UX-consistency release after Phase 2 closes if we still want
it." Not "let me also touch Framework.jsx."

This release re-anchors the methodology. v25.7.0.28 onwards
follows the recipe exactly.
