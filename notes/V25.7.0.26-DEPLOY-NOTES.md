## Pedagogical insight (OBS-027)

**Bulk import is per-row UX, not bulk UX.**

The instinct when designing a "bulk" surface is to treat it as one
atomic operation — "import 500 users" — with one success/failure
outcome. That model is wrong for two compounding reasons. First, real
spreadsheets always have at least one bad row (typo, missing column,
stale data), so an all-or-nothing model produces the worst UX of "we
imported nothing because of one row, please fix and re-upload."
Second, the admin's mental model is *not* "I'm importing a batch" —
it's "I'm creating 500 individual invites at once because typing
them one at a time is slow." The 500 invites stay independent
operations in the admin's head, and the system should match that.

So the design decisions follow:

- **Per-row independence at every layer.** Server processes rows in
  a sequential loop (not a Cypher UNWIND that would be transactionally
  atomic). One bad row doesn't block the other 499. Per-row results
  returned, including the specific error for each failure.
- **Preview before commit.** Client-side parsing and validation
  happens before any server call. Admin sees exactly what they're
  about to commit, with bad rows visibly flagged. Submit only sends
  the valid rows. The error rows are downloadable as CSV after the
  fact for fix-and-reupload, which preserves the admin's investment
  in the original file.
- **Partial-success is a first-class outcome.** The results screen
  shows total / created / updated / errors as four separate stat
  boxes, not a binary success/fail. "299 created, 1 error" is treated
  as a real outcome, not a degraded version of "300 created."

The tradeoff is throughput. A Cypher UNWIND would be ~5x faster on
the server side. At admin scale (tens to hundreds of rows per
upload, not thousands), the throughput hit doesn't matter; the per-
row reporting integrity does. If a future use case actually pushes
this into the "thousands of rows" regime, revisit — but not before.

The principle generalizes: any "bulk" feature where the items are
independent should preserve that independence in the implementation.
The only legitimate reason to make a batch atomic is when the items
are causally entangled — and individual users in an invite list
aren't.

## Four-lens block (OBS-018)

- **Teller:** Not user-facing. No teller workflow change.

- **Analyst:** Bulk import doesn't touch fraud-pattern data, but it
  does change how cohorts get onboarded. When fraud ops wants to
  enroll the entire NCB AML team in 3FFFs ahead of an audit
  reporting deadline, the difference between manually inviting 50
  people and uploading a CSV is hours of work — and the per-row
  reporting means an analyst can confirm "all 50 enrolled" or
  "47 enrolled, 3 had bad email addresses" with one screen, not 50
  separate confirmations.

- **SOC:** New endpoint sits behind the existing
  `requireRole('admin')` guard. Same blast radius as POST /invites
  per row — invites can grant trainee/manager/admin to
  off-allowlist emails, which is a privileged operation, but the
  per-row and overall caps (500 per upload) limit damage from
  either an admin mistake or a compromised admin account. Audit
  trail captures `invitedBy` per row, so post-incident review can
  identify which admin uploaded a given malicious invite.
  Client-side parsing avoids any unparsed file content reaching
  the server (XLSX/CSV files can carry macros or formulas; we
  read values, never execute them via SheetJS's read-only mode).

- **Executive:** Closes the last operational gap on the user
  management surface. Combined with extend/restore (.22), drill-
  down (.23), edit (.24), role management (.25), the admin surface
  is feature-complete for the reference-customer phase. Three of
  the five releases (.22, .23, .26) directly remove friction from
  scenarios a CRO would expect to "just work" during a 30-second
  product walkthrough. Forward path is now Phase 2 animation work,
  which is the curriculum-side argument for why customers should
  pay for 3FFFs in the first place.

## What ships in this release

### Server — `server/src/routes/admin.js`

New endpoint: **`POST /api/admin/invites/bulk`**

```
Body: { invites: [{ email, roles, notes?, expirationHours? }, ...] }
Response: {
  results: [{ email, status: 'created' | 'updated' | 'error', error?, invite? }],
  summary: { total, created, updated, errors }
}
400 if body has no `invites` array, empty array, or > 500 items
Per-row failures captured in results, not batch-fatal
```

Implementation: sequential loop over rows. Each row goes through the
same validation as `POST /api/admin/invites` (email format, valid
roles, expirationHours type), then a MERGE-by-email upsert with the
exact same Cypher pattern as the single-row endpoint (preserves
`invitedAt` on re-invite, clears `revokedAt`, refreshes mutable
fields). Per-row try/catch captures any DB errors without affecting
the rest of the batch. Idempotent — re-uploading the same file
produces "updated" rows for emails that already exist, not duplicates.

500-row cap enforced at the start. If you need to bulk-import more
than that, split the file. The cap is conservative; raising it later
is one constant.

### Client — `client/package.json`

New dependency: `xlsx@^0.18.5` (SheetJS Community Edition).
Handles both .xlsx parsing and .csv parsing through the same API.
Vite tree-shakes the parts we don't use; the production bundle
adds approximately 600KB pre-gzip when it loads. Dynamic-imported
in the modal so the bundle only loads on first parse — admins who
don't use bulk import never download it.

### Client — `client/src/lib/api.js`

New: `adminBulkInvite(invites)` → POST `/api/admin/invites/bulk`

### Client — `client/src/pages/AdminUsers.jsx`

`InvitesTab`:
- New state: `showBulkModal`
- New button next to `+ Invite user`: **`+ Bulk import…`**
- New `<BulkInviteModal>` mount with onComplete that closes the
  modal AND triggers `load()` to refresh the invites list

New component: **`BulkInviteModal`** (~330 lines including helpers)
- Four-phase state machine: `upload` → `preview` → `submitting` →
  `results`. Each phase is a distinct render with phase-appropriate
  UI; no surprise transitions.
- **Upload phase**: drag-and-drop area + click-to-select file input.
  Accepts .xlsx, .xls, .csv. "Download CSV template" button next to
  the upload area generates a sample CSV with headers + example rows
  inline (no server round-trip).
- **Preview phase**: scrollable table of all parsed rows (max 380px
  height, sticky header). Each row shows raw values from the file
  + a Status column with ✓ ready or ✗ {specific errors}. Error rows
  have a faint red row tint for visual scan. Footer shows valid /
  error counts and the Submit button (disabled if zero valid rows,
  labeled with the specific count).
- **Submitting phase**: simple "Creating N invites… Don't close this
  window" message. ESC dismissal disabled during this phase to avoid
  mid-flight cancellation.
- **Results phase**: 4-stat box grid (total / created / updated /
  errors) + per-row table + "Download error rows as CSV" button
  (only shown if errors exist) + "Upload another file" /
  "Done" buttons. The Done button calls `onComplete` which refreshes
  the parent invites list.

Helpers:
- `BulkPreviewBody`, `BulkResultsBody` — extracted child components,
  one per phase
- `csvCell(v)` — RFC 4180 escaping for the error-CSV download
- `triggerDownload(blob, filename)` — clean blob-URL download
  pattern with revokeObjectURL cleanup

Validation logic (client-side, mirrors server exactly):
- Email: required, regex `^[^@\s]+@[^@\s]+\.[^@\s]+$`
- Roles: required, must be subset of {trainee, manager, admin},
  comma-separated allowed (e.g. `manager,admin`)
- Notes: optional, max 500 chars (silently truncated)
- expirationHours: optional, must be a positive number if present
  (blank means use the default per the server endpoint)
- In-file duplicates: first occurrence wins, subsequent occurrences
  flagged with "Duplicate of an earlier row in this file" error

File parsing notes:
- Header row matched case-insensitively (`Email`/`email`/`EMAIL` all
  work). Required: `email`, `roles`. Optional: `notes`,
  `expirationhours` (no separator).
- Empty rows skipped (Excel often pads).
- Empty file → clear error in upload phase, not preview.
- > 500 rows → clear error in upload phase, not preview.

## Verification — 5 minutes

### Step 1 — Download the template (15 seconds)

1. `/admin/users` → Invites tab
2. Confirm new **`+ Bulk import…`** button next to `+ Invite user`
3. Click it. Modal opens at the Upload phase.
4. Click **`↓ Download CSV template`**. Browser downloads
   `3fffs-invite-template.csv`.
5. Open the downloaded file in any spreadsheet app or text editor —
   should have 4 columns and 3 example rows.

**Pass:** template downloads with the right shape.

### Step 2 — Upload a clean test file (60 seconds)

Use the template you just downloaded as-is, OR construct a test
file with these rows (any spreadsheet works):

```
email,roles,notes,expirationHours
test1@partnerbank.com,trainee,Test bulk row 1,168
test2@partnerbank.com,trainee,Test bulk row 2,168
test3@external.com,"manager,admin",Senior reviewer,720
```

Save as either .csv or .xlsx, then:

1. In the modal, drop the file into the upload area (or click to
   browse)
2. Modal transitions to Preview phase
3. Header reads "Preview — 3 rows parsed"
4. All 3 rows show ✓ ready in the Status column
5. Footer reads "✓ 3 valid", Submit button reads "Create 3 invites"
6. Click Submit. Brief Submitting phase, then Results phase.
7. 4-stat grid: Total 3, Created 3, Updated 0, Errors 0
8. Click **Done**. Modal closes; Invites list refreshes; the 3 new
   invites appear in the table.

**Pass:** end-to-end clean import works.

### Step 3 — Idempotence (30 seconds)

Re-upload the *same* file again.

1. Preview shows the same 3 rows, all ✓ ready
2. Submit
3. Results: Total 3, Created 0, **Updated 3**, Errors 0
4. Done

**Pass:** re-upload is idempotent; rows reported as updated, not
duplicated. Verify in the Invites list — still 3 rows for those
emails (plus your existing Candice + gmail invites), not 6.

### Step 4 — Validation errors caught client-side (60 seconds)

Construct a bad file:

```
email,roles,notes,expirationHours
notanemail,trainee,,
test4@partnerbank.com,wizard,,
test5@partnerbank.com,trainee,,
test5@partnerbank.com,trainee,Duplicate test,
,trainee,,
```

Upload. Preview should show:
- Row 2 (`notanemail`): ✗ Email format invalid
- Row 3 (`test4@... wizard`): ✗ Unknown role(s): wizard
- Row 4 (`test5@... trainee`): ✓ ready
- Row 5 (`test5@... trainee`): ✗ Duplicate of an earlier row
- Row 6 (empty email): ✗ Email is required

Footer: "✓ 1 valid · ✗ 4 with errors". Submit button: "Create 1
valid invites (skip 4 errors)".

Click Submit. Results: Total 1, Created 1, Updated 0, Errors 0.

**Pass:** client-side validation correctly classified every error
type, and only the valid row was sent to the server.

### Step 5 — Server-side validation safety net (30 seconds)

The client validation should catch everything the server would, but
the server is the actual gatekeeper. Test with curl:

```bash
read -s -p "JWT: " JWT_RAW; echo
JWT="${JWT_RAW#Bearer }"; JWT="${JWT## }"

curl -s https://server-production-7882.up.railway.app/api/admin/invites/bulk \
  -X POST \
  -H "authorization: Bearer $JWT" \
  -H "content-type: application/json" \
  -d '{"invites": [{"email": "bad", "roles": ["wizard"]}]}' | python3 -m json.tool

unset JWT JWT_RAW
```

**Pass:** response includes a per-row `error` for the bad email and
the unknown role; HTTP 200 (not 400) because per-row failures don't
fail the whole call.

## Honest accounting

Non-goals — explicit so they don't silently land in scope:

- **Mode 2 (bulk role staging for allowlisted users) is not in this
  release.** Only invite creation. Allowlisted users still
  auto-provision on first login with the trainee role; promote them
  individually via the user detail modal.
- **No bulk revoke / bulk extend / bulk delete.** Revoke and extend
  remain one-at-a-time on the Invites tab. If you bulk-imported 500
  invites and then want to revoke 50 of them, today that's 50 clicks.
  Revisit if this becomes a real workflow.
- **No email delivery from bulk import.** Same as the single-row
  case — Gap 1 in the master backlog. Bulk import creates the
  `:Invite` nodes; admin still copies the signup URL and
  out-of-band emails the recipients.
- **No file format auto-detect beyond what SheetJS does.** SheetJS
  identifies .xlsx vs .csv vs .xls from header bytes; if someone
  uploads a .json or .txt file, SheetJS may either fail to parse
  or interpret it weirdly. Acceptable failure mode — clear error
  message, retry with a real spreadsheet.
- **No upload progress bar.** The submitting phase shows a static
  "Creating N invites…" message but no per-row progress. At 500
  rows max and ~10ms per row server-side, total time is
  ~5 seconds; a progress bar would be over-engineering.
- **In-memory parsing only.** SheetJS parses the entire file into
  memory client-side. For 500 rows × 4 columns this is trivial.
  If we ever need to support 50,000-row files, switch to streaming
  parse — but premature for now.
- **No "create users" path.** This release creates `:Invite` nodes,
  not `:User` nodes. The user only exists after Auth0 login consumes
  the invite. Same as the single-row endpoint.

## Credential handling reminder (carried forward from v25.7.0.25
debugging)

When pasting opaque IDs/secrets into Railway/Auth0/etc., **always
use the source's copy icon, never retype, never extract from
screenshots**. The Client ID `0` vs `O` substitution that cost us
several debugging cycles in v25.7.0.25 was a typo introduced by
me reading from a screenshot. The diagnostic endpoint shipped in
v25.7.0.25.1 catches whitespace issues but cannot catch interior
character substitutions. The copy icon is the only path that's
guaranteed byte-faithful.
