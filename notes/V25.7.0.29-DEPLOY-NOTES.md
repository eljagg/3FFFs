## Pedagogical insight (OBS-027 — locked for F1007)

**Persistence outlives credentials.**

The trainee's mental model for "compromised account" is
"someone has my password" — a credential-shaped problem that is
solved by changing the password. Account Linking re-shapes the
problem as relational: someone now has a STANDING relationship
with the bank that grants them transactional authority, and
resetting the password does nothing about the relationship. The
diagnostic protocol is not "change password" but "audit
signatories, audit linked profiles, audit external links" — three
distinct audits the trainee must learn to perform.

Each sub-technique adds a second-order insight about the specific
attack surface:

- **F1007.001 (joint-add):** the call-centre process is a
  customer-service operation, not a security-sensitive change. The
  institutional control point is out-of-band callback to the
  registered phone on file, not the inbound caller's claimed number.
- **F1007.002 (linked-profile):** the persistence lives in surfaces
  the customer doesn\'t know to inspect. Active Devices UI parity
  between primary and secondary profiles is the diagnostic protocol fix.
- **F1007.003 (external link):** the link IS the exfil channel.
  Persistence and execution are the same configuration change, which
  makes external-bank linking the single most operationally damaging
  F1007 sub-technique. Defense requires a 7-day cool-off before
  high-value transfers are permitted to a newly-linked external
  account, plus inter-institutional intelligence sharing the region
  has discussed but not deployed at scale.

The composite teaching shape: F1007 is one of two techniques where
the trainee\'s default credential-rotation hygiene is insufficient
(the other being F1008 sub-techniques like Silent Alarm). Both
require expanding the diagnostic protocol from credentials to
configuration state. Both teach the same insight at increasing levels
of attack sophistication: F1008.001 surfaces the notification path
manipulation; F1007 surfaces the persistence-by-relationship pattern.

the Cypher migration to surface the sub-technique nodes.

```bash
cd /workspaces/3FFFs
```

```
    Hard-refresh the F3 Framework page; F1007 will surface with a "3 SUB" badge.
```

## Honest accounting

Non-goals — explicit:

- **No Cypher schema changes.** The `:PART_OF` and `:SUBTECHNIQUE_OF`
  relationship types are existing in the schema. The migration only
  adds new nodes + relationships of existing types.
- **No backend route changes.** The `/api/framework/tactics/:tacticId/
  techniques-tree` endpoint already supports the sub-technique
  hierarchy (it queries SUBTECHNIQUE_OF; F1018 + F1018.001 already
  exercise this code path).
- **No animation engine changes.** All three scene files use the
  existing `MultiActorSequenceAnimation` engine via the
  `multi-actor-sequence` shape. No new engine work.
- **No scenario nodes.** The animations are driven by the scene
  files directly; explicit `:Scenario` nodes (SC013-style) for
  Renee, Tariq, Marlon are deferred. Can be added later if you
  want them surfaced on `/scenarios` independent of the F3 page.
- **No audio recordings.** Audio remains muted by default per
  v25.7.0.15.5 architectural decision; the version bump is
  bookkeeping. ElevenLabs pivot remains in backlog.
- **No tests.** Following the existing project pattern of authoring
  scene files without an associated test suite. The verification
  protocol above is the test.

## Process notes (post-mortem from earlier in the session)

The v25.7.0.27.6 → .27.6.1 import-bug post-mortem (documented in
the .27.6.1 deploy notes under "Pedagogical insight") explicitly
called out the rule: "when adding a hook to a component file, the
very first action after writing the new code is to view the import
line and confirm the hook is in the named imports."

`accountLinking*Scenes.jsx` file uses no React hooks (pure data
exports), so no React import is needed. The TechniqueDetailSidebar
modifications add three named imports for the new scene modules,
and I verified each path resolves to a real file before completing
the edit. The audio version bump touches three locations in
audioNarration.js (header comment, console.log string,
