# v25.7.0.27.5 — Hotfix: tactic-viz interactive inside slide-out

Surgical fix to `TacticVisualizationSidebar.jsx`. The .27.2 → .27.4
slide-out wrapped the `VisualizationRenderer` inside a collapsible
Section ("How this visualization works"), which works correctly for
the kill-chain grid (Reconnaissance) — its click-to-toggle interactions
don't depend on layout context — but breaks `PositioningTimeline` and
`PositioningTwoViews`, which use drag scrubbers and split-screen
toggles that require unconstrained pointer-event scoping and full
container width to function.

Visible symptom: clicking "How attackers wait" or "The disguise" on
Positioning opens the slide-out, but the visualization renders
statically — the timeline scrubber doesn't drag, the two-views
controls don't toggle. Reconnaissance kill chain renders interactively
in its slide-out (unaffected by this issue) because click-to-toggle
nodes don't require the same layout context.

This release moves the `VisualizationRenderer` OUT of the Section
wrapper and renders it directly in the slide-out body. Description
and Roles continue to use Section wrappers (they're static content
where Section wrapping is correct). The "How this visualization
works" label still appears (as a styled heading, same visual as a
Section header) but the renderer no longer has a collapsible
container around it. This restores the unconstrained layout context
the visualizations had when they were inline (where they worked
correctly), while preserving the slide-out delivery mechanism Omar
specified.

## Pedagogical insight (OBS-027 — methodological)

**Component layout context is part of the API.**

Two visualization components rendered in the same container with the
same wrapper produced different behaviors. KillChainGrid worked
because its interactions are click-on-element atomic events that
don't care about container layout. PositioningTimeline and
PositioningTwoViews failed because their interactions (drag scrubber,
split-screen toggle) make implicit assumptions about pointer-event
scoping, container width, and ancestor positioning context — and
when those assumptions are broken by a wrapping Section's
collapsible button + marginBottom + content-div structure, the
interactions silently fail to fire.

The lesson: a visualization component's interactivity contract
includes its expected layout context. Wrapping a component in a
"compatible-looking" container is not actually compatible if the
container changes the layout context. The fix is either to scope
the wrapper to the contract (which is what this release does — drop
the Section and render directly), or to redesign the component to
work in any container (which would be substantial component-level
work, scoped to a future release).

Generalized: when a UI pattern works for some content types but not
others, the failing case is information about the pattern, not the
failing content. The Section wrapper is fine for static content
(Description, Roles) and fine for click-to-toggle visualizations
(kill chain). It's not fine for components with complex internal
interactions. The fix lives at the wrapper, not at the component.

## What ships in this release

### Modified component

`client/src/components/framework/TacticVisualizationSidebar.jsx`
— one structural change in the body section:

**Before (.27.4):**
```jsx
<Section
  title="How this visualization works"
  collapsible
  expanded={isExpanded('visualization')}
  onToggleExpand={() => toggleSection('visualization')}
>
  <VisualizationRenderer viz={viz} effectiveRole={effectiveRole} />
</Section>
```

**After (.27.5):**
```jsx
<div style={{ ...headerCommonStyle, marginBottom: 10, paddingBottom: 6, borderBottom: '1px solid var(--rule)' }}>
  How this visualization works
</div>
<div style={{ position: 'relative' }}>
  <VisualizationRenderer viz={viz} effectiveRole={effectiveRole} />
</div>
```

The "How this visualization works" header is preserved as a
styled label (same visual treatment as a Section header). The
renderer is wrapped in a minimal `<div style={{ position:
'relative' }}>` to provide a positioned ancestor for any
absolute-positioned children inside the visualization
(scrubber thumbs, tooltips, etc.) without imposing the rest
of the Section's structural constraints.

The Description and Roles Sections above are unchanged — they
contain static content where Section wrapping is structurally
correct.
