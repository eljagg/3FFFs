# v25.7.0.25 — Role management via Auth0 Management API

Closes the last operational gap in the Users & Invites surface. Admin
users can now grant and revoke roles (`trainee` / `manager` / `admin`)
from the user detail modal, with the change round-tripping to Auth0 so
authorization actually changes — not just a 3FFFs-side display update
that gets blown away by `syncUser()` on next login.

Auth0 Management API plumbing added:
- M2M client credentials → token caching with 5-minute pre-expiry refresh
- Role IDs hardcoded for the 3fffs-training tenant (saves a `/api/v2/roles`
  lookup round-trip per grant/revoke; tenant-pinned, must update if
  tenant changes)
- Atomic ordering: Auth0 first, Neo4j mirror second — if Auth0 rejects,
  the local graph is untouched
- 503 response if env vars missing (recoverable from partial deploys)
- Self-demotion guard for the admin role

Pre-flight verified: M2M token endpoint returns 24h Bearer with all six
expected scopes (`read:users update:users read:roles read:role_members
create:role_members delete:role_members`).

## Pedagogical insight (OBS-027)

**Bidirectional sync is for fields that affect authorization.**

This is the explicit complement to v25.7.0.24's pedagogical insight
(*don't fight your IdP — augment it*). The displayName solution worked
because names are display-only: 3FFFs can hold a value the IdP doesn't
know about, and nothing breaks. Roles are different. Roles flow through
the Auth0 Action into the JWT, the JWT gates `requireRole()` on the
server, and the server is what decides whether a request to
`/api/admin/users` succeeds or 403s. A 3FFFs-only role write would
silently fail to grant access on the user's next session — they'd see
a "promoted to manager" toast in the UI and then bounce off every
manager-gated endpoint until someone debugged it.

So roles round-trip. Auth0 first (the authoritative write), then a
mirror to the local Neo4j `:User.roles` array (so the modal reflects
the change before the user logs in again, and so other 3FFFs queries
that read `u.roles` see the new state). The ordering matters: if Auth0
rejects, Neo4j is untouched, and the admin sees a clean failure
message. If Neo4j fails after Auth0 succeeded, the next `syncUser()`
on the user's next login re-mirrors from Auth0 anyway — so eventual
consistency is automatic. Auth0 is always the convergence anchor.

The principle generalizes: when integrating with an upstream system,
classify each field by whether divergence would change authorization
or just presentation. Presentation-only → parallel field, no sync.
Authorization-affecting → bidirectional with the upstream as anchor,
local mirror for UX speed. The cost of getting this distinction wrong
in either direction is real:
- Treating a presentation field as authorization-affecting means
  unnecessary plumbing, more failure modes, and slower iteration
- Treating an authorization field as presentation-only means
  silently-broken access controls

3FFFs's `User.displayName` and `User.roles` now sit on opposite
sides of this line, with the architectural reasons made explicit
in the comments.

## Four-lens block (OBS-018)

- **Teller:** No direct teller workflow change. Indirect benefit:
  when a teller is promoted to a supervisor / manager role, the
  promotion in 3FFFs no longer requires waiting for IT to do
  Auth0 dashboard work — admin flips the role pill and the
  promoted user has manager-tier access on their next session.

- **Analyst:** Same as teller — promotion / demotion no longer
  requires an IT ticket. More importantly, the audit trail
  (`User.lastRoleChangeAt`, `User.roleChangedBy`) becomes
  queryable from Aura: "who was promoted to manager in the last
  30 days, by whom" is a one-line Cypher query, useful for
  quarterly access review.

- **SOC:** This is the lens that gets the most attention this
  release. Specific guardrails:
  - **Self-demotion guard:** an admin cannot revoke their own
    `admin` role. Server-side check (rejects with 400) and
    client-side affordance (× button is replaced with a 🔒 icon
    on the current user's admin pill).
  - **Scope-limited M2M token:** the M2M app's authorized scopes
    are exactly the six needed (`read:users`, `update:users`,
    `read:roles`, `read:role_members`, `create:role_members`,
    `delete:role_members`). Even if the Client Secret leaked,
    the blast radius is "grant/revoke roles to existing Auth0
    users in this tenant" — not delete users, not change passwords,
    not touch other tenants.
  - **Atomicity:** Auth0 write first, Neo4j mirror second. If
    Auth0 fails, no half-state. Neo4j is the eventual-consistency
    follower, not a parallel source of truth.
  - **Confirm on every change:** browser-native `confirm()` for
    each grant or revoke. Yes, basic, but it's the right friction
    for a privileged action.
  - **Audit fields written on every change:** `lastRoleChangeAt`
    + `roleChangedBy` on the User node, surfaceable in the modal
    sub-label ("↻ role last changed 11h ago by …@barita.com")
    and in raw Cypher.
  - **503 if Auth0 not configured:** the role endpoints check
    for the env vars at request time. Missing = clean 503 error
    response, not a cryptic crash. Means a partially-configured
    deploy is safe.

- **Executive:** The user management surface is now feature-
  complete for the reference-customer phase. Six releases ago
  (v25.7.0.21) this surface was: invites that lapsed silently, no
  way to inspect who was using the platform, no way to fix bad
  names, no way to reassign banks, no way to promote managers
  without an IT ticket. As of v25.7.0.25 every one of those is
  closed. The Manager assignments tab — which has read "Assign
  the manager role in Auth0 first." for the entire history of
  the platform — finally has a path that doesn't require Auth0
  dashboard access (grant `manager` from the user modal, then
  use the existing Manager assignments tab to wire up the
  `:MANAGES` edges).

## What ships in this release

### Server — new `server/src/lib/auth0-management.js`

~140-line module wrapping the four operations 3FFFs needs:

- **Configuration check:** `isAuth0Configured()` returns false if
  any of `AUTH0_DOMAIN`, `AUTH0_M2M_CLIENT_ID`,
  `AUTH0_M2M_CLIENT_SECRET` are missing. Higher-level endpoints
  use this to return 503 instead of crashing.
- **Token cache:** module-level `cachedToken` + `cachedExpiresAt`,
  refreshed when within 5 minutes of expiry. Typical role change
  hits Auth0 once (the role endpoint), not twice (token + role).
- **Internal request helper:** `managementRequest(path, init)`
  with one retry on 401 (handles the rare case of token
  invalidation between cache and call).
- **Public functions:**
  - `assignRoleToUser(auth0UserId, roleName)` — POST
    `/api/v2/users/:id/roles`
  - `removeRoleFromUser(auth0UserId, roleName)` — DELETE
    `/api/v2/users/:id/roles`
- **Hardcoded role ID table** (3fffs-training tenant):
  ```js
  trainee → rol_GepPsVvm8XJA7DRi
  manager → rol_ye9qNc9xR8KgCMmJ
  admin   → rol_wemMbZP48YNuVZ6E
  ```
  Tenant-pinned. If the Auth0 tenant migrates or roles are
  recreated, this table needs updating to match. Comment in the
  file makes this explicit.

### Server — new endpoints in `server/src/routes/admin.js`

**`POST /api/admin/users/:id/roles`** — grant a role:
```
Body: { role: 'trainee' | 'manager' | 'admin' }
Response: { ok: true, granted: role, roles: [...new roles array...] }
400 if role is not one of the three valid values
404 if user not found in Neo4j
502 if Auth0 rejects the grant
503 if Auth0 Management API not configured
```

**`DELETE /api/admin/users/:id/roles/:role`** — revoke a role:
```
Response: { ok: true, revoked: role, roles: [...new roles array...] }
400 if attempting self-demotion of admin (the lockout guard)
400 if role is not one of the three valid values
404 if user not found
502 if Auth0 rejects
503 if Auth0 Management API not configured
```

Both endpoints write the audit trail (`u.lastRoleChangeAt`,
`u.roleChangedBy`) on success. Both are idempotent — granting an
already-granted role or revoking an already-revoked role both
succeed silently (Auth0's behavior; we mirror it).

### Server — `GET /api/admin/users/:id` payload

Now also returns `lastRoleChangeAt`, `roleChangedBy`, and
`roleChangedByEmail` (via OPTIONAL MATCH on the changing admin's
User node).

### Client — `client/src/lib/api.js`

Two new functions:
- `adminGrantRole(userId, role)` — POST
- `adminRevokeRole(userId, role)` — DELETE

### Client — `client/src/pages/AdminUsers.jsx`

`UserDetailBody`:
- New import: `useAuth0` from `@auth0/auth0-react` to detect
  whether the modal is showing the current admin's own profile
  (`isViewingSelf`).
- New state: `showAddRole`, `roleBusy`. Computed `userRoles` and
  `missingRoles` arrays drive the editable role pill UI.
- Replaced static `<RolePills>` in the metadata bar with an
  inline editable version:
  - Each role pill has a small `×` button to revoke (with
    `confirm()` dialog before the API call)
  - On the current user's admin pill, the × is replaced with a
    🔒 icon and a "Cannot revoke your own admin role" tooltip —
    matches the server-side guard
  - `[+ Role]` button after the pills opens a small popover
    listing only roles the user doesn't have; clicking grants
    after a confirm
- New audit sub-label below the metadata bar: "↻ role last
  changed X ago by Y" when `lastRoleChangeAt` is set
- Editorial nudge in edit mode updated: now says "Email is not
  editable here" (singular) and explicitly mentions roles as
  editable via the × / + Role affordances

## Deploy

Single-step. No migration. No new schema (audit fields on User
are property additions; Neo4j is schemaless on properties).

```bash
cd /workspaces/3FFFs
unzip -o v25.7.0.25-3FFFs.zip
rm v25.7.0.25-3FFFs.zip
git add -A
git status
git commit -m "v25.7.0.25: Role management via Auth0 Management API"
git push origin main
```

Wait for Railway to redeploy. The new env vars
(`AUTH0_M2M_CLIENT_ID`, `AUTH0_M2M_CLIENT_SECRET`) you added before
this build are read at request time, not at startup, so the new
endpoints will work as soon as the server is live.

## Verification — 5 minutes

### Step 1 — Verify Auth0 wiring (60 seconds)

Open `/admin/users`, click your own row to open the modal. Look at
the role pills row in the metadata bar.

**Pass:** your `admin` pill shows a 🔒 icon (not a ×) — that's the
self-demotion guard rendering. Other pills (if you have any) show
× buttons.

If the page loads but role pills don't have × or 🔒 affordances,
the client deploy didn't take. If the modal shows a 503 error
when you try to use the affordances, env vars aren't set or Railway
hasn't picked them up yet.

### Step 2 — Grant yourself the manager role (45 seconds)

Currently you have only `admin`. Let's add `manager`.

1. Click the **+ Role** button at the end of the role pills row
2. Popover opens, showing "trainee" and "manager" (the two roles
   you don't have)
3. Click **+ manager**
4. Confirm dialog: "Grant manager role to o.mcleod@barita.com? This
   will be reflected in Auth0 immediately." → click OK
5. Brief pause (Auth0 round-trip; usually <500ms)
6. Toast: ✓ Granted manager
7. Modal updates: a `MANAGER` pill appears next to `ADMIN`, the
   `+ Role` button now offers only `+ trainee`

Verify in Auth0 dashboard: User Management → Users → click your
user → Roles tab. You should see `manager` and `admin` listed.

Verify in Aura:
```cypher
MATCH (u:User {email: 'o.mcleod@barita.com'})
RETURN u.roles, u.lastRoleChangeAt, u.roleChangedBy
```
**Pass:** `u.roles` is `['admin', 'manager']` (or in some order),
`u.lastRoleChangeAt` is recent, `u.roleChangedBy` is your own
user id.

### Step 3 — Audit sub-label renders (10 seconds)

Below the metadata bar in the modal, you should now see:
**↻ role last changed just now by o.mcleod@barita.com**

**Pass:** sub-label visible. Hover for absolute timestamp.

### Step 4 — Try to revoke your own admin (30 seconds)

The 🔒 should prevent you from clicking × on `admin`. To test the
server-side guard directly:

```bash
curl --request DELETE \
  --url https://YOUR-RAILWAY-SERVER-URL/api/admin/users/YOUR-AUTH0-USER-ID/roles/admin \
  --header "authorization: Bearer YOUR-USER-JWT"
```

(Get your JWT from the browser dev tools → Network tab → any
admin API request → Request Headers → Authorization.)

**Pass:** response is `{"error":"You cannot revoke your own admin
role — that would lock you out. Have another admin do it."}` with
HTTP 400. Server guard works.

### Step 5 — Revoke the manager role you just granted (30 seconds)

1. Click × next to the `MANAGER` pill
2. Confirm: "Revoke manager role from o.mcleod@barita.com? They'll
   lose manager-tier access on their next session." → OK
3. Toast: ✓ Revoked manager
4. `MANAGER` pill disappears, `+ Role` popover now offers both
   `+ trainee` and `+ manager` again

Verify in Auth0 dashboard the `manager` role is removed. Verify
in Aura that `u.roles` is back to just `['admin']` and
`u.lastRoleChangeAt` updated.

### Step 6 — Manager assignments tab unblock (60 seconds)

This is the bigger payoff: the Manager assignments tab has shown
"Assign the manager role in Auth0 first." since the platform
launched. Let's verify it's now usable.

1. Navigate to `/admin/users` → Manager assignments tab
2. Currently shows "0 managers" because no user has the `manager`
   role yet
3. Go back to Users tab. Click any user (could be the eljagg user
   if you've consumed that invite, or any future user once you
   onboard one). For the test, let's grant `manager` to your own
   account again temporarily.
4. Re-grant `manager` to yourself (Step 2 procedure)
5. Go back to Manager assignments tab
6. **Pass:** you should now appear as a manager option in the
   assignment UI; you can assign yourself to manage other users
7. Once verified, revoke your `manager` role to clean up

(This step is exploratory — exact UX of the Manager assignments
tab assignment flow is unchanged from v25.7.0.21.)

## Honest accounting

Non-goals — explicit so they don't silently land in scope:

- **No bulk role assignment yet.** That's v25.7.0.26 (bulk user
  import), which will reuse the auth0-management lib for the
  role-assign mode.
- **No new role types.** The system supports exactly trainee /
  manager / admin. Adding new roles requires (a) creating the
  role in Auth0, (b) updating the hardcoded `ROLE_IDS` table in
  `auth0-management.js`. Three roles cover the planned scope; if
  3FFFs ever needs more (e.g. `auditor`, `viewer`), revisit then.
- **No Auth0 user creation from 3FFFs.** Users still come into
  existence by logging in via Auth0 (allowlisted domain or
  consumed invite). Bulk import (v25.7.0.26) handles the
  bulk-invite path; it does not pre-create Auth0 users.
- **No password reset / MFA management from 3FFFs.** Both stay on
  the Auth0 dashboard. Out of scope for the M2M client's authorized
  scopes by design (blast-radius limit).
- **Role IDs are tenant-pinned.** If you migrate to a new Auth0
  tenant or recreate the roles, the hardcoded table must be
  updated. There's no dynamic lookup fallback by design (the
  whole point of hardcoding is to save round-trips).
- **No retroactive backfill of `roleChangedBy` for users
  pre-v25.7.0.25.** Users whose roles were set in Auth0 dashboard
  before this release won't have audit fields — the audit only
  captures changes made through 3FFFs going forward.

## Forward to v25.7.0.26 — bulk user import

Per your direction, bulk user add is next. Two-mode design:

**Mode 1 — Bulk INVITE** (off-allowlist domains, e.g. partner banks
or external reviewers). Creates `:Invite` nodes for each row. No
Auth0 dependency.

**Mode 2 — Bulk ROLE ASSIGNMENT** (allowlisted domains, e.g.
@barita.com employees). Users will auto-provision on first login;
the bulk import pre-stages their intended roles so they hit the
right curriculum on day one. Reuses the `auth0-management.js`
helper from this release. Requires the user to log in at least
once before role grants land (Auth0 needs the user object to
exist), so this mode either:
- Creates Auth0 users via the Management API ahead of time
  (requires `create:users` scope, which is not currently in our
  M2M's authorized list)
- Or stages the roles in 3FFFs and applies them after first login
  (defers to syncUser-time application)

The second is simpler and doesn't expand the M2M's scope blast
radius. The first is more immediate. Decision deferred to the
v25.7.0.26 scoping turn.

  whether they're in the existing build)

  per-row classification (invite vs role-assign) and validation
  errors
- Submit: server batches the work, returns per-row results
  (created / updated / errored)
- Error rows downloadable as CSV for fix-and-reupload
