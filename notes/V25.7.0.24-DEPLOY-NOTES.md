# v25.7.0.24 — Editable user detail (display name + bank reassignment)

Third consecutive operational-tooling release. Closes the next set of
paper-cuts on the User detail modal that v25.7.0.23 surfaced:

1. **`(no name)` for users whose Auth0 profile has no name field.**
   Now editable via a `displayName` override that survives every login.
2. **Hard-coded bank assignment via email-domain auto-link.** Now
   admin-overridable from the modal, with the auto-domain link in
   `syncUser()` updated to respect manual assignments.

Role promote/demote (manager / admin) is **not** in this release. Roles
live in Auth0 and durable role changes from 3FFFs require the Auth0
Management API — that's its own integration with a config story
(machine-to-machine app, env vars, role-id mapping per tenant) and ships
as v25.7.0.25 separately.

## Pedagogical insight (OBS-027)

**Don't fight your identity provider — augment it.**

Auth0 owns email, name, and roles for every user, and `syncUser()`
overwrites those fields on the `:User` node on every login. If you
let an admin edit `User.name` directly in 3FFFs, the next login
silently obliterates the edit. The naive fix is to round-trip the
edit through the Auth0 Management API so Auth0 itself is updated.
That works but it's a substantial integration with its own
operational footprint (M2M client credentials, token caching, role-id
lookup, per-tenant config) and it's overkill when the underlying
need is "give the user a sensible display name in our UI."

The cleaner pattern is a parallel field — `User.displayName` — that
the IdP sync explicitly does NOT touch. The UI resolves
`displayName || name || '(no name)'`. The IdP keeps owning identity;
3FFFs keeps owning the display layer; both can change independently
without colliding. The same pattern would apply to a future
`User.preferredEmail` if customers ever need an alias different from
their login email.

The principle: when you're integrating with an upstream system that
owns a field, don't try to make that field bidirectional. Add a
sibling field and make the resolution explicit. The bidirectional
case is real (and that's what v25.7.0.25 will need for roles,
because roles affect authorization, which Auth0 must know about) —
but reach for it only when the field genuinely has to be the same
value in both systems. For display-only data, parallel fields are
simpler and never get out of sync because there's nothing TO sync.

## Four-lens block (OBS-018)

Admin tooling release; lenses adapt:

- **Teller:** No change. Teller workflow is unaffected.

- **Analyst:** Bank reassignment becomes the means by which a fraud
  analyst who moves between institutions retains their progress
  history without an account reset. With the syncUser change in
  this release, an admin can move e.g. a Scotia analyst to JNCB by
  flipping the bank dropdown — the engagement data, completed
  scenarios, and quiz history all carry over because the `:User`
  node is unchanged. Their email domain and Auth0 role might still
  reflect Scotia until the analyst's IT resets them, but the
  3FFFs-side reassignment is independent and immediate.

- **SOC:** New `PATCH /api/admin/users/:id` endpoint sits behind
  `requireRole('admin')`. No new attack surface for non-admin users.
  The endpoint cannot grant or remove roles (that's a deliberate
  guardrail; roles flow only from Auth0 in this release). Bank
  reassignment changes a `:MEMBER_OF` edge but does not affect
  authentication or scope-based authorization — bank scoping is a
  presentation concern, not a security one. Audit fields
  (`displayNameSetAt`, `displayNameSetBy`, `:MEMBER_OF.assignedBy`)
  capture who made the change.

- **Executive:** Two specific demo flaws disappear: the `(no name)`
  placeholder is no longer permanent, and bank assignment is no
  longer a one-shot decision baked in at first login. Both are
  small but they're the kind of thing a CRO catches in a 30-second
  product walkthrough and asks about. Now they don't ask.

## What ships in this release

### Server — `server/src/lib/auth.js` syncUser change

The auto-domain `:MEMBER_OF` link in `syncUser` now respects manual
overrides. Old behavior: every login, if the email domain matches a
`:Bank.domains` entry, MERGE a `:MEMBER_OF` edge to that bank. New
behavior: only fire the auto-domain link if the user has **zero**
existing `:MEMBER_OF` edges. Once an admin has explicitly set a bank
(or unset one), syncUser leaves the assignment alone forever.

```cypher
MATCH (user:User {id: $id})
OPTIONAL MATCH (user)-[existing:MEMBER_OF]->(:Bank)
WITH user, count(existing) AS existingCount
OPTIONAL MATCH (b:Bank) WHERE $domain IN b.domains
FOREACH (_ IN CASE WHEN b IS NULL OR existingCount > 0 THEN [] ELSE [1] END |
  MERGE (user)-[r:MEMBER_OF]->(b)
  ON CREATE SET r.linkedAt = timestamp(), r.method = 'auto-domain'
)
```

OBS-021 N/A (the MERGE clause ordering is unchanged). OBS-025
preservation: existing user-to-bank edges are untouched on every
login pre and post change; the new behavior only differs for the
admin-override case which didn't exist before. Verifiable with
`MATCH (u:User)-[r:MEMBER_OF]->() RETURN count(r)` — the count is
stable across deploys for any user that has logged in at least once.

### Server — new endpoints in `server/src/routes/admin.js`

**`PATCH /api/admin/users/:id`** — admin edit endpoint:

```
Body: { displayName?, bankId? }
  displayName: string  → set User.displayName + audit fields
               '' / null → remove the override; UI falls back to Auth0 name
               undefined → field untouched (PATCH semantics, not PUT)
  bankId:      string  → reassign user to that :Bank (validated; 400 if missing)
               '' / null → unassign (delete all :MEMBER_OF edges)
               undefined → field untouched
Response: { user, bank } — refreshed payload so the client doesn't re-fetch
404 if user id not found
400 if bankId provided but :Bank doesn't exist
```

PATCH semantics: only fields explicitly present in the body are
modified (`hasOwnProperty` check, not truthiness). That means an
empty-string displayName is a real "clear the override" signal, while
omitting the field is a "leave it alone" signal. Empty strings vs
`undefined` matter; we lean into the distinction.

Audit fields written on edit:
- `displayName` set: `u.displayNameSetAt`, `u.displayNameSetBy`
- `displayName` cleared: all three properties REMOVEd
- bank reassigned: edge property `r.method = 'admin-override'` and
  `r.assignedBy` set to the editing admin's user id

**`GET /api/admin/banks`** — lightweight list of all `:Bank` nodes
for the bank-reassign dropdown. Returns `{ banks: [{ id, displayName,
name, domains }] }` ordered by `displayName`. Lazy-loaded by the modal
when the user clicks Edit (not on initial open).

### Server — `GET /api/admin/users` and `GET /api/admin/users/:id`

Both queries updated to return `displayName` alongside `name` so the
client can resolve the display value identically in the list view
and the modal. No behavior change for users without a `displayName`

## Deploy

Single-step. No migration required — `displayName` is a new property
on existing `:User` nodes (Neo4j is schemaless on properties; users
without the override simply have it as `null`). Edge property changes
(`assignedBy`, `method='admin-override'`) are also schemaless.


Should return `1`. If you get `2`, the syncUser change didn't deploy

## Honest accounting

Non-goals — explicit so they don't silently land in scope:

- **Email is not editable.** Auth0 owns it; changing it in 3FFFs
  alone would be reverted on next login and breaks the invite-by-
  email lookup. Email changes happen at the Auth0 user level.
- **Roles are not editable.** Same reason; coming in v25.7.0.25 with
  Auth0 Management API plumbing.
- **No bulk reassign.** One user at a time. Adjacent to Gap 3.
- **No bank create / edit / delete from the UI.** New banks come
  from the seed file (`server/src/seed/data/banks.js`). Adding
  banks-management UI is a future surface and isn't urgent.
- **Display name is not pushed to Auth0.** It exists only in the
  3FFFs Neo4j graph. If a customer also wants their Auth0 profile
  updated, they update it in Auth0 directly. This is a deliberate
  scope boundary — see the pedagogical insight above for why.

## Forward to v25.7.0.25 — role management via Auth0 Management API

Required setup (your side, before v25.7.0.25 deploys):

1. In your Auth0 dashboard, create a Machine-to-Machine application
   ("3FFFs Management API client" or similar)
2. Authorize it for the **Auth0 Management API** with these scopes:
   - `read:users`, `update:users`, `read:roles`, `read:role_members`,
     `create:role_members`, `delete:role_members`
3. Copy the Client ID and Client Secret
4. Add three env vars to the 3FFFs Railway server:
   - `AUTH0_M2M_CLIENT_ID`
   - `AUTH0_M2M_CLIENT_SECRET`
   - `AUTH0_DOMAIN` (already exists for the existing Auth0 client;
     reuse)
