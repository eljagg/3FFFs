# v25.7.0.27.1 — Hotfix: tactic-level visualizations to slide-out

Hotfix to v25.7.0.27. Brings tactic-level visualizations
("How attackers wait", "The disguise", Reconnaissance kill chain,
Defense Evasion two-views) into alignment with the slide-out
interaction model used by every technique animation.

Pre-hotfix state (v25.7.0.3 / v25.7.0.4 design): tactic-level
visualizations rendered as inline collapsible cards on the tactic
landing page. Click the card title → expand inline below it.
Two different interaction patterns coexisted on the Framework
page: technique animations slid out from the right via
`TechniqueDetailSidebar`, but tactic-level visualizations
expanded inline.

Post-hotfix state: every detail-content interaction on the
Framework page opens via the same right-side slide-over panel.
Click a tactic-visualization card → slide-out opens with the
visualization rendered inside, identical UX to clicking a
technique card.

Omar flagged this directly: "you did not follow the slide out
from the right as we have been doing... 'How attackers wait'
and 'The disguise' should not be where they are. Both should
be in the slideout as all the previous." This release does
exactly that.

## Pedagogical insight (OBS-027)

**Unified interaction model for detail content.**

Two interaction models for two functionally adjacent content types
is friction tax. The user is mentally placing
"How attackers wait" on the Positioning tactic page next to
"Account Manipulation: Change E-Delivery / Notification Settings"
under it — both teach about the Positioning phase, both surface
the same pedagogical territory of the patient pre-execution window
— and they should respond to the same interaction. Click → opens
detail panel from the right. Esc / × / backdrop click → dismiss.
Done.

The previous coexistence (slide-over for technique animations,
inline-collapse for tactic-level visualizations) was an accumulated
artifact of release sequencing: tactic visualizations shipped first
in v25.7.0.3 / v25.7.0.4 with an inline pattern that made sense at
the time; the technique slide-out shipped later in v25.7.0.8 with
a deeper interaction model; the two never got reconciled. This
release reconciles them.

The principle generalizes: when two surfaces in the same product
display structurally similar content (interactive teaching panels
attached to graph entities), they should share interaction model
unless there is a specific pedagogical reason to differ. There
isn't one here. The visualizations and the animations are both
"open this to learn how this works" content. One slide-out panel
pattern across both.

## What ships in this release

### New component

`client/src/components/framework/TacticVisualizationSidebar.jsx` —
~125 lines. Mirrors the pattern of `TechniqueDetailSidebar` but
scoped to tactic-attached visualizations:

- Fixed-position right-side slide-over via Framer Motion
  (`initial={{ x: '100%' }} animate={{ x: 0 }}` slide-in;
  exit reverses)
- Backdrop dim with click-to-close
- ESC key dismiss
- Wide panel `min(1400px, 95vw)` matching animation-equipped
  technique sidebar width — tactic visualizations are wider than
  text content and benefit from the same horizontal real estate
- Header with tactic-id breadcrumb (e.g. "TACTIC VISUALIZATION ·
  FA0001"), title, subtitle, close × button
- Body renders `<VisualizationRenderer viz={viz}
  effectiveRole={effectiveRole} />` — same renderer the inline
  pattern used, no changes to downstream visualization
  components

Why a separate sidebar (not extension of `TechniqueDetailSidebar`):
`TechniqueDetailSidebar` is bound to a specific technique id
and fetches full technique metadata (description, roles,
mitigation, animation scenes, cross-refs); tactic-level
visualizations have a different shape and different rendering
path. Reusing the existing sidebar would have required
type-tagging and conditional rendering branches. A parallel
small component is cleaner and isolates the change.

### Framework.jsx modifications

`client/src/pages/Framework.jsx` — 4 specific changes:

1. **New import:** `TacticVisualizationSidebar` added to imports
2. **Removed import:** `VisualizationRenderer` no longer imported
   here (moved to `TacticVisualizationSidebar.jsx`)
3. **Removed dead code:** the `VIZ_DEFAULTS`,
   `isVizDefaultExpanded`, `vizToggleOverrides`,
   `isVizExpanded`, `toggleViz` block (~25 lines) that backed
   the inline expand/collapse pattern is gone — replaced by
   `vizSidebar` state (`{ open, viz }`) and
   `openVizSidebar` / `closeVizSidebar` handlers
4. **Inline rendering replaced by click-to-open card:**
   the previous `visibleViz.map(viz => …)` block that rendered
   a button + chevron + conditional `<VisualizationRenderer />`
   inline is now a button-only card. `onClick` calls
   `openVizSidebar(viz)`; rendering happens inside the
   `<TacticVisualizationSidebar>` instance
5. **Page-level sidebar instance:** `<TacticVisualizationSidebar
   open={vizSidebar.open} viz={vizSidebar.viz}
   effectiveRole={effectiveRole} onClose={closeVizSidebar} />`
   added next to the existing `<TechniqueDetailSidebar />`
   instance at the bottom of the page render, so it overlays
   the entire Framework page when open

The `vizCollapseBody` style is left in the styles object because
it's a pure style fragment (no runtime impact unused) and removing
it requires touching a different code region; cleanup deferred to
the next general code-quality pass.
