## What changed and why

The Detection Controls section in every animation rendered as a
CSS-grid `auto-fill, minmax(280px, 1fr)` of dense-text control
cards. At slide-out width with 3+ controls (F1042, F1043, F1007.x,
F1008.001, F1018, T1621, F1072, F1018.001 — i.e. most of the
animation library) the grid wraps into multiple rows, leaves
substantial empty space below the last row, and forces the trainee
to read three side-by-side dense paragraphs in 13px font.

Omar identified the layout pain on F1042 during v25.7.0.30
verification. The fix is master-detail: compact left list of cards
that show name + checkbox + active/firing affordances, and a right
detail panel that uses the freed-up space to render the selected
control's full meta text in 14.5px reading font with proper line
height.

Behavioural rules that were preserved exactly:

- **Toggle still toggles.** Click the checkbox to activate /
  deactivate a control. Same `onToggle(controlId)` callback, same
  `activeControls` state-set on the engine side. The "Would have
  flagged N of N events" callout still appears when active. The
  "Active at stage X" stage hint still appears when toggled active
  but the signal isn't firing yet at the current stage. The
  `naive` badge still appears for naive controls. The
  controlToggleActiveNaive dimmed treatment still applies.
- **Engine state model unchanged.** No new state is introduced on
  the engine side. The component manages a local `selectedId` UI
  state for which control is showing in the detail panel — that's
  it. Step through stages, change controls, change techniques —
  all existing engine behaviour works unchanged.
- **derivedRevealsAtStagesByControl** is still passed through
  (per v25.7.0.14.1 — engine-derived stage reveals override scene
  metadata when present).

New affordances added:

- **Selected indicator** on left list — accent-coloured left
  border on the currently-selected card.
- **Firing indicator** on left list — when a control is toggled
  active and its signal is revealed at the current stage,
  highlights with a subtle outer-glow box-shadow plus a "★ Firing
  now at this stage" hint inside the card. Catches the trainee's
  eye for the moment the catch landing happens — useful when
  toggling controls while playing through a stage.
- **Default selection** — first control in the list auto-
  selects on slide-out open / technique switch. Detail panel is
  populated immediately rather than blank-by-default.

What was kept intentionally out of scope:

- **Master-detail for Revealed Signals.** Signals already render
  as a vertical stack (one card per revealed signal, full-width).
  That layout is already readable. Master-detail there would
  introduce the click-to-select interaction without solving a
  visible pain point. Deferred to backlog if signals layout ever
  becomes the friction point.
- **Mobile / narrow-viewport collapse.** Grid collapses to single-
  column on widths < 600px (CSS handles it); a more bespoke
  inline-expandable list pattern is on the existing
  narrow-viewport responsive backlog item.

## Pedagogical insight (locked v25.7.0.31 — UX-meta)

**The layout container is part of the API.** The same dense-text
control cards rendered acceptably in a wide single-column main view
and unacceptably in a slide-out at narrower width with multiple
columns wrapping. The content hadn't changed; the container had,
and the container's geometry imposed reading constraints on the
content that the content wasn't designed for.

Process change banked: when a UI component is moved into a new
container (Section, slide-out, modal, embed), assume the layout
contract is violated until proven otherwise. Test the move
under each interaction the component supports (read, scan,
toggle, focus). The v25.7.0.27.5 PositioningTimeline scrubber
fix and the v25.7.0.27.6 useEffect-import bug were both
container-context regressions. This release is the first
proactive layout response to a container-context observation
(rather than a reactive fix after a regression report).

## Four-lens block (OBS-018)

- **Teller:** Tellers reading detection-control descriptions
  during F1042 / F1043 / F1007.x / etc. now see the full meta
  text in larger reading font on the right side of the controls
  section. Click between controls to compare. Toggle the
  checkbox to see the catch behaviour. Cleaner separation of
  read-and-compare vs. toggle-and-see-effect actions.
- **Analyst:** The "★ Firing now at this stage" indicator
  surfaces the moment a toggled control catches a signal at the
  current stage — useful when stepping through stages and
  watching which control would have caught what. Right detail
  panel shows the full meta text without size constraint, which
  helps when the institutional rationale for the control is
  longer than 5 lines (most controls are; the existing layout
  truncated the visual presentation in unhelpful ways).
- **SOC:** Five-file release; backend untouched. New shared
  component MasterDetailControls.jsx is added; the three engine
  files lose their inline `<div style={...controlsGrid}>...
  {controls.map((c) => <ControlToggle .../>)}</div>` block and
  call `<MasterDetailControls .../>` instead. The orphaned
  ControlToggle function and `controlsGrid` style remain in the
  engine source files for safe-rollback compatibility — they're
  now dead code but removing them adds risk for negligible
  gain. Will be cleaned up in a future sweep.
- **Executive:** No behavioural change to the demo flow. The
  demo experience is improved — same content presented with
  better hierarchy. If executives compare v25.7.0.30 demo
  recordings to v25.7.0.31, the controls section reads more
  cleanly with no information loss.

## Honest accounting

This is the first **proactive** layout response in the project —
not a reactive fix to a regression report. The pedagogical insight
above (layout container is part of the API) was banked from the
v25.7.0.27.5 / .27.6 / .29.1 prior post-mortems, and this release
applies that learning to a friction point Omar identified during
v25.7.0.30 verification rather than in a deploy-failure context.

The shared MasterDetailControls component centralizes what was
previously triplicated across three engines. Future changes to the
controls UX (e.g., keyboard navigation between controls,
deep-linking to a specific control via URL hash, animation between
detail-panel transitions) only need to touch one file.

Orphaned `ControlToggle` function definitions remain in each
engine for safe rollback. Removing them in a future cleanup pass
once the master-detail layout is verified across multiple
animations and verification cycles.
