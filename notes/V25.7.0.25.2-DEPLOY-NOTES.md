# v25.7.0.25.2 — syncUser race-condition fix for the role mirror

Tiny server-only hotfix on top of v25.7.0.25.1. Closes a dual-write
race that Omar caught by checking the Auth0 dashboard and the Aura
graph against the modal's success state — a check I should have asked
him to run as part of v25.7.0.25's verification, but didn't.

The bug: after a role grant, the local Neo4j `:User.roles` array was
silently reverting to the old value on the *next* request from the
browser, because `syncUser()` overwrites `user.roles` from the JWT's
roles claim, and the user's JWT still carries the pre-grant roles
until they re-log in. The audit fields (`lastRoleChangeAt`,
`roleChangedBy`) survived because `syncUser` doesn't touch them — so
the modal showed "↻ role last changed just now" while Neo4j had
already been quietly clobbered. Manager assignments tab read from
Neo4j and showed "0 managers" even though Auth0 had `manager`
assigned correctly.

The fix is one Cypher CASE expression. Five lines of business logic.

## Pedagogical insight (OBS-027)

**A dual-write isn't atomic just because both writes succeeded — it
has to also survive whatever the system does next.**

v25.7.0.25's role-grant endpoint did the dual-write correctly: Auth0
first (authoritative), Neo4j second (mirror), with rollback on
Auth0 failure. By the standard "atomic dual-write" definition that's
correct. But "correct at the moment of the write" is not the same as
"correct from the user's perspective five seconds later," and that
gap is where the bug lived.

The Auth0 Action issues a JWT at login time with the roles claim
baked in. That JWT is opaque and immutable for its lifetime
(typically hours). When the user makes a subsequent request, the
JWT's roles claim represents the user's role-state *as of last
login*, not as of right now. `syncUser()` was treating the JWT as
the live source of truth on every request, which is correct for
fields that don't change between logins (email, domain) but wrong
for fields that 3FFFs itself can mutate (roles, after v25.7.0.25
shipped).

The fix is to teach syncUser that local state can be newer than the
JWT, and to use a tie-breaker (`lastRoleChangeAt > iat`) to detect
when that's the case. Auth0 still wins the long-term race — the
user's *next* login will issue a fresh JWT with the new roles claim,
and `iat > lastRoleChangeAt` will flip back, returning syncUser to
its original behavior. Convergence is automatic.

The principle generalizes: any time you have a mirror of data that
the host system can mutate independently of the upstream, you need
either (a) a way to detect "local is newer than upstream" and skip
the upstream-wins overwrite, or (b) push every local mutation
upstream and never trust local-only state. v25.7.0.25 already did
(b) for the Auth0 round-trip; v25.7.0.25.2 adds (a) for the
gap between "we wrote to Auth0" and "the user's next login picks
it up."

The general lesson, applied beyond this codebase: a "write succeeded"
return value from a transactional API is not the end of the
verification. The right test is "does the user-observable state still
reflect the write five seconds later, after every middleware in the
request path has had a turn?" That's the test that catches race
conditions before users do.

## Four-lens block (OBS-018)

- **Teller / Analyst:** Indirect benefit. The Manager assignments
  tab will now reflect role grants without requiring the granted
  user to log out and back in. Means an admin can grant `manager`
  to a teller and immediately wire up a `:MANAGES` edge in the
  same session.

- **SOC:** No new attack surface. The CASE expression is a
  read-then-conditional-write within a single Cypher query, atomic
  at the Neo4j transaction level. No opportunity for partial state.
  The `iat` claim used as the tie-breaker is JWT-signed by Auth0 and
  cannot be forged by the client. Worst case if `iat` is missing
  from a payload (shouldn't happen with Auth0): code falls back to
  `Date.now()`, which means "treat JWT as fresh" — same as
  pre-v25.7.0.25.2 behavior, no regression.

- **Executive:** Closes the last paper-cut on the role-management
  surface. Combined with v25.7.0.25, role grants are now
  immediately effective in 3FFFs without "log out + back in"
  workarounds. Demo-ready for the reference-customer phase.

## What ships in this release

### Server — `server/src/lib/auth.js`

`syncUser()` modified — the SET clause for `user.roles` becomes
conditional:

```cypher
SET user.roles = CASE
  WHEN user.lastRoleChangeAt IS NOT NULL AND user.lastRoleChangeAt > $iatMs
  THEN user.roles                  -- local Neo4j is newer; keep it
  ELSE $roles                      -- JWT is fresh enough; overwrite from JWT
END
```

`iatMs` derives from the JWT's `iat` claim (issued-at, in seconds,
multiplied to ms). If the claim is missing for any reason, falls
back to `Date.now()`, which means "JWT treated as fresh" —
preserving the pre-v25.7.0.25.2 behavior exactly. No behavior
change for users without a `lastRoleChangeAt` audit field (the
common case for users who've never had their role changed via 3FFFs).

OBS-021 N/A (no MERGE clause changes).
OBS-025 preservation: existing user nodes unchanged on initial deploy.
First syncUser after deploy will keep stale Neo4j roles for any
user where `lastRoleChangeAt > iat`, which is the desired behavior.

## Verification — 2 minutes

### Step 0 — first, fix the current bad state

The race condition has already silently demoted you in Neo4j (Auth0
shows `[admin, manager]`, Neo4j shows `[admin]`). Easiest path back
to a clean state:

1. Log out of 3FFFs entirely
2. Log back in (Auth0 issues a fresh JWT with both roles)
3. Open Aura, run:
   ```cypher
   MATCH (u:User {email: 'o.mcleod@barita.com'})
   RETURN u.roles, u.lastRoleChangeAt, u.roleChangedBy
   ```
   Should now show `u.roles = ['admin', 'manager']` (or in either
   order) — the fresh JWT carried both roles, syncUser wrote them
   correctly.

### Step 1 — confirm the race is closed

After v25.7.0.25.2 deploys:

1. From `/admin/users`, click your row → **× MANAGER pill** → confirm
   to revoke. Toast: ✓ Revoked manager.
2. Without logging out, navigate to a different page (e.g. Manager
   assignments tab → back to Users)
3. Re-open your row → MANAGER pill should still be absent
4. Verify in Aura: `u.roles = ['admin']` (just admin)

**Pass:** the revoke survived the navigation. Without
v25.7.0.25.2, syncUser would have re-added `manager` from the JWT's
stale claim.

### Step 2 — confirm the inverse case still works (cross-user grant)

1. Re-grant yourself `manager` (this exercises the bug-fix path)
2. Verify in Aura immediately: `u.roles = ['admin', 'manager']`
3. Navigate around the app — Aura should *still* show both roles
4. **Pass:** dual-write integrity preserved

### Step 3 — confirm normal login still resyncs

1. Log out, log back in
2. JWT now carries `[admin, manager]` (Auth0's truth as of the
   grant)
3. Aura: `u.roles = ['admin', 'manager']`, `lastRoleChangeAt` still
   present (audit preserved)

**Pass:** normal login flow still works the same as pre-fix —
backwards compatible.

### Step 4 — clean up to leave your account in production state

If you don't want to actually be a manager in your real production
account, revoke `manager` after Step 3. Aura should show
`u.roles = ['admin']` and stay that way through navigation now.

## Honest accounting

- **The original bug was caught by Omar's spot-check, not by my
  verification path in v25.7.0.25's deploy notes.** I asked him to
  verify the modal showed the new pill and the audit sub-label;
  I should also have asked him to verify the Auth0 dashboard AND
  the Cypher reflected the change. Adding that as a standing
  pattern: any future dual-write feature gets verification steps
  for both sides of the write, not just the side that's most
  visible.
- **The fix doesn't address the related "stale JWT means stale
  request authorization" issue.** If an admin revokes another
  user's `manager` role, that user's existing JWT will still grant
  `manager`-tier access until it expires (typically hours).
  Auth0 doesn't support JWT invalidation in their hosted plans;
  the workaround if you need immediate revocation is to reduce
  the JWT lifetime in Auth0 settings (currently default; can be
  lowered to ~5 minutes). Out of scope for this hotfix; flagged
  for future consideration if 3FFFs ever needs immediate-revoke
  semantics for compliance.
- **Self-grants now persist without re-login**, but the user's
  effective authorization at the route level still uses
  `requireRole(...)` against the JWT's roles, which won't reflect
  the new role until re-login. So you can self-grant `manager` and
  see it in the UI, but you won't actually be authorized for
  manager-only routes until you log back in. Same caveat as above —
  not a regression, just a known property of JWT-based auth.
