# v25.7.0.25.1 — Auth0 env var trim defenses + diagnostic endpoint (hotfix)

Hotfix on top of v25.7.0.25. Two changes, both server-only, both small:

1. **`.trim()` defensive reads** on the three Auth0 env vars
   (`AUTH0_DOMAIN`, `AUTH0_M2M_CLIENT_ID`, `AUTH0_M2M_CLIENT_SECRET`) so
   leading/trailing whitespace from clipboard pastes can never again
   produce `access_denied` errors. Same fix I'd promised for v25.7.0.26;
   brought forward.

2. **New diagnostic endpoint** `GET /api/admin/debug/auth0-status` that
   reports the status of each env var (length, first/last char, whether
   trimming changed it) and runs a live token test against Auth0,
   returning the raw response. This is the diagnostic that will tell us
   exactly why role grant is currently failing, without touching the UI
   or guessing at credential issues.

Why a hotfix release line and not v25.7.0.26: the bulk-import work is
unrelated risk. Shipping a 4-line fix + a 60-line diagnostic in
isolation is cleaner than bundling it into a 4-6 hour bulk-import PR
that might get its own bugs. v25.7.0.26 starts after this lands and
the role-grant flow is verified working.

## Pedagogical insight (OBS-027)

**When debugging a system failure, the cheapest fix is the diagnostic
that makes future failures self-evident.**

The current debugging loop is: Omar tries grant → 401 → I guess at
the cause → Omar checks Railway → still wrong → repeat. That loop
costs a turn per guess and doesn't scale. The fix is a one-time
diagnostic surface that turns "guess and check" into "look and
know."

The diagnostic endpoint reports (a) what each env var actually
contains (without exposing values), (b) whether trimming changed it,
(c) the literal Auth0 response body when called with those env vars.
A single GET hits all three in one round-trip. After this hotfix,
"Auth0 isn't working" stops being a multi-turn diagnostic exercise
and becomes a single-curl observation.

The same pattern generalizes: every time a subsystem produces an
opaque failure mode (Auth0's deliberately-uninformative
`access_denied` is a textbook example), the right move is a
diagnostic surface that exposes the shape of the problem without
exposing the secret. Diagnostic endpoints have a maintenance cost
(more surface area to think about), but the cost is paid once and
the savings compound on every future failure.

The trim defenses are the complement: once you understand a class
of failure, the system should make that class impossible to
reproduce. v25.7.0.25 already paid the cost of understanding
"clipboard whitespace breaks Auth0 credentials"; trimming at read
time means we never debug it again, no matter who pastes what
where.

## Four-lens block (OBS-018)

This is a hotfix on internal plumbing; lenses are brief.

- **Teller / Analyst / Executive:** No user-visible change.
  Diagnostic endpoint is admin-only.

- **SOC:** New endpoint is gated by the existing
  `requireRole('admin')` guard. Returns env var metadata (lengths,
  first/last char) but never the actual values. The first character
  of a 64-character Client Secret is not enough to reconstruct it,
  and length-leak is acceptable trade-off for the diagnostic value.
  Live token-test attempt is logged in the same way other Auth0
  calls are (via the existing token cache infrastructure). Safe to
  leave in production.

## What ships in this release

### Server — `server/src/lib/auth0-management.js`

- New internal `readEnv(name)` helper — reads `process.env[name]`,
  returns `null` if absent, otherwise returns `.trim()`-ed value
- New exported helpers: `getEnvDomain()`, `getEnvClientId()`,
  `getEnvClientSecret()` — wrap `readEnv()` for the three Auth0
  env vars
- New exported helper: `describeEnvVar(name)` — returns
  `{ present, rawLength, trimmedLength, hadWhitespace, firstChar,
  lastChar }` for diagnostic reporting (does NOT include actual
  value)
- `isAuth0Configured()` rewritten to use the trimmed helpers
- `getManagementToken()` reads via trimmed helpers
- `managementRequest()` reads domain via trimmed helper

Net effect: every Auth0-bound credential is now `.trim()`-ed at
read time. Whitespace in any of the three env vars is silently
tolerated. The vars in Railway can have leading/trailing whitespace
and the system still works.

### Server — `server/src/routes/admin.js`

New endpoint:

```
GET /api/admin/debug/auth0-status
Response: {
  configured: boolean,
  env: {
    AUTH0_DOMAIN: { present, rawLength, trimmedLength, hadWhitespace, firstChar, lastChar },
    AUTH0_M2M_CLIENT_ID: { ... },
    AUTH0_M2M_CLIENT_SECRET: { ... },
  },
  tokenTest: {
    attempted: boolean,
    status: number (HTTP status from Auth0),
    success: boolean,
    body: object | string (Auth0 response, with access_token redacted on success),
    error?: string,
  },
  hint: string,
}
```

Admin-only via the router-level guard. No actual secret values in
the response. Live token test runs against Auth0 with the trimmed
env vars; if Auth0 succeeds, the response includes the granted
scopes (the actually useful confirmation).

## Verification — 90 seconds

### Step 1 — Hit the diagnostic endpoint

After Railway redeploys, in Codespaces (or any terminal):

```bash
# Get your JWT from browser dev tools → Network tab → any admin API
# request → Request Headers → Authorization → copy the value AFTER "Bearer "
read -s -p "JWT: " JWT; echo

curl https://3ffs-production.up.railway.app/api/admin/debug/auth0-status \
  -H "authorization: Bearer $JWT" | python3 -m json.tool

unset JWT
```

(Replace the URL with your actual server's Railway URL — the API
endpoint, not the client. From your earlier screenshots that's
something like `3fffs-server-production.up.railway.app` or similar.
You can find it on Railway → server service → Settings → Public
Networking.)

### Step 2 — Read the output

Three things to look at, in order:

**(a) `env.AUTH0_M2M_CLIENT_SECRET.hadWhitespace`** — if `true`,
that confirms the secret in Railway has leading or trailing
whitespace. The trim defenses mean it works anyway, but you can
clean up the Railway env var when you have a moment for cleanliness.

**(b) `tokenTest.success`** — if `true`, Auth0 accepted the
trimmed credentials and returned a token. Role grant should now
work in the modal. If `false`, look at `tokenTest.body` for the
exact Auth0 error.

**(c) `tokenTest.body.scope`** — when success is true, this should
list all six scopes:
`read:users update:users read:roles read:role_members create:role_members delete:role_members`

If any are missing, the M2M app authorization is incomplete; revisit
the Client Access scope checklist for the M2M app in Auth0
dashboard.

### Step 3 — Retry the role grant in the modal

If `tokenTest.success` is true, go back to `/admin/users`, open
your own user, click **+ Role → + manager**. Should succeed this
time, with the toast and modal update from v25.7.0.25.

### Step 4 — Paste me the diagnostic output

Either way (success or failure), paste me the JSON output of the
diagnostic endpoint. If success, that confirms the path is clean
and we move to v25.7.0.26. If failure, the body of `tokenTest.body`
tells us exactly what Auth0 is complaining about and we can target
the fix.

## Honest accounting

- This is a hotfix, not a feature. No new user-facing capability.
- The diagnostic endpoint stays in the codebase as a permanent
  diagnostic surface — useful any time the Auth0 integration
  misbehaves in the future.
- If `tokenTest.success` is true after this deploys but role grant
  still fails in the modal, the issue isn't credential-related and
  we'll need a separate investigation (likely client-side or
  endpoint-routing).
