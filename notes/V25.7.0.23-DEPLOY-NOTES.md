# v25.7.0.23 — User detail drill-down (Users & Invites Gap 2)

Second consecutive operational-tooling release. v25.7.0.22 closed the
Extend / Restore loop on the Invites tab; v25.7.0.23 closes the read
loop on the Users tab. Clicking any row on `/admin/users` now opens a
comprehensive per-user modal showing progress totals, manager
assignments in both directions, invite history (when applicable),
completed scenarios, the last 50 stage attempts, and the last 30 quiz
answers — all in one round-trip.

The Users tab was previously list-only: clicking a row did nothing.
That's fine when the question is "who's here?", but as the platform
moves toward reference-customer onboarding the more important
question becomes "is this specific person actually using it, and how
well?" The drill-down is the first surface that answers the second
question.

## Pedagogical insight (OBS-027)

**The list view answers "who's here?"; the drill-down answers "is
this person actually using it?" — and only the second question
matters for a behavior-change product.**

A list of users with rolled-up scenario counts is a vanity surface.
You can show a CRO at NCB or JNCB a screen with 200 names on it and
they will nod politely. What sells the next contract is being able
to point at one name on that list and say: "Here is the analyst we
onboarded six weeks ago. She has completed Reconnaissance and
Initial Access scenarios, scored 82% on related quiz questions, and
attempted three Phishing decision stages — getting two right and
one wrong. The wrong one was the variant where the smishing message
spoofs the tax authority instead of the bank, which suggests we
should weight that pattern more heavily in her next assigned
module."

That's the demo that closes the contract. None of it is possible
without per-user drill-down. The list view is the table of contents;
the modal is where the actual signal lives. This is also the
foundation for the eventual behavior-change evidence the pricing
analysis (Appendix A of the continuation file) flagged as the
critical "watch out for" item — KnowBe4 wins on "33% phishing
susceptibility drops to ~5% after 12 months." 3FFFs needs the same
shape of evidence, and it starts with being able to inspect one
user's trajectory at a time.

## Four-lens block (OBS-018)

This is admin tooling, so the four lenses adapt to "who benefits
from this surface":

- **Teller:** Not user-facing. No teller workflow change.

- **Analyst:** When a user shows low engagement on the list view,
  the drill-down lets fraud ops triage *why* — never logged in past
  the first session? Logged in but skipped quizzes? Got the same
  Initial Access decision wrong three times in a row? Each pattern
  suggests a different intervention (re-onboarding, manager nudge,
  weighted re-assignment). The recent-attempts list shows
  ✓/✗ inline so the failure pattern is visible at a glance without
  clicking into individual scenarios.

- **SOC:** Endpoint sits behind the same `requireRole('admin')`
  guard as every other admin route. No new attack surface. No PII
  is exposed beyond what the list view already showed (email, name,
  domain, role, last-seen) — the modal just adds engagement data
  that the user themselves already sees on their own progress page.
  Six parallel queries with `Promise.all`; no N+1 risk because the
  drill-down is one user at a time.

- **Executive:** The drill-down is the surface where 3FFFs's value
  proposition becomes legible to a buyer. Without this, "we have
  500 users on the platform" is an unsubstantiated claim. With it,
  "here is a representative user's actual trajectory through the
  framework, with the specific decision points where the curriculum
  caught a real misconception" is a sales asset. Pricing analysis
  Watch-out #2 (proof of behavior change) starts here.

## What ships in this release

### New endpoint

`server/src/routes/admin.js` — `GET /api/admin/users/:id`

Single round-trip drill-down. Six sub-queries dispatched in parallel
via `Promise.all`:

1. **user** — base node fields (id, email, name, domain, roles,
   firstSeenAt, lastSeenAt)
2. **bank** — `:MEMBER_OF` lookup → `{ id, displayName }` or null
3. **managers** — `(mgr)-[:MANAGES]->(this user)` with assignedAt
4. **manages** — `(this user)-[:MANAGES]->(report)` (only meaningful
   if the user has the manager role; client hides the section
   otherwise)
5. **invite** — matching `:Invite` by email, with full lifecycle
   fields including the `lastExtendedAt` / `extendedBy` audit added
   in v25.7.0.22
6. **progress** — totals (same shape as in `GET /users`), completed
   scenarios with timestamps, last 50 stage attempts with correct
   flag and parent scenario, last 30 quiz answers with question
   text and tactic id

All sub-queries are read-only (MATCH + RETURN, no MERGE / SET / DELETE).
OBS-021 N/A. OBS-025 preservation N/A — no writes.

Returns 404 if no user with that id exists.

### Frontend

`client/src/lib/api.js`
- New: `adminGetUser(id)` — GET `/api/admin/users/${id}`

`client/src/pages/AdminUsers.jsx`
- `UsersTab` now tracks `selectedUserId` state. Rows are clickable
  (cursor:pointer, paper hover background) and open the modal by id.
- New component `UserDetailModal` — fetches the detail payload on
  mount, renders loading / error / body states, dismisses on
  click-outside or ESC key.
- New component `UserDetailBody` — composes the sections from the
  payload. Sections render conditionally on the data:
  - Header always present (name, email, role pills, bank, first/last
    seen)
  - Progress totals always present (4-stat grid: scenarios, stages,
    quizzes, accuracy %)
  - Manager assignments always present (shows "No manager assigned"
    when empty; shows "Manages" sub-section only if user has the
    manager role)
  - Invite history rendered only when an `:Invite` matched the user's
    email (auto-allowlisted users have no invite, so the section is
    hidden rather than empty)
  - Completed scenarios section hidden if zero completed
  - Recent stage attempts section hidden if zero attempts; capped at
    max-height 320px with overflow scroll
  - Recent quiz answers section hidden if zero answers; capped at
    max-height 280px with overflow scroll
  - Empty-state footer for users with literally zero engagement
    (logged in but never attempted anything) — clearer than four
    hidden sections in a row
- Helper components: `DetailSection`, `StatBox`
- New styles: `subLabelStyle`, `fieldLabelStyle`, `personPill`,
  `listRowStyle`

### Step 6 — Cross-check Cypher (optional, 30 seconds)

In Aura browser, pick any non-trivial user id from the list and run:

```cypher
MATCH (u:User {id: $userId})
OPTIONAL MATCH (u)-[c:COMPLETED]->(sc:Scenario)
OPTIONAL MATCH (u)-[a:ANSWERED]->(q:Quiz)
OPTIONAL MATCH (u)-[att:ATTEMPTED_STAGE]->(st:Stage)
RETURN
  count(DISTINCT sc) AS scenariosCompleted,
  count(DISTINCT q)  AS quizzesAnswered,
  sum(CASE WHEN a.correct THEN 1 ELSE 0 END) AS correctAnswers,
  count(DISTINCT st) AS stagesAttempted
```

**Pass:** the four counts match the modal's stat boxes for that user.

## Honest accounting

Non-goals — explicit so they don't silently land in scope:

- **No write actions in the modal.** This release is read-only. You
  can't reset a user's progress, change their role, or revoke their
  invite from the modal. Role changes still happen in the Auth0
  dashboard; invite revocation still happens on the Invites tab.
  Adding write actions is a future surface — likely worth waiting
  until you've used the read-only version against real users to see
  what writes you actually want.
- **No charts.** The progress totals are numeric stat boxes, not bar
  or line charts. With at most a handful of test users in the system
  today, charts would be misleading visual noise. Re-evaluate after
  the first cohort of 50+.
- **No per-scenario drill-down from the modal.** Clicking a completed
  scenario row doesn't open the scenario itself. Possible future
  enhancement; the data is there in `scenarioId`.
- **No export.** Can't CSV-export a user's full attempt history yet.
  Adjacent to Gap 3 (bulk operations); revisit when you actually need
  to take user data out of the platform for a customer.
- **Last 50 attempts and last 30 quiz answers are hard caps.** Power
  users will eventually exceed both. The lists scroll within their
  containers; older history is queryable from Aura directly. If the
  caps become limiting, paginate at that point — premature
  optimization to do it now.
