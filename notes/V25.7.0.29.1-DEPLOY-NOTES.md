## What was wrong (post-mortem)

The `MultiActorSequenceAnimation` engine reads these fields off
each stage object:

```
stage.id
stage.label              ← short eyebrow text
stage.title              ← stage title
stage.caption            ← long descriptive paragraph
stage.durationMs         ← duration in ms (NOT 'duration')
stage.messages[]         ← array of {id, fromActor, toActor, kind, label, suspicious?, audio?}
stage.actorStateChanges  ← optional {actorId: newState}
stage.revealedSignalIds  ← array of signal IDs revealed at this stage
stage.finalHeadline      ← required on the LAST stage only
```

And the default export shape is:

```js
export default {
  meta: { techId, techName, ... },
  engine: 'multi-actor-sequence',  // REQUIRED top-level field
  stages: STAGES,
  controls: CONTROLS,
  signals: SIGNALS,
  actors: ACTORS,
}
```

The v25.7.0.29 scene files used:

| Field | What I shipped | What the engine reads |
|---|---|---|
| Stage caption | `body` ❌ | `caption` |
| Actor flow | `actorActions` (with embedded state) ❌ | `messages` (with separate `actorStateChanges`) |
| Stage duration | `duration` ❌ | `durationMs` |
| Eyebrow | (missing) ❌ | `label` |
| Signal reveal | separate `signals` array with `stage` field ❌ | per-stage `revealedSignalIds` array |
| End screen | (missing) ❌ | `finalHeadline` on last stage |
| Engine field | (missing) ❌ | `engine: 'multi-actor-sequence'` in default export |
| Meta keys | `techniqueId`, `techniqueName`, `totalStages` ❌ | `techId`, `techName`, `stageCount`, `totalDurationMs`, `scenarioContext` |

All three files rebuilt from scratch using
`silentAlarmScenes.jsx` (the working v25.7.0.27 reference) as
the structural template. Same content, same scenarios, same
characters (Renee Patterson · NCB, Tariq Mohammed · Republic
Bank, Marlon Grant · JN Bank), same pedagogical insights,
same detection controls — just with field names the engine
actually reads.

## Pedagogical insight (OBS-027 — methodological, post-mortem)

**Verify the engine contract directly, not transitively.**

The original v25.7.0.29 scene files were authored by reading
the v25.7.0.28 `cardTestingScenes.jsx` exports through `grep`
and inferring shape from named-export patterns. That approach
captured the *outline* (ACTORS / CONTROLS / SIGNALS / STAGES /
META) but missed the *contract* (which fields each stage object
must contain for the engine to render it). The build succeeded;
the engine even iterated correctly through the stages array
length; nothing flagged the contract mismatch until a real
click triggered render.

The general lesson: **when authoring data that an engine
consumes, read the engine — not just the data**. The v25.7.0.27
engine code (`MultiActorSequenceAnimation.jsx`) names every
field it reads off each stage. Confirming each field name in
the engine before authoring scene data costs ~5 minutes; missing
the contract costs a release-and-hotfix cycle.

Process change going forward: for any future scene-file
authoring, the very first action is to read the relevant
engine source file and enumerate the stage fields it accesses
(grep for `stage.` and `currentStage.` patterns). The
engine source IS the contract. Reading existing scene files
is a useful complement but not a substitute.
