# v25.7.0.22 — Extend / Restore invites (Users & Invites Gap 4)

First non-animation release in the v25.7.0.x series. Pivots away from
Phase 2 animation work to close an operational friction point Omar
flagged after Phase 1 wrapped at v25.7.0.21: the only way to keep an
expired invite alive today is to re-enter the full +Invite User modal,
re-typing roles and notes.

This release adds a single-responsibility lifecycle endpoint —
`POST /api/admin/invites/:id/extend` — and surfaces it in the Invites
table as a context-aware action: `[Extend ▾]` for active and expired
rows, `[Restore ▾]` for revoked rows. Same server endpoint backs both
labels; the popover quick-picks +1 day / +7 days / +30 days / Never
expires, with a custom-hours input for the long tail.

Empirical motivation: of the two test invites issued to date
(beckfordcandice@gmail.com, eljagg@gmail.com), both lapsed at the
default 7-day expiry with `useCount = 0`. Manual re-issue would mean
re-typing the role and the personal note for each — friction that
compounds the moment Omar starts onboarding reference customers.

## Pedagogical insight (OBS-027)

**Lifecycle state changes deserve dedicated endpoints, not parameter
overloading on upserts.**

The pre-existing `POST /api/admin/invites` was doing double duty:
create-or-re-issue, via `MERGE (i:Invite {email: $email})`. That works
— it correctly upserts — but it forces the caller to re-supply every
field every time, which means the UI has to reopen the full +Invite
User modal even when the admin's actual intent is "just give them
more time." The mental model and the implementation diverged.

`POST /api/admin/invites/:id/extend` is the inverse: a single
endpoint with a single concern. It changes only `expiresAt`, clears
`revokedAt`, and stamps `lastExtendedAt` + `extendedBy`. Email,
roles, notes, original `invitedAt`, `useCount`, `consumedAt`,
`lastUsedAt` are all preserved. Because the cleared `revokedAt` also
restores access, "Restore" is the same operation — only the UI label
changes. The endpoint stays small and the admin's intent stays
visible in the audit trail.

The principle generalizes: any time a UI action becomes "open a heavy
form to change one thing", that's a smell that the API is missing a
narrower verb. The fix is usually a new sub-route, not a new modal.

## Four-lens block (OBS-018)

This is admin tooling, not a fraud-pattern animation, so the four
lenses adapt accordingly.

- **Teller:** Not user-facing. No teller workflow change. Frontline
  staff continue to be auto-provisioned via the barita.com /
  jncb.com / etc. domain allowlist; the Invites tab remains
  reviewer-and-partner-only.

- **Analyst:** New audit fields (`lastExtendedAt`, `extendedBy`)
  visible in the table as a "↻ extended {time}" sub-label below the
  status pill. Useful when fraud ops eventually needs to answer
  "who has been keeping access to 3FFFs alive past their original
  expiry?" — a real compliance question once the platform has more
  than one bank tenant. The audit is queryable from Neo4j directly:
  `MATCH (i:Invite) WHERE i.lastExtendedAt IS NOT NULL RETURN
  i.email, i.lastExtendedAt, i.extendedBy ORDER BY
  i.lastExtendedAt DESC`.

- **SOC:** New endpoint sits behind the same `requireRole('admin')`
  guard as every other admin route. No new attack surface for
  non-admin users. The soft-revoke pattern is preserved — revoked
  invites are not deleted, they're flagged; restoring them re-uses
  the same node id, so any audit log retains continuity. Custom-hours
  input on the popover validates as a positive finite number on both
  client and server.

- **Executive:** Reduces operational drag during the upcoming
  reference-customer phase. When Omar invites a CRO at NCB or
  JNCB to evaluate the platform and they don't get to it inside
  7 days (which the data says is the realistic case — both extant
  invites expired with zero use), the recovery path is now one
  click instead of full-modal re-entry. Small but compounding on
  the way to the first paid contract.

## What ships in this release

### New endpoint

`server/src/routes/admin.js` — `POST /api/admin/invites/:id/extend`

```
Body: { expirationHours }
  expirationHours: positive number (hours from NOW until new expiry),
                   or null for "never expires"
Response: { invite: { ... full invite shape including audit fields ... } }
404 if id not found
400 if expirationHours is non-null and not a positive finite number
```

Cypher (NOT a MERGE, so OBS-021 ordering N/A — pure MATCH + SET):

```cypher
MATCH (i:Invite {id: $id})
SET i.expiresAt      = $newExpiresAt,
    i.revokedAt      = null,
    i.lastExtendedAt = timestamp(),
    i.extendedBy     = $me
RETURN ...
```

`GET /api/admin/invites` updated to also return `lastExtendedAt`,
`extendedBy`, and `extendedByEmail` (via `OPTIONAL MATCH (extender:User
{id: i.extendedBy})`) so the client can render the audit sub-label
without a second round-trip.

### Schema (additive — no migration needed)

Two new properties on `:Invite` nodes:

- `lastExtendedAt` — `timestamp()` of most recent extend/restore
- `extendedBy` — `:User.id` of the admin who extended

Neo4j is schemaless on properties, so older invites (created before
this release) will simply have these as `null`. The UI handles `null`
gracefully — no sub-label rendered, no "extended by" line.

OBS-025 preservation: existing endpoints unchanged in behavior. Node
and relationship counts unchanged. Property mutations are scoped to
the four named fields above, all on a single MATCHed node.

### Frontend

`client/src/lib/api.js`
- New: `adminExtendInvite(id, body)` — POSTs to
  `/api/admin/invites/${id}/extend`

`client/src/pages/AdminUsers.jsx`
- New constant: `EXTEND_OPTIONS` (+1d / +7d / +30d / Never)
- New state in `InvitesTab`: `extending`, `extendAnchor`
- New handlers: `openExtendPopover`, `closeExtendPopover`, `handleExtend`
- Status cell: now renders "↻ extended {time}" sub-label when
  `inv.lastExtendedAt` is non-null. Hover for absolute timestamp +
  extender email.
- Action cell: replaced `{!isDone && <Revoke>}` with context-aware
  rendering — `[Restore ▾]` for revoked rows, `[Extend ▾] [Revoke]`
  for active rows, `[Extend ▾]` only for expired rows.
- New component: `ExtendPopover` — fixed-position floating menu
  anchored via `getBoundingClientRect` to the trigger button.
  Quick-pick options + custom-hours input. Click-outside dismisses.
  Header copy flips between "Extend expiry" and "Restore with new
  expiry" based on `invite.revokedAt`.
- New styles: `extendBtn` (subtle bordered pill, paired with the
  underlined-link Revoke), `popoverItemStyle` (full-width left-aligned
  menu items).

### Step 2 — Extend an expired invite (45 seconds)

Both extant test invites (beckfordcandice@gmail.com, eljagg@gmail.com)
should currently show as **Expired**.

1. In the Action column on either expired row, confirm `[Extend ▾]`
   button is visible (and `Revoke` is NOT shown, since revoking an
   already-lapsed invite is meaningless)
2. Click `[Extend ▾]`
3. Popover opens below the button, header reads **"Extend expiry"**
4. Click `+30 days`
5. Popover closes, toast appears: *"Extended {email} — expires
   {absolute timestamp}."*
6. Status pill flips from **Expired** (orange) to **Pending** (red)
7. Below the pill, new sub-label appears: **↻ extended just now**
8. Expires column updates to show the new relative time (e.g. "in 30d")

**Pass:** status flips, sub-label appears, expires column updates,
no errors.

### Step 3 — Revoke and Restore the same invite (45 seconds)

1. On the now-Pending row, click `Revoke`. Confirm the browser
   `confirm()` dialog. Status flips to **Revoked**.
2. Confirm Action column now shows `[Restore ▾]` only (Revoke
   is gone — can't revoke what's already revoked)
3. Click `[Restore ▾]`. Popover opens, header reads **"Restore
   with new expiry"**.
4. Click `+7 days`
5. Popover closes, toast: *"Restored {email} — expires {timestamp}."*
6. Status flips back to **Pending**, sub-label updates to **↻
   extended just now**
7. Action column returns to `[Extend ▾] [Revoke]` pair

**Pass:** the same endpoint cleanly handles both extend-from-expired
and restore-from-revoked, label flips correctly per state.

### Step 4 — Custom hours + Never expires (30 seconds)

1. Click `[Extend ▾]` on any non-revoked row
2. In the **Custom hours** field at the bottom of the popover, type
   `72` and click `Apply` (or press Enter)
3. Toast: *"Extended {email} — expires {timestamp 3 days from now}."*
4. Click `[Extend ▾]` again, this time pick `Never expires`
5. Toast: *"Extended {email} — no expiration."*
6. Expires column shows **Never** in faint text

**Pass:** custom input validates and applies; null expirationHours
correctly means never-expires through the full stack.

## Honest accounting

What this release does NOT do — explicit non-goals so they don't
silently land in scope:

- **Email delivery is still not wired.** Extending an invite does not
  email the recipient that their access has been refreshed. Gap 1
  remains in the backlog.
- **Bulk extend is not in this release.** With 2 invites total, the
  one-at-a-time UX is fine; the friction of one-at-a-time also keeps
  honest pressure on the question of whether the default expiry
  should be longer. Re-open if invite count crosses ~20.
- **No magic-link flow.** The signup URL remains
  `window.location.origin`; the recipient's Auth0 login matches by
  email. Magic links would be a separate, larger piece of work and
  aren't needed to close this friction loop.
- **Manager assignments tab unchanged.** It works correctly; the
  "0 managers" empty state is a config issue (no users have the
  `manager` role in Auth0 yet), not a code defect.
- **No new pedagogical content, no new animation, no audio change.**
  Pure operational tooling.
