# v25.7.0.27.6.1 — Hotfix: missing useEffect import in PositioningTwoViews

Single-line fix to a runtime crash introduced in v25.7.0.27.6.

`PositioningTwoViews.jsx` was importing `{ useState, useMemo,
useCallback }` from React. v25.7.0.27.6 added a `useEffect`
for the playback interval but didn't update the import. Vite
doesn't catch undefined identifiers at build time — the
component built successfully and shipped, then crashed at
runtime when a user clicked "The disguise" on the Positioning
tactic with `ReferenceError: useEffect is not defined`. The
ErrorBoundary caught the crash and rendered "A component
crashed while rendering" — exactly what it's there for, but
the bug is real.

`PositioningTimeline.jsx` ("How attackers wait") was unaffected
because it already imported `useEffect` from a previous version.

This release adds `useEffect` to the import statement of
`PositioningTwoViews.jsx`. One line. No other changes.

## Pedagogical insight (OBS-027 — methodological)

**Add the import the same turn you add the call site, not the
turn after.**

The bug is small enough that the post-mortem is the value. When
adding new code that depends on a hook the file doesn't already
use, the import-statement update has to land in the same edit as
the new code. Three reasons it didn't:

1. The file already had several React hooks (useState, useMemo,
   useCallback). The import line *looked* full enough that it was
   easy to assume useEffect was already there. It wasn't.
2. Vite + React's runtime model means undefined identifiers don't
   surface until the component mounts. There's no compile-time
   check. The build succeeded; the bug was invisible until a real
   click triggered render.
3. PositioningTimeline (the file edited just before) already
   imported useEffect, which I had visually confirmed. I carried
   that confirmation forward to PositioningTwoViews without
   re-checking — they're "the same kind of file" only at a high
   level; their import lines are independent.

The general lesson: **when reusing a pattern across two files,
verify each file's imports independently.** Pattern-reuse is
mostly safe, but the import statement is one of the things that
varies file-to-file. Whenever a hook is added to a useEffect call
site, the import update has to be part of the same diff.

Process fix going forward: when adding a hook to a component
file, the very first action after writing the new code is to
view the import line and confirm the hook is in the named
imports. If it isn't, add it before doing anything else.
