# v25.7.1.2 — Auth race-condition hotfix

**Release type:** Client-only patch on top of v25.7.1.1. Two files. No server, no schema, no migration, no new dependencies.
**Driver:** A latent race condition in `auth.jsx` ↔ `api.js` was exposed by the v25.7.1 client deploy. Symptom: on hard-refresh of Home (or any page that fires API calls in `useEffect`), 4–6 endpoints would return 401 because the request fired before AuthGate wired up the token getter. The user appeared signed in (avatar in nav, `o.mcleod`), stats reset to zeros, and the new Daily Signal panel showed "Authentication required" instead of a question.
**Sequence:** v25.7.0.2.2 → v25.7.1 (engagement layer) → v25.7.1.1 (vite loadEnv path fix) → **v25.7.1.2 (this release, auth race-condition fix)**.

This bug existed before v25.7.1 — but was masked because most page-load API calls happened to hit a Network or React tick that landed AFTER the AuthGate effect. The v25.7.1 deploy added more components that fire API calls in useEffect on Home (`DailySignal`, `ReviewQueue`), which pushed the timing window the wrong way and made the race surface reliably.

## The bug, in one sentence

`AuthGate` calls `setTokenGetter()` inside a `useEffect`, which fires AFTER children's `useEffect`s mount. So `Home.jsx` calls `api.getProgress()` / `getDailySignal()` / `getReviewQueue()` while `tokenGetter` is still `null`, the request goes out without an `Authorization` header, and the server returns 401.

## The fix (two parts)

**`api.js`** — `request()` now waits up to 3 seconds for `setTokenGetter()` to be called before firing. A queue of `pendingResolvers` is drained when the setter is finally wired; the 3-second timeout is a safety net that lets the request go out un-authed (preserving old behaviour for genuinely-public endpoints) so a misconfigured Auth0 setup never hangs the UI forever. Same fix applied to the `downloadCertificate` function which has its own auth-attaching code path.

**`auth.jsx`** — `AuthGate` now also calls `setTokenGetter()` synchronously during render the moment `isAuthenticated === true`, in addition to the existing useEffect (kept as a backstop). The setter just stores a function reference, so calling it during render is idempotent and safe. This means by the time React commits the render and children's useEffects fire, `tokenGetter` is already wired up.

The two halves work together: the sync render call removes the race in the common case, and the api.js queue handles the edge case where a child's effect somehow fires before AuthGate's render runs.

## Honest reflections

**This bug should have been caught in code review of the original `auth.jsx` whenever it was first written.** The `useEffect`-based wiring is a textbook React-effect-ordering pitfall, especially in a tree where children also have effects that depend on the wired state. I didn't introduce it in v25.7.1 — but I also didn't catch it when adding three new useEffect-based components to Home that exercised it more heavily.

**The fix uses a render-time mutation, which is normally bad practice.** I went with it anyway because (a) `setTokenGetter` is idempotent — re-calling it just overwrites a reference to the same function — and (b) the alternative (refactoring everything to use Auth0's `useAuth0` hook directly inside `api.js`) would touch every API call site in the codebase. The render-time call is paired with the api.js queue, so even if a child somehow effects before this render commit, the queue catches it.

**OBS-022 (refined again):** Race conditions in auth wiring are particularly insidious because they only manifest under certain timing conditions. From now on:
> When wiring authentication state into a non-React module (api client, fetch helpers, etc.), do it synchronously during render of the gate component AND in a useEffect, not just one or the other.
