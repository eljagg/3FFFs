# v25.7.0.2 — Visualization registry + first interactive viz (ISS-023)

**Release type:** Spike. New schema entity (`Visualization`), new edge (`VISUALIZES`), new endpoint surface, one user-facing visualization, one admin preview page.
**Driver:** ISS-023 — feedback that the app reads as too static. The Gemini "Reconnaissance" visualization research crystallised the gap. This release ships the foundation (registry + renderer + telemetry contract) plus one anchor viz, deliberately small, so we can react to real Railway behaviour before committing to a wider rollout in v25.7.1.
**Sequence:** Inserted ahead of Comments. v25.7.1 (Comments / ISS-018) is now slipped to v25.7.2; v25.7.1 is reserved for the wider visualization rollout.

This is a foundation drop. The architectural commitment: visualizations are **code that reads from a thin graph registry**, not configuration stored in Neo4j. The registry says *which* viz attaches to *which* entity for *which* roles; the React component decides *how* to render. Adding a new viz instance against an existing kind = one seed entry. Adding a new viz kind = one component + one switch case. No schema migration in either case beyond the seed.

## Summary

**Schema additions:**
- `Visualization` node label (id, kind, title, subtitle, roles[], order, config (JSON string), createdAt, updatedAt)
- `VISUALIZES` edge from `Visualization` to any of `Tactic` / `Technique` / `Concept` / `Stage` / `FrameworkPhase` / `MitreTactic` / `MitreTechnique` (allowlist enforced server-side)
- `INTERACTED_WITH` edge from `User` to `Visualization` (count, firstAt, lastAt, lastEventType — aggregate, not full event log)
- New uniqueness constraint on `Visualization.id`

**Server-side:**
- `server/src/routes/visualizations.js` — split into a public router (GET endpoints, no auth) and an authed router (POST telemetry endpoint, behind requireAuth). Same pattern as `/api/framework`.
- `server/src/seed/data/visualizations.js` — registry seed. v25.7.0.2 ships one entry: `VIZ-RECON-KILLCHAIN`, kind `kill_chain_grid`, attached to `Tactic { id: 'TA0043' }`, audience `analyst / soc / executive`.
- `server/src/seed/add-visualizations.js` — idempotent migration with idempotent-aware preservation check (follows `add-sc014.js` canonical pattern). Validates seed-entry shape, soft-warns on >2 KB config blobs, hard-fails on attachment to non-existent entities.
- `server/src/index.js` — mounts the public router before the auth wall, the authed router after.
- `server/package.json` — `migrate:visualizations` script.

**Client-side:**
- `client/src/components/visualizations/VisualizationRenderer.jsx` — dispatcher. Switches on `viz.kind` to the implementing component. Emits `viz_opened` telemetry on first render per (viz id, role). Audience-gates output (suppresses for roles not in `viz.roles[]`). Hands an `onEvent(type)` callback into the inner component so individual viz kinds don't have to know about the API.
- `client/src/components/visualizations/KillChainGrid.jsx` — first viz kind. Three-column kill chain, click-to-trace path highlighting, defense toggles that visibly mitigate nodes (the falsifiability hook). Uses CSS variables exclusively, no Tailwind. Inline SVG icons (consistent with AppShell).
- `client/src/pages/AdminVisualizations.jsx` — internal preview page at `/admin/visualizations`. Lists every registered viz with metadata, renders each in isolation against the live graph, allows per-viz role override so we can verify audience gating without changing the global role pill.
- `client/src/pages/Framework.jsx` — when an expanded tactic has at least one visible-to-audience visualization, the body switches to a two-column grid (text/takeaway/techniques on the left, viz panel on the right). Tactics with no viz fall back to the original single-column layout — no empty right pane. Per-tactic viz cache so repeat expand/collapse doesn't re-fetch.
- `client/src/lib/api.js` — three new methods (`listVisualizations`, `getVisualizationsFor`, `emitVizEvent`).
- `client/src/App.jsx` — `/admin/visualizations` route registered behind `<AdminRoute>`. **Not added to top nav** (see "Operational notes" below).

**Migration:** `migrate:visualizations` is fresh-seed (+1 Visualization node + 1 VISUALIZES edge) on first run, idempotent on re-run.

## Architectural decisions worth re-stating

**Visualizations are code, not data.** The graph stores wiring (which viz attaches to which entity, which roles see it, what config it consumes). The React component decides how to render. We do not store SVG, HTML, or rendering logic in Neo4j. This is the same trap as Pattern B in our multi-tenancy thinking — configurability theater that pretends to be data. Recognise it; refuse it.

**Config blob has a soft 2 KB cap.** The seed migration warns if any entry exceeds it. Past that size, the right move is almost always to push structural data back into the graph as nodes and edges. Config holds presentation choices (which nodes to highlight, ordering hints, color semantics). It does not hold graph data.

**The telemetry contract is at the wrapper layer.** `VisualizationRenderer` calls `api.emitVizEvent(vizId, type)` on behalf of inner components. Inner components receive an `onEvent(type)` prop and never know about the endpoint. This means future viz kinds inherit telemetry for free, and the event vocabulary is centralised (`VALID_EVENT_TYPES` server-side).

**Falsifiability is the design gate.** What makes a visualization pedagogically interesting is that user input *changes the conclusion*. In `kill_chain_grid`, defense toggles visibly break the chain. Every future viz kind has to have an analogous hook. If a viz can't fail under user input, it's decoration with click handlers, and we don't ship it.

**One viz kind ≠ one viz instance.** v25.7.0.2 ships one of each, but the architecture separates them deliberately. v25.7.1 will ship more `:Visualization` *instances* (one per F3 tactic, plus concept-attached) using mostly the same kinds, plus probably 2-3 new kinds for specific contexts (phishing email dissection, fund-flow Sankey, exec ROI calculator).

**Reconnaissance content model = teaching abstraction (Option A).** F3's Reconnaissance tactic only has two indexed techniques — not enough to render a meaningful kill chain from real graph data. Rather than pollute the F3 graph with nodes that aren't part of MITRE's curated F3 subset, the Reconnaissance kill-chain is authored as a teaching abstraction with explicit framing in the panel ("Illustrative — adversary lifecycle for reconnaissance. The F3 techniques formally indexed for this tactic appear below."). The technique IDs cited in node labels (T1593, T1589, T1110…) are real ATT&CK IDs so curious users can drill out to attack.mitre.org.

## Interactivity Design (OBS-018)

This is the first release where the four-lens block has explicit asymmetry by design — only three of four roles get the new viz on TA0043. Documenting the asymmetry (rather than sneaking it in) is the discipline.

### Frontline / Teller

**No change for v25.7.0.2.** Tellers expanding `TA0043 Reconnaissance` see the same single-column layout as before — description, executive takeaway, the two F3 technique cards.

**Why deferred:** the kill-chain framing (MITRE technique IDs as node labels, defense toggles framed as enterprise controls) is SOC-and-up content. A teller-tuned reconnaissance interaction looks different — closer to "spot the social-engineering signals in this customer interaction" than a kill-chain grid. That's a separate viz kind, queued for v25.7.1.

### Fraud Analyst

**Sees the kill-chain on TA0043.** Hover/click semantics let an analyst trace which reconnaissance vectors feed which downstream attacks — useful as case-context priming when investigating e.g. an account takeover.

### SOC / Cyber Team

**Sees the kill-chain on TA0043.** Primary audience for this viz. The defense toggles are explicitly framed as enterprise controls (MFA, network segmentation, lookalike-domain monitoring) and the accompanying hint text ties each defense to its mitigation rationale. Functions as a kill-chain dependency map.

### Risk Manager / Exec

**Sees the kill-chain on TA0043.** Less direct relevance than for SOC, but the "defenses break the chain" framing is the right shape for budgeting conversations — it makes individual control investments visibly causal. An exec-tuned variant (e.g. defense-cost vs residual-risk projection) is in the v25.7.1 backlog as a separate kind.

**Discipline note:** Tellers being deferred on the FIRST viz to ship is the four-lens block at work. We'd rather ship three lenses well than four lenses unevenly. v25.7.1 will close the teller gap with a teller-tuned viz kind, and document its three-out-of-four for the other roles.

## Operational notes

**The `/admin/visualizations` page is not in the top nav.** Bookmarkable URL only. Deliberate — it's a dev tool, and adding nav entries is irreversible in user habit. If you find yourself going there many times a day, we can promote it in v25.7.1 via a small admin sub-nav, or surface a link from `/admin/users`.

**No new env vars, no new external services.** Same Railway deployment, same Neo4j, same Auth0.

## Honest tradeoffs

**The win:** the spike is small, contained, and end-to-end. Server seed → endpoint → renderer → component → page integration → admin preview. Every piece of plumbing v25.7.1 needs is exercised by v25.7.0.2 against real production traffic.

**The risk that landed:** if the role-conditional layout (two-column-with-viz vs single-column-without) reads as inconsistent UX rather than intentional asymmetry, we'll know quickly because tellers will see the difference between TA0043 (no viz) and other tactics (no viz) as identical, while analysts/SOC/exec will see one tactic light up and the rest stay flat. We accept that for the spike — the alternative is shipping a viz to all four roles before we know if the SOC-tuned one even works.

**The risk we are NOT taking:** we are not yet committing to a viz density target ("every tactic must have one"). v25.7.1 will decide that based on what TA0043 actually demonstrates. If kill-chains are great for half of F3 and bad for the other half, the right answer is mixed kinds or mixed coverage, not forced uniformity.

**The risk we are explicitly taking:** the Reconnaissance viz cites real ATT&CK technique IDs (T1593, T1110, etc.) that aren't in our F3 graph. Mitigation is the in-panel framing note. If a viewer screenshots only the viz body and assumes those IDs are F3-indexed, we have a small but real misrepresentation risk. Acceptable for the spike; revisit if it becomes a real misread.
