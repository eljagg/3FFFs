# v25.0 Deployment Notes — Frameworks foundation (Phase 1 of 5)

**This release is foundation only — no UI changes, no scenario changes, no
visible difference to the analyst until v25.1 ships.** What it does is extend
the Neo4j graph with the AASE/CBEST/TIBER-EU/iCAST ontology, expose a read-only
API at `/api/frameworks/*`, and seed the AASE content from the ABS Singapore
guidance (with stub data for the other three frameworks).

If you deploy this and don't run the migration script, nothing breaks — the new
endpoints just return empty results. The migration is the meaningful step.

Railway will auto-deploy the server. **At this point /api/frameworks works but
returns empty results because the migration has not run yet.** That is expected.

## Then run the migration

The migration is a one-shot Node script that talks to Neo4j Aura. It does NOT
run automatically on deploy — you trigger it explicitly so we don't accidentally
re-seed every time the server restarts.

**Option 1 — Railway CLI (preferred):**

```bash
railway run --service server npm run migrate:frameworks
```

This injects the production env vars (NEO4J_URI, NEO4J_USER, NEO4J_PASSWORD)
and runs against the live Aura instance. You'll see step-by-step progress:

**Option 2 — Codespace + Aura directly (fallback if Railway CLI is misbehaving):**

```bash
cd /workspaces/3FFFs/server
NEO4J_URI="neo4j+s://2179b314.databases.neo4j.io" \
NEO4J_USER="2179b314" \
```

The migration is safe to re-run. Every node and relationship uses MERGE keyed
on stable IDs, so re-running just refreshes properties — it never duplicates.

## Rollback

If the migration goes sideways: `git revert HEAD && git push` reverts the code
side. The graph nodes added by the migration stay in Aura but are isolated —
no existing query touches them. To remove them too:

```cypher
MATCH (n) WHERE n:Framework OR n:FrameworkPhase OR n:Role OR n:Deliverable
       OR n:Concept OR n:ThreatActor OR n:Regulator
DETACH DELETE n;
```

The pre-existing F3 / Scenario / Stage / User / Quiz graph is completely
untouched by this migration — the preservation check at the end of the
migration script confirms this with a hard assertion.

## Why no UI in this release

Two reasons:
1. **Validation isolation.** A schema migration on a live Aura instance is the
   highest-risk part of this whole 5-phase build. Shipping it alone, with
   exhaustive preservation checks and zero UI dependencies, means if anything
   breaks we know exactly where.
2. **Reuse.** v25.1 (the SC010 scenario) and v25.4-5 (the Frameworks page)
   both consume these endpoints. Shipping the API once and consuming it twice
   is cleaner than mixing the schema work into the first feature build.
