# v25.4.1 Deployment Notes — Hotfix: AASE phases query + nav rename

**Two issues from v25.4 production testing, fixed together.** No migration.
No new components. Surgical edits to two existing files.

## Issues fixed

### ISS-006 — Cypher error on AASE phases query

**Symptom:** opening `/frameworks` and selecting AASE produced the error
"In a WITH/RETURN with DISTINCT or an aggregation, it is not possible to
access variables declared before the WITH/RETURN: p (line 9, column 16
(offset: 358)) ' ORDER BY p.order'".

**Root cause:** the `/api/frameworks/:id/phases` endpoint had a trailing
`ORDER BY p.order` AFTER the `RETURN p, collect(DISTINCT ...)` projection.
Cypher does not allow referencing variables (`p` here) past an
aggregating projection — `collect()` is the aggregation, and after it
runs, `p` is no longer available in scope. The query was authored in
v25.0, never invoked in production until v25.4 added the consumer
component, and broke the moment it was first hit.

**Fix:** removed the trailing ORDER BY. Sorting still works because the
WITH on the line above already orders rows before they reach
collect(DISTINCT). The `framework-detail` query (`/api/frameworks/:id`)
uses the same pattern correctly and was not affected; only
`/:id/phases` had the bug.

**Why this didn't fail in development:** the route was authored ahead of
its consumer. Without a smoke test against every endpoint at deploy
time, the bug sat dormant until v25.4 wired up `Frameworks.jsx` and the
client made the first real call. Lesson noted — future releases that
add new endpoints will include a curl-level smoke test in the deploy
notes.

### ISS-007 — Nav naming collision and ordering

**Symptom:** the v25.4 nav had two adjacent entries called "Framework"
(F3 fraud) and "Frameworks" (cyber resilience) with no visual distinction.
User reported confusion.

**Fix:** rename and reorder the nav. New layout:

```
Home · Scenarios · My Coverage · Matrix · F3 Framework · Explorer · Cyber Frameworks · Play · Quiz · Tutor
                                          └────────── reference cluster ──────────┘
```

- "Framework" (singular) renamed to "**F3 Framework**" — explicit about
  what content it covers
- "Frameworks" (plural) renamed to "**Cyber Frameworks**" — disambiguates
  from F3
- "Explorer" moved to sit between the two — the three reference pages
  now cluster together with Explorer as the bridge (it's also a
  reference page, so the grouping is logical)

URL paths are unchanged: `/framework` and `/frameworks` and `/explorer`
remain. Only the visible labels move. Existing bookmarks, deep links,
and the role-pill simulation feature all continue to work.

## Lessons learned (added to deploy practice)

Added to my own working notes:
- **OBS-010:** When authoring server endpoints ahead of their UI
  consumers, include a curl-level smoke test in the deploy notes that
  hits the endpoint and validates shape. Caught here only because the
  user surfaced it in production — that should not be the testing
  surface.
- **OBS-011:** Aura's Cypher version is strict about post-aggregation
  variable scope. When future queries include `collect(DISTINCT ...)`,
  do not reference pre-projection variables in a trailing ORDER BY.
  Sort with WITH before the projection.

These observations carry into v25.5 and beyond.
