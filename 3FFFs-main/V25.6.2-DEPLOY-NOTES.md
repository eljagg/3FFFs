# v25.6.2 — CBEST per-framework concept summaries (ISS-014)

**Release type:** Property enrichment + ConceptSidebar four-lens upgrade
**Driver:** ISS-014 — fill in the `pending: true` CBEST framework summaries on existing concept nodes
**Source of truth:** Bank of England, *CBEST Threat Intelligence-Led Assessments — Implementation Guide*, 2024 edition (PRA, CC-BY 4.0)

This release closes out the v25.6 freshness pass arc. Where v25.6.0 corrected drifted terminology, v25.6.1 added the CBEST FrameworkPhase taxonomy, v25.6.2 fills in the **per-concept × per-framework** content layer — the ConceptSidebar's CBEST tab no longer reads "Content pending" for the 9 concepts that anchor SC013 and SC014. And it does it the OBS-018 way: every CBEST framework card now carries four-lens role-conditional content (teller / analyst / soc / executive) matching the FrameworkPhaseSidebar pattern.

---

## Summary

**9 APPEARS_IN_FRAMEWORK edges enriched** — 8 AASE-defined concepts (CRITICAL-FUNCTION, CONCESSION, LETTER-OF-ENGAGEMENT, TEST-HALT-MODEL, EXERCISE-SECRECY, ESCALATION-PATH, NO-FLY-ZONE, THREAT-MATRIX) gain CBEST-specific summary text and four-lens roleContent. The CBEST-defined CONCEPT-CREST-ACCREDITATION concept keeps its v25.6.0 summary and gains the same four-lens roleContent for symmetry.

**Net effect for users:** open any concept chip on SC013 / SC014 → expand the CBEST card → four lens tabs appear, your role's tab dotted, content tailored. The same lens-switching pattern as the FrameworkPhase sidebar — perspective-switching is one tap away.

**No new nodes. No new edges. No new edge types.** Property updates only on existing edges. The migration is the gentlest of the v25.6 series.

## Schema impact

The migration sets two new properties on existing `APPEARS_IN_FRAMEWORK` edges:

```cypher
// For 8 AASE concepts × CBEST framework (currently pending: true):
SET r.summary     = 'CBEST-specific summary text...',
    r.roleContent = '<JSON-stringified 4-lens object>',
    r.pending     = false,
    r.vintage     = '2024'

// For CONCEPT-CREST-ACCREDITATION × CBEST (already populated, just adds roleContent):
SET r.roleContent = '<JSON-stringified 4-lens object>'
```

Server endpoint `/api/frameworks/concepts/:id` parses `r.roleContent` back to a JS object before returning. Pre-v25.6.2 callers continue to work — they just see `roleContent: null` on edges that haven't been enriched (AASE today, TIBER/iCAST always for now). Forward-compatible.

## Interactivity Design (OBS-018)

This is the third release in the four-level Interactivity Design pattern (after v25.6.0 deploy notes, v25.6.1's FrameworkPhase nodes). Now the ConceptSidebar matches: every framework card with `roleContent` shows the same four lens tabs as FrameworkPhaseSidebar, with the user's role-tab marked by an accent dot, the same fade-on-switch motion, the same content shape (`summary` + `keyPoints[]`).

### Frontline / Teller

**What changes:** When a teller-role user lands on a concept chip in SC013 (governance scenarios) or SC014 (helpdesk vishing), the CBEST tab now shows them why the concept matters at branch level — in plain language, without governance jargon. The "Critical Function / IBS" concept, for example, becomes "*systems your branch uses every day; if they're scoped, they get tested live*" rather than the analyst's "*Important Business Service per CBEST 2024d Scope Specification*".

**What they see:** Phase-level CBEST chips on stages still use the FrameworkPhase wedge from v25.6.1. The new payoff is the *concept* chip — they can now read about Critical Function (or any of the other 8 concepts) through their own lens, without translating governance language. Tap concept chip → expand CBEST card → Frontline tab pre-selected → 1-paragraph summary + 2-3 keyPoints all calibrated to their seat.

**What they learn:** That the same concept (e.g. Test/Halt model, Threat Matrix, No-Fly Zone) shows up in both AASE and CBEST framings, just with different operational implications for them. Builds the durable mental model — concept first, framework variation second.

### Fraud Analyst

**What changes:** Every CBEST card now expands to show four-lens content. The analyst lens (default for analysts) is the most procedurally detailed — deliverable references (CBEST 2024d, 2024e, 2024f, etc.), workflow location, SC013/SC014 stage cross-references. But the analyst can also tap the **Executive lens** to see how the same concept reads upstairs ("scope sign-off is one of three executive-decision moments"), or the **SOC lens** to see what the secrecy implications mean for their colleagues.

**What they see:** When they tap any concept chip on SC013 — say, CONCEPT-EXERCISE-SECRECY on stage 1 — the CBEST card now reads:

> "Secrecy applies firm-wide; the CG is the only group permitted full visibility. Secrecy compromise — actual or suspected — must be notified to the regulator immediately (BoE §3.2.2). No specific clock; the bar is 'without delay' and judgement-based on the nature of the compromise."

Plus three keyPoints citing CG composition discipline, the v25.5.2 misframing correction (4-hour window myth), and procurement-stage compromise consequences.

**What they learn:** The vocabulary that maps directly into a real CBEST conversation — including the deliverable citations (CBEST 2024c through 2024k), the BoE section anchors (§3.2.2, §5, §7.2.3), the canonical TTPs (MITRE ATT&CK / Cyber Kill Chain). They walk away with the analyst's working dictionary, not a tourist's overview.

### SOC / Cyber Team

**What changes:** SOC-relevant concept lenses are *secrecy-aware* by design. CONCEPT-CRITICAL-FUNCTION's SOC lens explicitly states "you do not learn which IBSs are scoped — by design"; CONCEPT-EXERCISE-SECRECY's SOC lens says "you are deliberately outside CBEST secrecy — that is the entire point of the test"; CONCEPT-ESCALATION-PATH's SOC lens spells out that real-time escalations route around the CG (Annex B RACI). The concept content reinforces SC014's hard-mode rule: treat everything as real, escalate normally, the CG is not on your call.

**What they see:** When they tap CONCEPT-TEST-HALT-MODEL on any SC014 stage, the SOC lens reads:

> "You don't invoke the halt — that's the CG's call. But your escalation behaviour is what feeds halt decisions. Treating real activity as test (or test as not-our-problem) is what produces a missed halt or a wasted halt. The asymmetry: missed halts are findings, wasted halts are noted but recoverable."

Three keyPoints reinforce: escalate everything as real, Test/Halt failures are graded in the D&R Capability Assessment, SC014-S6 plays the canonical test/real ambiguity moment.

**What they learn:** The procedural rationale behind SC014's hard rule. Not "treat everything as real because the trainer says so" — *"treat everything as real because the asymmetry of consequences favours that posture, and the D&R Assessment grades the responses regardless"*. The training is more durable when the rule's grounding is visible.

### Risk Manager / Exec

**What changes:** The Executive lens on every CBEST concept card frames the concept around accountability and decision authority. CONCEPT-CRITICAL-FUNCTION's exec lens names the Scope Specification sign-off as "one of three executive-decision moments in CBEST"; CONCEPT-CREST-ACCREDITATION's exec lens names the procurement gate as the second; CONCEPT-CONCESSION's exec lens names CG approval and regulator visibility. The three-decision-moments arc becomes navigable through the concept layer.

**What they see:** Clicking CONCEPT-ESCALATION-PATH from SC013-S1, the Executive lens reads:

> "You sit on the CG (typically as executive sponsor or delegate). The CGC reports to you. CG composition discipline is the primary risk control: small, senior, decision-authoritative."

Three keyPoints cite the BoE Annex B RACI, the CGC reporting line, and the composition rule. Combined with the FrameworkPhaseSidebar from v25.6.1, the executive can navigate a concept → its CBEST framework lens → its specific phase → the practiced-in scenario stage in roughly four taps.

**What they learn:** Where in the CBEST procedural arc each accountability moment lives, and what's at stake at each. The three executive-decision moments — scope sign-off, procurement gate, remediation prioritisation — are visible across all four lenses but the Executive lens is where they're explicitly named as decision moments rather than just process steps.

### Cross-lens behaviour

The lens tabs on the ConceptSidebar are the same as on FrameworkPhaseSidebar — same labels, same accent-dot for the user's role, same motion. **A user can read all four lenses on the same concept**, which is the most useful interactive moment: a fraud analyst on CONCEPT-EXERCISE-SECRECY clicks the SOC lens to see what CBEST secrecy actually means for the colleagues they don't share information with. An executive on CONCEPT-CONCESSION clicks the Analyst lens to see what de-chaining looks like procedurally before signing off the CG decision. The fluency is in the perspective-switching, not in any single lens.

This is what "interactivity is the most important feature" looks like at the concept layer.

## Honest reflections on tradeoffs

**Path C (hybrid scope) chosen over Path A (minimum viable) and Path B (full enrichment).** Path A would have shipped 8 single-text summaries and closed ISS-014 mechanically. Path B would have taken much longer because it would also enrich AASE / TIBER-EU / iCAST with the same four-lens pattern. Path C — schema and UI universally support four lenses, but only CBEST gets fully populated this release — sets the architecture without the volume burden. AASE / TIBER / iCAST four-lens enrichment is a future v25.6.x or v25.7 candidate.

**The Concept ID rename is still deferred.** v25.6.0 changed the display name to "CBEST Accreditation" but the graph ID is still `CONCEPT-CREST-ACCREDITATION`. v25.6.2 doesn't touch this. URL surfaces and debug refs still say CREST. Same v25.6.x cleanup target.

**AASE FrameworkPhase enrichment is still deferred.** AASE phases exist as nodes (the FrameworkPhase=4 in the pre-snapshot) but don't yet have a chip wedge or roleContent. v25.6.1's FrameworkPhaseSidebar is framework-agnostic and ready to receive AASE phases when their content is authored. v25.6.x or v25.7.

**Risk of regression: minimal.** Migration touches edge properties only. No new nodes, no new edges, no new edge types. ConceptSidebar's existing single-text rendering path stays intact for any framework card without `roleContent`. AASE concepts read identically to before. SC001-SC012 see zero change.
