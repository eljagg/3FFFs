# v25.7.2 — Display correctness fixes (accuracy math + hardcoded technique count)

**Release type:** Bug-fix release. Three small files. No schema, no migration, no new dependencies, no client architecture changes.
**Driver:** Two display bugs visible on Home page that would undermine credibility in any external pitch (Barita, JNCB, Scotia JM, CIBC). Pre-existing — neither was introduced by v25.7.1, but both became more visible during the engagement-layer release because users started looking at the Home page more carefully. Must be right before any external demo.
**Sequence:** v25.7.1.3 → **v25.7.2 (this release)**. Next: v25.7.0.3 (reconnaissance flow viz, separate release).

## The bugs

**Bug 1 — "400% accuracy" on Home stats.** A user with 1 quiz attempt that was correct showed `400%`. A user with 2 attempts of the same quiz, both correct, showed `200%`. The query in `USER_PROGRESS_SUMMARY` used `count(DISTINCT q) AS quizzesAnswered` for the denominator but `sum(CASE WHEN a.correct ...)` for the numerator. When a user answered the same quiz multiple times, the numerator scaled but the denominator stayed at 1. 4 correct attempts to 1 unique quiz → 400%.

**Bug 2 — "126 F3 techniques" hardcoded copy.** Three places hardcoded the number 126:
1. `client/src/pages/Home.jsx` line 31: `progress?.totalTechniques ?? 126` (fallback for stats grid)
2. `client/src/pages/Home.jsx` line 67: `"See which of the 126 F3 techniques you've mastered..."` (blurb on Coverage card)
3. `server/src/routes/progress.js` line 15: `totalTechniques: 126` (fallback in `/api/progress` when no row returned)

The actual F3 graph has 114 techniques. The 126 was a pre-launch estimate that became stale when the framework was finalized in April 2026.

## The fixes

**Fix 1** — `server/src/lib/graph-queries.js`, `USER_PROGRESS_SUMMARY` query:

```cypher
// Before
count(DISTINCT q)  AS quizzesAnswered,
sum(CASE WHEN a.correct THEN 1 ELSE 0 END) AS correctAnswers

// After
count(a)           AS quizzesAnswered,
sum(CASE WHEN a.correct THEN 1 ELSE 0 END) AS correctAnswers
```

Both numerator and denominator now count over the same set: relationship attempts. 4 correct attempts to 1 unique quiz → 4/4 = 100%, never 400%. The semantic shift is "% of your attempts that were correct" rather than "% of unique quizzes you got right" — which is the more useful metric for a learner anyway, because retrying a quiz multiple times until you get it right shouldn't inflate your accuracy.

**Fix 2** — `client/src/pages/Home.jsx`:

```jsx
// Before
const total = progress?.totalTechniques ?? 126
<ActionCard ... blurb="See which of the 126 F3 techniques..." />

// After
const total = progress?.totalTechniques ?? 0
<ActionCard ... blurb={total > 0
  ? `See which of the ${total} F3 techniques you've mastered, and which are still ahead.`
  : "See which F3 techniques you've mastered, and which are still ahead."} />
```

Stats grid now shows `0/0` instead of `0/126` when data isn't loaded — better empty state. The Coverage card blurb is dynamic from the same source. If the API call hasn't returned yet (no number), the blurb gracefully omits the count rather than showing a phantom number.

**Fix 3** — `server/src/routes/progress.js` fallback when no row returned:

```js
// Before
totalTechniques: 126, techniquesEncountered: 0,

// After
totalTechniques: 0, techniquesEncountered: 0,
```

Same logic: 0 is the correct value for "unknown/not-loaded", not a stale pre-launch estimate.

## Honest reflection

These bugs were both visible during v25.7.1 testing (the 400% appeared in your Home screenshot multiple times today, and I called out the 126 separately when looking at the Coverage card blurb). I should have rolled them into v25.7.1 itself rather than splitting them into v25.7.2 — but at the time of the v25.7.1 deploy, I correctly didn't want to expand scope mid-release. Now that v25.7.1 is stable (5 deploys later), this is the right next release: small, focused, fixes things that are visibly wrong before any external demo.

These are also the kind of bugs that erode trust quickly. A teller seeing 400% accuracy will silently doubt every other number on the page. A risk manager seeing "126 F3 techniques" when the actual framework has 114 will quietly note that the product team isn't paying attention to detail. Fixing both, even though they're tiny code changes, is high-leverage work for credibility.
