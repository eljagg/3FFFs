hearing "Welcome to JNCB. For balance, [stutter] press one. [click]
For [pause] card services [restart] services, two." learns the wrong

actually hear when calling a Caribbean bank's IVR."

## Diagnosis

Two coordinated bugs, both rooted in Chromium speechSynthesis quirks
that v25.7.0.15.3's architecture was unintentionally vulnerable to.

**Root cause #1 — `cancel()` then `speak()` race window.**

ProcessAnimation, TimelineThresholdAnimation, and
MultiActorSequenceAnimation all had this pattern:

```js
useEffect(() => {
  stopAllNarration()              // 1. cancel() called at effect entry
  // ...
  setTimeout(() => speakMessage(...), 350)
  return () => stopAllNarration() // 2. cancel() also in cleanup
}, [...stage deps])
```

When the stage advances:
1. The cleanup function runs → calls `cancel()` (1st cancel)
2. The new effect runs → calls `cancel()` again (2nd cancel)
3. After 350ms, `speak()` is called

Chromium has a documented race condition where calling `speak()`
within ~500ms of one or more `cancel()` calls causes the new
utterance to start, get interrupted by the still-settling cancel
state, and restart — producing the jitter/stutter Omar reported.

**Root cause #2 — Chrome's speechSynthesis pause-after-N-seconds bug.**

Chromium-based browsers have a long-known bug where speechSynthesis
pauses itself between utterances and after ~14 seconds of continuous
speech, causing audible gaps and jitter. The Vishing animation's long
multi-message stages would be vulnerable, but the most obvious
manifestation was IVR's longer single-utterance stages (10-14 words
each — long enough to hit the bug window).

**Why IVR was "more jittery than OSINT":**
- IVR audio per stage: 14, 12, 11, 10 words (all hit the bug zone)
- OSINT audio per stage: 6, 8, 9, 9, 10, 9, 7 words (mostly finish before the bug manifests)

This perfectly fits Omar's symptom report.

## What changed (3 engines + audioNarration.js)

### `audioNarration.js`
- Bumped version to `v25.7.0.15.4`. New console log: `[3FFFs audio v25.7.0.15.4 — Chrome keep-alive]`
- **Added Chrome keep-alive interval**: a `setInterval` that runs every 5 seconds while the hook is mounted. When speech is actively playing (and not paused), it calls `pause()` immediately followed by `resume()`. This keeps the speechSynthesis queue alive and prevents the documented Chromium pause-mid-utterance bug.
- Harmless on Safari (Safari doesn't have the bug; pause/resume on idle is a no-op).
- Properly cleaned up on unmount.

### `ProcessAnimation.jsx`
- **Removed redundant `stopAllNarration()` call at effect entry.** The previous effect's cleanup already called cancel; calling it again at the next effect's start meant double-cancel within milliseconds, which is the Chrome race trigger.
- **Increased speak-after-cancel delay from 350ms to 600ms.** 600ms is past Chrome's documented race window (~500ms). The first speech of each stage now reliably starts cleanly.

### `TimelineThresholdAnimation.jsx`
- Same fix as ProcessAnimation. (Currently no scenes use audio in this engine, but the fix is applied for v25.7.0.16+ readiness.)

### `MultiActorSequenceAnimation.jsx`
- Same fix as ProcessAnimation: removed redundant entry cancel, added 600ms delay before enqueueing stage's utterances.

### Scene files
**No changes.** All 6 scene files byte-identical to v25.7.0.15.3.

## Honest accounting

Three notes:

1. **The Chrome keep-alive pattern is well-documented but unfortunate.**
   It's working around a 9-year-old browser bug. The implementation is
   defensive (only pings during active speech, wraps in try/catch,
   harmless on browsers without the bug). This is the standard
   solution used by every speechSynthesis-based web app I've seen.

2. **600ms delay is conservative.** Caption appears, then audio
   starts after a brief pause — the "newscaster cadence" of "show
   then tell." If you find this too long, I can dial back to 400-500ms
   in v25.7.0.15.5; below 400ms re-enters the Chrome race window.

3. **My v25.7.0.15.3 testing didn't catch this because** browser TTS
   testing is hard to automate and the symptom is browser-specific.
   The diagnostic pattern Omar provided ("IVR more jittery than
   OSINT") was the key signal that pointed straight to the
   pause-after-N-seconds bug — long utterances hit the bug, short
   ones don't. That symptom asymmetry is the canary I should have
   asked about earlier in the diagnostic cycle.

If after v25.7.0.15.4 the audio is still jittery, the next layer to
investigate is voice-specific behavior (some macOS voices have their
own quirks) or a deeper Chromium-version-specific issue. Tell me your
browser + OS and I'll go deeper.
