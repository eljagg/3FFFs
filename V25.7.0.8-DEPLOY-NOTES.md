## Four-lens block (OBS-018)

- **Teller lens:** A teller currently sees ALL techniques in a tactic
  flat — including ones they can't act on (3DS Bypass, PaReq
  manipulation). Cognitive noise. After this release: same content
  visible (cross-team awareness) but greyed-out with explicit "your
  role doesn't primarily act on this technique" labeling. Their primary
  techniques remain full-size and actionable. Click any technique →
  sidebar opens with full content + cross-references to scenarios that
  demonstrate it. NOTE: v25.7.0.8 ships with all techniques marked
  "all roles primary" by default — actual role-relevance authoring
  per-tactic is planned for v25.7.0.10+. Teller will see the GREYING
  STRUCTURE working but won't see actual role gating until content
  arrives.
- **Analyst lens:** Same structural improvement. Cross-references to
  scenarios are particularly useful here — clicking F1008.001 in the
  sidebar surfaces "Demonstrated in 1 scenario: SC007" with a
  click-through. Turns the framework from a dictionary into a
  navigable web.
- **SOC lens:** Two-tier hierarchy reflects how SOC actually thinks
  about techniques (parents are control categories; sub-techniques are
  specific implementations to detect). The hierarchy is honest to
  their mental model. Sidebar shows sibling sub-techniques for lateral
  exploration — analyst-style navigation that didn't exist before.
- **Executive lens:** Smaller win. Exec sees fewer cards in the
  default view (parent-only with sub-counts), can drill if needed.
  Coverage stats per parent technique would be a future enhancement
  more directly useful for exec — not in this release.

## Pedagogical insight (OBS-027)

The current grid teaches techniques as a flat dictionary — F1063 =
Delete Relevant Emails, F1064 = Device Fingerprint Spoofing. A trainee
can read each card and walk away knowing 9 definitions but not knowing
how they relate to each other or to scenarios they've seen. The
hierarchy + cross-references this release ships teaches that techniques
EXIST IN A NETWORK: they group under parent concepts (F1008.001 / .002
/ .005 are all Account Manipulation), they appear inside scenarios
(F1008.001 is THE Day 8 move in SC007), they can be observed by certain
roles.

The lever this release exists to enable: training the trainee's reflex
to ASK "what's this technique connected to?" rather than just "what
does this technique mean?" Definitions are dead; networks are alive.

## Architecture decisions (locked into this release)

1. **Two-tier hierarchy via existing graph edges.** The data layer
   already has `:Technique-[:SUBTECHNIQUE_OF]->:Technique` and
   `:Technique-[:PART_OF]->:Tactic` from the original v25.7.0.0 seed.
   No data migration needed. New endpoint
   `/api/framework/tactics/:id/techniques-tree` queries the existing
   structure and returns hierarchical JSON.

2. **New endpoint, not modify existing.** The existing
   `/api/framework/tactics` endpoint returns techniques as a flat list.
   Modifying it to nest hierarchy would break any other consumer.
   v25.7.0.8 adds three new endpoints under `/api/framework`:
   - `GET /tactics/:tacticId/techniques-tree` — hierarchical tree
   - `GET /techniques/:techId` — single technique with parent/siblings/children
   - `GET /techniques/:techId/cross-refs` — scenarios + beats that reference
   Existing endpoints unchanged. Other consumers of `/tactics` keep working.

3. **Role-relevance authored per-tactic in subsequent releases.** Per
   Q-A confirmed, v25.7.0.8 defaults all techniques to "all roles
   primary" — the role-gating UI works and is testable, but no
   technique gets greyed out yet. Per-tactic role authoring is planned
   for v25.7.0.10+. This decouples the structural release from content
   authoring and lets us ship the foundation faster.

4. **Single-child collapse.** Per Q-B confirmed, when a parent
   technique has exactly 1 sub-technique, the grid renders as a single
   card without expansion (the sub-technique IS the meaningful
   technique in that case). Endpoint marks `collapseToSingle: true` and
   the client renders accordingly.

5. **Atomic techniques.** Per Q-C confirmed, parent techniques with no
   sub-techniques render as parent cards with no expand affordance.
   Click goes straight to sidebar. Visual treatment: small "atomic"
   chip in the card header to signal the difference from expandable
   parents.

6. **Role-gating treatment.** Per Q-D confirmed, non-relevant
   techniques render at reduced opacity with explicit "your role
   doesn't primarily act on this technique" label, but stay clickable.
   Hides nothing — preserves cross-team awareness while focusing
   visual cognitive load on role-primary techniques.

7. **Beat cross-references via techId string, not via edge.** The
   existing `:Beat` nodes store `techId` as a string property. No
   `:Beat-[:REFERENCES_TECHNIQUE]->:Technique` edge exists. v25.7.0.8
   queries beats by `techId` string match — works fine for current
   scale (8 beats today, will be ~50-100 after SC008/SC011/SC013 are
   authored). If query performance becomes a problem at higher scale,
   a future release adds proper edges via migration. Not blocking
   today.

## Known limitations / non-goals

- **No animations yet.** "How this technique works" sidebar section is
  a placeholder. v25.7.0.9 ships first batch of 5 high-value
  animations.
- **No actual role-relevance content.** All techniques default to all
  roles primary. v25.7.0.10+ adds per-tactic authoring. The greying-out
  visual treatment works, just doesn't activate for any technique yet.
- **No mitigation guidance.** Placeholder. Future content authoring.
- **Sidebar doesn't deep-link to specific beats.** "Featured in beat"
  rows route to `/scenarios/:id` and the user has to expand the
  storyboard themselves. Beat-level deep-linking is a future polish
  item if requested.
- **Parent-card click ambiguity.** Cards with sub-techniques click =
  expand inline. Cards without sub-techniques (atomic) click = open
  sidebar. The visual cue is the "Show sub-techniques" vs "Open
  detail" hint at the bottom of each card. Some users may still
  expect both behaviors on parents (expand AND open). Easy follow-up
  if feedback warrants.

## Honest accounting

This is a meaningful release — 4 new files, 1 substantially modified,
3 new endpoints. The core decisions I want to flag:

1. **Single sidebar instance, content swaps.** Could have used a
   per-card sidebar. Single instance is simpler state management AND
   feels right interactively (lateral nav between siblings without
   close/reopen).

2. **Lateral navigation in the sidebar (sibling/parent/child clicks).**
   This wasn't in your Q-A through Q-E answers explicitly, but it felt
   like the natural pedagogical fit for "techniques as a network."
   If you'd rather sidebar always close on technique-link click and
   reopen for the new technique, easy change.

3. **Cross-refs section comes BEFORE animation/mitigation
   placeholders.** I considered ordering placeholders first
   (description first, then "how it works", then "where it appears")
   but the cross-refs are the only LIVE content right now and burying
   them below two placeholder sections felt wrong. Order is:
   description → roles → mitigation placeholder → animation placeholder
   → cross-refs → siblings/children. When animation content lands in
   v25.7.0.9+, I'll reconsider this ordering.

If anything in the foundation feels wrong on first look — particularly
the role-gating defaults or the click-to-expand vs click-to-open
ambiguity — easy to iterate. The data layer is unchanged so backouts
are cheap.
