## Four-lens block (OBS-018)

- **Teller lens:** A teller training on SC007 sees Day 8 ("crew swaps her
  e-delivery email to throwaway Gmail") and the per-role guidance placeholder
  where their specific actions would be authored. The whole point: "this is
  what attacker positioning looks like through your lens" presented as a
  story they'll remember, not a definition list. The visceral memory hook
  is exactly the OBS-027 pedagogical insight this release implements.
- **Analyst lens:** Same beats, but the per-role guidance (when authored
  in v25.7.0.9) emphasizes case investigation patterns: "see Days 11-14 —
  the 3-incoming, no-salary, no-family pattern is your tell." For analysts
  the storyboard is a worked case file. The summary card's "Detection
  window: Days 1-8" stat is also analyst-relevant — it tells them how
  many days of latitude they have once Position has begun.
- **SOC lens:** Counterfactual beat at the end of SC007 is most relevant
  for SOC — "if Auto-freeze on early e-delivery change was active, account
  freezes Day 8." For SOC the storyboard is a tuning artifact: each
  detection counterfactual maps to a control they'd configure. v25.7.0.9
  per-role guidance will make this explicit.
- **Executive lens:** Scenario summary card at the top with "Days from open
  to cashout: 14, USD 8K avg loss" is the right entry point. Execs want
  shape and impact, not beat-by-beat. The summary card answers their
  question; the beats are there if they want to drill in. The list of
  stub scenarios in the picker also surfaces where v1+ content investment
  will go — useful context for resource-allocation conversations.

## What this release does

Adds a **scenario storyboard** to the F3 Framework page, sitting between
the search results and the tactic list. It is a peer of the tactic list,
not nested inside any tactic, because scenarios cross multiple tactics
(SC007 has Positioning + Execution beats and references the
Counterfactual frame).

### The page-level structure becomes

```
F3 Framework page
├── Title + lede + search
├── Search results (when querying)
├── Scenarios — collapsible (NEW, v25.7.0.5)
│   └── ScenarioStoryboard component
│       ├── Picker tabs (SC007 active, SC008/SC011/SC013 disabled stubs)
│       ├── Summary card (when scenario loaded)
│       └── Two-column workbench (timeline left, sticky detail right)
└── Tactics list (existing — Reconnaissance through Monetization)
```

### Why a new collapsible section instead of a per-tactic section

The storyboard is per-scenario, but scenarios cross multiple tactics. SC007
Account Rental has Positioning beats (Days 1-8) AND Execution beats (Days
11-14). Putting it under any single tactic would either fragment the
narrative or break the tactic's "this section is about THIS tactic"
coherence. Treating scenarios as a separate page-level concern is honest
about the data shape.

The per-tactic technique grids (introduced in v25.7.0.4.4) stay unchanged.
This release adds the storyboard alongside them, not as a replacement.

## Server changes

### New seed data: `server/src/seed/data/scenario-beats.js`

Two exports:
- `SCENARIO_BEATS` — keyed by scenario ID, value is an ordered array of
  entries (beats interleaved with phase markers). SC007 has 11 entries:
  3 phase markers + 8 beats covering Days 0–14 of the Account Rental
  scenario plus a counterfactual.
- `STORYBOARD_SUMMARIES` — keyed by scenario ID, value is the summary
  card content (headline, subtitle, meta, stats).
- `STORYBOARD_STUBS` — picker entries for SC008/SC011/SC013.

### New migration: `server/src/seed/add-scenario-beats.js`

Idempotent post-seed migration. Adds:
- `:Beat` nodes connected to existing `:Scenario` nodes via `:HAS_BEAT`
- `:BeatPhase` nodes (phase separators) connected the same way, with
  `order` on the relationship determining their position in the timeline
- `storyboardHeadline`, `storyboardSubtitle`, `storyboardMetaJson`,
  `storyboardStatsJson` properties on the `:Scenario` node

Follows OBS-021 (MERGE → ON CREATE SET → ON MATCH SET → SET) and OBS-025
(preservation check via edge count after migration).

Run after the main `npm run seed` finishes:

```bash
node server/src/seed/add-scenario-beats.js
```

### New API route: `GET /api/storyboard/scenarios` and `GET /api/storyboard/scenarios/:id`

Both endpoints PUBLIC (no auth) — same access pattern as `/api/framework`,
`/api/visualizations`, `/api/mitre`. Reference content, not user-specific.

`/scenarios` returns:
```json
{ "scenarios": [
  { "id": "SC007", "name": "Account Rental — ...", "severity": "medium", "hasBeats": true },
  { "id": "SC008", "name": "ATM Skimming",  "plannedRelease": "v25.7.0.6", "hasBeats": false },
  ...
] }
```

`/scenarios/:id` returns:
```json
{
  "scenario": { "id": "...", "headline": "...", "subtitle": "...", "meta": {...}, "stats": [...] },
  "entries": [
    { "id": "SC007-PHASE-1", "phase": "POSITIONING — disguise + persistence" },
    { "id": "SC007-B1", "day": 0, "actor": "victim", "kind": "victim-event", ... },
    ...
  ]
}
```

The `entries` array is pre-ordered by relationship `order` server-side.
Phase markers and beats are interleaved in the same array; the client
discriminates via `entry.phase` (truthy = phase marker).

## Client changes

### New component: `client/src/components/scenarios/ScenarioStoryboard.jsx`

Self-contained React component. Pure inline styles per project convention.
No external CSS file. Accepts a `defaultScenarioId` prop (default `'SC007'`).

Internal subcomponents:
- `ScenarioTabs` — picker, renders authored and stub scenarios; stubs are
  disabled with their planned-release tag in the title attribute on hover
- `SummaryCard` — headline + meta + stats panel
- `Timeline` — vertical day-axis with beat cards, including the climax
  visual treatment for high-impact beats and the side-color band
  (accent for attacker, success-green for victim events, warning-amber
  for detection counterfactuals)
- `DetailPanel` — sticky right-side panel with full beat content;
  technique reference + "why this beat matters" + per-role placeholder

Loading and error states for both the scenario list and the per-scenario
fetch. Falls back to the first authored scenario if `defaultScenarioId`
isn't found in the picker data.

### Framework page integration

Two changes to `client/src/pages/Framework.jsx`:

1. **Import + render block.** Storyboard imported and rendered as a
   collapsible section between search results and the tactic list. Click
   the section heading to expand.

2. **Collapse state.** New `storyboardExpanded` boolean state. Defaults
   to `false` (collapsed) so first-time visitors land on the familiar
   tactic list, with the storyboard available behind one click.

The toggle styling is similar to the per-tactic visualization toggles
established in v25.7.0.4.8 — accent eyebrow, display-font title, subtitle,
chevron — but slightly more prominent because it's a page-level peer of
the tactic list.

## Known limitations / non-goals

- **Narrow-viewport responsive treatment.** The two-column workbench
  (timeline + detail) doesn't reflow on viewports below ~900px. Per
  project convention (no other component has responsive breakpoints),
  this is documented as a backlog item, not a launch blocker.
- **No per-role guidance content yet.** Each beat detail shows a
  placeholder where per-role guidance will live. Authoring this is
  v25.7.0.9 work — the highest-leverage content addition we have, since
  it ties scenario beats to specific role actions (the OBS-018 four-lens
  pedagogy at the most concrete level the framework can reach).
- **Search doesn't index storyboard beats.** The Framework page search
  searches tactics + techniques + scenario titles but doesn't drill into
  beat headlines or narratives. Worth considering for a later release if
  user feedback suggests it.

## Honest accounting

This is the third release in a row where my first instinct (v25.7.0.4.2
margin-bleed, v25.7.0.4.3 sibling structure, v25.7.0.4.5 page-content-width)
was overcomplicated for the actual problem. v25.7.0.4.x ended up landing
the layout in 8 iterations rather than 1.

For this release I tried harder to think through the architectural
question first (where does Design C live: per-tactic or page-level?
how does it relate to the existing scenario seed and Two Views config?)
before writing code. Two consequential decisions made deliberately:

1. **Storyboard at framework level, not per-tactic.** Scenarios cross
   tactics; nesting them under one tactic fragments the narrative. The
   page now has parallel "tactics" and "scenarios" sections.
2. **New `:Beat` node label rather than reusing `:Stage`.** Stages are
   quiz-shaped (with `question` + `options` for the Quiz/Play feature).
   Beats are narrative-shaped. Different schemas, different audiences.
   Forcing them together would have constrained both.

If either of these decisions feels wrong on contact with reality, the
fix is a backout — beats can be migrated to live under a different parent
node, the storyboard can be moved into a per-tactic visualization. But
I think these are the right calls and want to ship them this way to find
out.
