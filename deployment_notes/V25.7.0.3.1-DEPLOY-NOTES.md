# v25.7.0.3.1 — OBS-021 hotfix on add-visualizations.js

**Release type:** Server-only one-line hotfix on top of v25.7.0.3. One file. No client changes, no schema, no new dependencies. Migration script fix only — does not affect runtime app behaviour.
**Driver:** The `npm run migrate:visualizations` script started failing with a Cypher parse error after the v25.7.0.3 deploy when re-running the seed. The error: `Invalid input 'ON': expected an expression... at line 9, column 8`. Root cause: an OBS-021 violation (`MERGE → SET → ON CREATE SET` instead of the correct `MERGE → ON CREATE SET → SET`) that has been latent in `add-visualizations.js` since v25.7.0.2 but only started failing after a Neo4j Aura version bump tightened the Cypher parser. Running the migration is required for v25.7.0.3 to seed the new VIZ-POSITIONING-TIMELINE entry into Neo4j; without it, the Framework page won't render the new Positioning visualization.
**Sequence:** v25.7.0.3 → **v25.7.0.3.1 (this release)**.

## What happened, in one paragraph

`add-visualizations.js` line 96-105 had `MERGE` followed immediately by `SET` followed by `ON CREATE SET`. That clause order is invalid Cypher — the parser requires `ON CREATE SET` and `ON MATCH SET` clauses to come BEFORE any unconditional `SET`. The same violation pattern bit us before in v25.7.0.2.1 (different script), and OBS-021 was added to the locked-observation set: **`MERGE → ON CREATE SET → ON MATCH SET → SET`**. This particular file slipped past v25.7.0.2.1's sweep — it was authored after that hotfix landed and shipped its own violation. The script worked anyway during v25.7.0.2 testing because every run after the first one took the MATCH path (where `ON CREATE SET` doesn't execute), masking the parse error from showing up. After Aura's parser tightened, even the parse step failed regardless of which path would execute.

## The fix

```cypher
// Before (invalid Cypher, OBS-021 violation)
MERGE (v:Visualization {id: $id})
SET v.kind = $kind, ...
ON CREATE SET v.createdAt = timestamp()

// After (correct order)
MERGE (v:Visualization {id: $id})
  ON CREATE SET v.createdAt = timestamp()
  SET v.kind = $kind, ...
```

Logically identical: on a fresh node, `createdAt` is set first, then all the unconditional properties; on an existing node, `ON CREATE SET` skips and `SET` updates everything else (preserving the original `createdAt`). Same end-state, valid Cypher.

## Sweep result — why this is the only fix

A grep across `server/src/seed/*.js` and `server/src/routes/*.js` for `MERGE` + `ON CREATE SET` patterns confirmed:
- `add-visualizations.js`: ❌ violated (this fix)
- `add-banks.js`: ✓ correct order
- `add-caribbean.js`: ✓ correct order
- `add-aase-execution-scenarios.js` / `add-aase-scenario.js` / `add-sc013.js` / `add-sc014.js`: ✓ no `ON CREATE SET` (use only unconditional `SET`)
- `routes/admin.js`, `routes/banks.js`, `routes/quiz.js`, `routes/scenarios.js`, `routes/visualizations.js`: ✓ correct order

This is the only OBS-021 violation in the codebase. Going forward, OBS-021 should be checked at PR time for any new MERGE-with-ON-CREATE-SET pattern.

## Honest reflection

**This bug should have been caught when v25.7.0.3 shipped.** OBS-021 is a locked observation. I didn't sweep for it before shipping the new visualization release. The lesson: when shipping a new visualization that depends on the existing migration script, re-validate the migration script itself rather than assuming "it worked last time so it'll work now." Sweeping for OBS-021 violations in migration scripts on every release that touches them is now part of my pre-ship checklist.

**OBS-021 enforcement going forward.** Adding to my mental pre-ship checklist:
> Before shipping any release that depends on an existing migration script, grep that script for `MERGE` + `ON CREATE SET` and verify the order. The Cypher parser doesn't fail consistently — it can let the violation through if execution always takes the MATCH path on test data — so syntax-checking the script in isolation isn't enough.
