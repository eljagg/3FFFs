# v25.7.0 — Multi-tenant foundation + per-user progress (ISS-017 / 019a / 019b / 021)

**Release type:** Major foundation release. New schema entities (`Bank`, `MEMBER_OF`, `MANAGES`), new endpoint surface, two new client pages.
**Driver:** ISS-017 (per-user progress) + ISS-019a (Bank entity) + ISS-019b (manager-report) + ISS-021 (multi-tenancy Pattern A foundation, Pattern C ready).
**Sequence:** First in the v25.7.x series. Comments + messaging + search ship as v25.7.1 / v25.7.2 / v25.7.3.

This is the architectural shift flagged in the previous discussion — the platform crosses from "content tool" to "workforce-management surface for risk training." The OBS-018 discipline is preserved: every new endpoint and page carries role-conditional behaviour for the four user lenses (teller / analyst / soc / executive).

## Multi-tenancy strategy (ISS-021)

This release establishes Pattern A (single shared deployment, logical isolation by Bank). Pattern C (per-bank dedicated deployment) is **architecturally ready** but not exercised in this release.

**Pattern A — what's running today.** All banks share one Railway deployment, one Aura DB, one Auth0 tenant. Bank isolation is enforced at the application layer:

- `MEMBER_OF` edge from User to Bank
- Manager endpoints filter by `bankId` matching the requesting user's bank
- Admin sees across banks (in shared mode) or pinned to the dedicated bank (in dedicated mode)
- Search will be bank-scoped for managers in v25.7.3

**Pattern C — what would change.** A dedicated deployment for one bank would set:
```
DEPLOYMENT_MODE=dedicated
DEDICATED_BANK_ID=BANK-BARITA
```
The shim's `effectiveBankId()` then returns the dedicated bank id regardless of session, so every query is automatically scoped to that bank. No code changes; just env vars.

**What this means for you operationally.** When a bank asks for "their own deployment" you can say yes, the codebase supports it, the lead time is hours-to-a-few-days for infrastructure provisioning. Pricing this is a business decision — typical premium is 2-3x the shared-tier cost.

## Interactivity Design (OBS-018)

### Frontline / Teller

**What changes:** Trainees of all four roles get a `/me/progress` page. For tellers, the page emphasises **completion ledger first** — the row of scenario cards with "completed / in progress / not started" states. Aggregate stats (scenarios completed, correct rate) appear first as motivational hooks; competency profiling and framework analytics are below the fold and de-emphasised.

**What they see:**
- Aggregate banner: scenarios completed, attempts, correct rate, avg confidence
- Completion ledger as the primary surface — every scenario as a clickable card
- Concept mastery section below ledger
- Framework progress section last

**What they learn:** A clear sense of "which scenarios I've finished" and "which I should come back to." Low cognitive overhead. The dashboard is for motivation, not for analysis.

### Fraud Analyst

**What changes:** Same `/me/progress` page with **balanced ordering** — completion ledger and competency profile both prominent. Concept cards sort by correct rate descending; weak concepts (the ones the analyst should revisit) live at the bottom of the list, which is itself a learning signal.

**What they see:** The full Options A + B + C view in canonical order. Status of every scenario; mastery score per concept; per-framework attempt totals.

**What they learn:** The competency profile is where the analyst-tier value lives. They can spot patterns ("I'm 90% on supplier-onboarding controls but 40% on procurement gates") and use the chip cross-links to drill back into the source scenario.

### SOC / Cyber Team

**What changes:** SOC users see **MITRE ATT&CK exposure bumped to the top** of the competency section. The 30 most-attempted MITRE techniques appear as a list with per-technique correct rate. This matches the SOC mental model — they think in TTPs, not concepts.

**What they see:** Aggregate banner first, MITRE exposure list as the second section (above general competency), then the rest of the dashboard.

**What they learn:** Which TTPs they've practiced detection/response on, where their detection coverage is weakest, and (when ISS-018 ships in v25.7.1) which specific stages a manager has flagged for revisit.

### Risk Manager / Exec

**What changes:** Executive users see an **aggregates-first, ledger-then-frameworks ordering**. The aggregate banner serves as a one-glance KPI summary; framework totals come before concept mastery because executives think in frameworks (CBEST, AASE, TIBER), not individual concepts.

**What they see:** Aggregates → ledger → frameworks → competency. Tellers see ledger more prominently; execs see frameworks more prominently.

**What they learn:** Which frameworks they've engaged with most, which scenarios remain incomplete (from a compliance/training-completion perspective), and which governance-tier scenarios (SC013) need revisiting before quarterly reviews.

### Manager view (the multi-role moment)

A manager opening `/team/:userId` for any direct report or bank member sees the same dashboard but the section ordering follows **the trainee's** role, not the manager's. A risk-manager opening a SOC analyst's progress sees the SOC's MITRE-first ordering. This is intentional — managers learn what the trainee sees, which makes coaching conversations more grounded.

## Honest reflections on tradeoffs

**The MANAGES edge populator is admin-only via API; admin UI is deferred.** v25.7.0 ships the `POST /api/banks/me/manages` endpoint but no admin form for it. An admin can hit the endpoint directly via curl or the AdminUsers page (when a small enhancement lands). v25.7.1 or v25.7.x will surface this in the AdminUsers UI.

**No real-time notifications yet.** A manager opening `/team/:userId` sees current progress but doesn't get notified when the user attempts a new stage. v25.7.2 (messaging) lays the groundwork for this; v25.8 candidate for true real-time.

**Bank theme application is partial.** Bank theme properties (`primaryColor`, `accentColor`, `displayName`) are returned by `/api/banks/me` and the user's MyProgress page shows the bank pill in the bank's primary colour. Full theme propagation through the AppShell (header, sidebar, accent throughout the app) is a v25.7.x candidate — easy to add once we know what banks actually want.

**Performance: per-user-progress queries.** The `/me/full` endpoint runs 6 parallel Cypher queries. With our current data volumes (~14 scenarios, ~9 concepts, ~14 framework totals), the response is sub-100ms. As scenarios grow, the conceptMastery and frameworkProgress queries may need indexes added.

**Pattern C operational gap.** The shim is in place but a real Pattern C deployment requires manual provisioning of:
- A separate Railway environment with `DEPLOYMENT_MODE=dedicated` + `DEDICATED_BANK_ID=BANK-XXX`
- A separate Neo4j Aura instance
- A separate Auth0 tenant (or shared with allowlist scoped to the bank's domain)
- Custom DNS configuration

When the first customer asks, this becomes a one-time runbook authoring exercise. Estimated 1-2 days of focused infrastructure work.
