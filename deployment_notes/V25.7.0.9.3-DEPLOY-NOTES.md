## Four-lens block (OBS-018)

- **Teller / Analyst / SOC / Executive:** Same lens applies to all four
  roles. Different role contexts will collapse different sections at
  different times. An Analyst studying a technique's cross-references
  may collapse Description and Animation to focus on the scenarios.
  An Executive reviewing exposure may collapse everything except the
  Description and Mitigation. The pattern lets each role configure
  the surface to their current focus without prescribing one workflow.

## Pedagogical insight (OBS-027)

**Information sized for the user's current question.** v25.7.0.9.2
introduced the pattern (collapse Description + Animation once consumed).
v25.7.0.9.3 generalizes it: every content-heavy section yields to the
user's current focus. The lever this enables: a sidebar that's a
working surface, not a fixed display — the trainee shapes it to their
investigation rather than scrolling past content they've already
absorbed.

This is the polish version of the same principle from v25.7.0.9.2 —
not a new pedagogical idea, just the application taken to its logical
end.

## The change

### Refactored collapse state from per-section booleans to a Set

**Was (v25.7.0.9.2):**
```javascript
const [descriptionExpanded, setDescriptionExpanded] = useState(true)
const [animationExpanded, setAnimationExpanded] = useState(true)
```

**Now (v25.7.0.9.3):**
```javascript
const [collapsedSections, setCollapsedSections] = useState(() => new Set())

function isExpanded(sectionId) {
  return !collapsedSections.has(sectionId)
}

function toggleSection(sectionId) {
  setCollapsedSections(prev => {
    const next = new Set(prev)
    if (next.has(sectionId)) {
      next.delete(sectionId) // expand
    } else {
      next.add(sectionId)    // collapse
      if (sectionId === 'animation') {
        setAnimationPauseSignal(c => c + 1) // pause animation when its section collapses
      }
    }
    return next
  })
}
```

Why Set-based: 6+ sections × 2 setters each = 12+ state hooks if we
kept the per-section boolean pattern. Set-based is one hook + one
helper function, and adding a new collapsible section is a one-line
change at the call site.

Membership semantics: a section ID in the Set means **collapsed**.
Empty Set = everything expanded (which is the default after
technique-change reset).

### Six (or so) sections now collapsible

| Section | ID | When visible |
|---|---|---|
| Description | `description` | When technique has a description (always, for F3 techniques) |
| Mitigation guidance | `mitigation` | Always (placeholder content for now) |
| How this technique works | `animation` | Always (placeholder OR live animation) |
| Demonstrated in N scenarios | `scenarios` | When at least 1 scenario uses this technique |
| Featured in N storyboard beats | `beats` | When at least 1 beat references this technique |
| Cross-references (empty state) | `crossrefs-empty` | When neither scenarios nor beats reference this technique |
| N sub-techniques | `subtechniques` | When technique is a parent with children |
| Sibling sub-techniques | `siblings` | When technique is a sub-technique with siblings |

Per technique view, user sees a subset (4-6 typical). Roles section
remains always-visible (chip row, no collapse value).

### Animation auto-pause-on-collapse: still works

Special case handling preserved from v25.7.0.9.2: when the `animation`
section toggles from expanded to collapsed, the `animationPauseSignal`
counter bumps, ProcessAnimation pauses, retains its current stage.
User must click Play to resume after re-expanding.

### Reset behavior: unchanged

Collapse state resets on every technique change — both outer prop
changes (clicking different technique in grid) AND internal
navigation (clicking sibling/parent/child link in sidebar). Each
new technique view starts with everything expanded. Per-session
persistence is still NOT implemented — open sidebar, close, reopen
= fresh state. Add later if useful.

## Honest accounting

Three notes:

1. **The Set-based refactor is a quiet improvement** that pays back
   beyond this release. Future sections we add — guidance content
   when v25.7.0.10+ authoring lands, telemetry sections, version
   history — all just add a `collapsible expanded={isExpanded('foo')}
   onToggleExpand={() => toggleSection('foo')}` to their call site.
   No new state hooks needed.

2. **There's an inconsistency I want to flag:** the empty-state
   "Cross-references" section uses ID `crossrefs-empty`, while the
   populated states use `scenarios` and `beats`. Conceptually the
   empty state is a placeholder for the same content. If a user
   collapses `scenarios`, then internally navigates to a different
   technique that has NO scenarios (showing the empty-state instead),
   the empty-state section opens fresh — because they're different
   IDs. This is correct because the reset effect runs anyway, but
   I want to flag it in case future work changes the reset behavior.

3. **All-collapsed state could become hard to read.** If a user
   collapses every section, the sidebar becomes a wall of section
   headers with no visible content. That's correct behavior (user
   asked for it) but it's worth flagging that some users might
   accidentally end up there and not realize they need to expand
   something. Easy follow-up if it becomes a problem: a small "expand
   all" / "collapse all" button at the top of the sidebar body.
