# v25.7.0.1 — Admin UI for manager assignment (ISS-022)

**Release type:** Hotfix-of-hotfix (single-file UI addition; no schema, no data, no backend)
**Driver:** ISS-022 — v25.7.0 shipped the manager-assignment endpoint without a UI for it; this release surfaces it.
**Trigger:** v25.7.0 deployed successfully but you can't actually populate `:MANAGES` edges without curling the API by hand. Half-day fix to make the team-page flow testable with real data.

## What ships

A new "Manager assignments" tab on the existing `/admin/users` page (admin-only, alongside Users + Invites).

**Two sections in the new tab:**
1. **Unassigned users** — bank members with no manager assignment, with an inline "Assign manager" button that pops a select-and-confirm form
2. **Manager assignments** — every user with the manager role grouped with their direct reports; per-report "Revoke" link

**Backed by existing endpoints** (no server change needed):
- `GET /api/banks/me/members` (existing, v25.7.0)
- `POST /api/banks/me/manages` (existing, v25.7.0)
- `DELETE /api/banks/me/manages/:reportId` (existing, v25.7.0)

## Files changed

- `client/src/pages/AdminUsers.jsx` — added third tab + ~200 lines of `ManagersTab` + `UnassignedRow` + `ManagerCard` components

No server change. No schema change. No data change.

**Indirect benefit.** Once admin assigns managers via this UI, the manager's `/team` page populates with their direct reports. v25.7.0's clickable team table then becomes operationally useful — managers can drill into each report's `/team/:userId` progress page. Without this UI, managers had no direct reports because there was no way to populate the `MANAGES` edge.

## Browser verification

After Railway redeploys:

1. Navigate to `/admin/users` (admin role required)
2. Three tabs visible: **Users**, **Invites**, **Manager assignments**
3. Click **Manager assignments**
4. **Unassigned users section** — should show bank members without a manager (initially: any non-admin/non-manager users). For each, "Assign manager" button works:
   - Click → inline select dropdown appears with all users having the manager role
   - Pick a manager → click Confirm → toast confirms success → row disappears from unassigned
5. **Manager assignments section** — shows every user with the manager role; each has a sub-list of their direct reports with "Revoke" links

If you don't yet have any users with the `manager` role in Auth0, the second section will be empty and the unassigned section will be your only working surface. Assign the manager role in Auth0 first; then the assignments tab becomes useful.

## Honest reflections

This is a deliberately small release because the v25.7.x sequence is going to be big. v25.7.1 (Comments) is meaningfully larger; getting v25.7.0.1 out cleanly first lets us walk a small change end-to-end before committing to the bigger build.

**One UX detail worth noting:** the design currently shows "Unassigned users" first because it's more actionable. If you have a lot of users and most are unassigned, that section can get long. Filtering/sorting is deferred — the table is plain HTML for now. v25.7.x or later if it becomes a usability problem.

**One thing the UI doesn't do (yet):** A user can have multiple managers (the `:MANAGES` edge is many-to-many, not one-to-one). The current UI assigns one at a time, but doesn't surface "this user already has manager X — add Y as well." It just lets you assign again. Server-side `MERGE` is idempotent so multiple clicks for the same (manager, report) tuple don't break anything; they just confirm "already exists." If you want multi-manager visibility in the UI, that's a v25.7.x enhancement — flag it if it matters.
