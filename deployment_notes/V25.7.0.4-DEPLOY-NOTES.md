# v25.7.0.4 — Positioning "Two Views" disguise-reveal visualization

**Release type:** Visualization release. One new component, one updated dispatcher, one updated index, one updated seed file. Same shape as v25.7.0.3; no schema migration required (the existing `migrate:visualizations` script handles the new entry idempotently per the v25.7.0.3.2 fix).

**Driver:** Companion to v25.7.0.3's PositioningTimeline. After two prototype iterations and explicit user direction ("come on Claude, step up your design skills"), the team settled on a "two views" disguise-reveal frame that's structurally different from a timeline. Both visualizations attach to FA0001 and render together when the user expands the tactic. The Framework page's `tacticViz` loop already handles multiple visualizations per tactic, so no Framework.jsx changes are needed.

**Sequence:** v25.7.0.3 (timeline) → v25.7.0.3.1 (OBS-021 fix) → v25.7.0.3.2 (preservation-check formula fix) → **v25.7.0.4 (this release)**.

## What ships

A new visualization kind `two_views`, registered to render the `PositioningTwoViews` component. One example seeded: SC007 Account Rental.

The visual concept ports the prototype that was approved in static + interactive iterations during the v25.7.0.3.3 design loop:

- **Left column (cream banking-CRM dashboard)**: full customer record for Allison Brown — identity header with avatar, risk badge (color-tier reactive), quick-stats row, recent activity table, account settings table. This is "what your bank sees" with no enhanced detection.
- **Right column (charcoal threat panel)**: Money mule actor card at top (flush with Allison's identity block on the left), hidden signals list, detection controls in a 2×2 grid, outcome card below.
- **Interactivity**: toggling a defender control surfaces the matching hidden signal in the bank's view (row lights amber, truth tag appears below the data, the threat-row dims out on the right because it's no longer hidden). Risk score recalculates dynamically as signals surface. Once 3+ signals surface, "Active" → "FREEZE" status, outcome card turns green: "POSITIONING DETECTED."

## How this differs from PositioningTimeline (v25.7.0.3)

Both visualizations cover the same FA0001 tactic but emphasize different teaching axes:

- **Timeline emphasizes WHEN signals appear** — temporal axis, scrubber-driven, "you have N days to catch this"
- **Two Views emphasizes WHAT'S HIDDEN vs WHAT'S VISIBLE** — disguise axis, control-driven, "the truth was always there, detection rules surface it"

Both render. Both telemeter via the existing viz-events plumbing. We can use the per-viz event counts (`viz_opened`, `viz_control_toggled`, etc.) to learn which framing engages users more — that informs v25.7.x decisions on whether to consolidate around one approach.

## The kind name is `two_views`, not `positioning_two_views`

Deliberate choice. The disguise-reveal frame generalizes:

- **Reconnaissance**: "what we know about this customer" (left) vs "what the attacker has compiled from OSINT" (right). Same component shape; different content.
- **Defense Evasion**: "the cover story" (left, looks legitimate) vs "the action being hidden by the cover" (right). Same.
- **Monetization**: "the wire transfers we processed" (left) vs "the cashout structure they're connected to" (right). Same.

By keeping the kind generic, future visualizations for those tactics reuse the same React component with different config, rather than creating four near-identical components. This is the same generalization principle that made the kill-chain-grid component reusable.

## Bank-view color palette

The cream banking-dashboard palette is duplicated INSIDE the component file as a `BANK = {...}` constants object rather than imported from a shared CSS file. Reasons:

- The visualization renders inside the dark editorial app shell, so the cream surface needs to be self-contained
- Other future visualizations (Reconnaissance disguise, Monetization disguise, etc.) using the same aesthetic can `import { BANK } from './PositioningTwoViews.jsx'` — or the constants get promoted to a shared `palettes.js` module when a third visualization adopts them
- Keeping the palette definition next to its single consumer right now means changes to the bank-view aesthetic are local

The palette values are documented in `3fffs-visualization-color-tokens.md` (delivered alongside the v25.7.0.3.3 prototype iteration) and locked.

## OBS-018 (four-lens) for this visualization

### Frontline / Teller

**Material change.** Tellers are in the audience (`roles: ['teller', 'analyst', 'soc', 'executive']`). The disguise-reveal frame is teller-relevant because positioning patterns (e-delivery email change to a free-email domain, mobile device added shortly after open, incoming-only transaction patterns) are observable at the teller window. Seeing a calm "Allison Brown · Active" record on the left and the "Money mule crew" panel on the right that they could have surfaced trains the teller's instinct: "the calm dashboard isn't always the truth." The naive control card (SMS the customer) demonstrates why the obvious response fails when the mule controls the device — important for tellers who might otherwise default to "I'll just SMS them."

### Fraud Analyst

**Material change.** The 2×2 controls grid maps directly to "rule prioritization" — what an analyst would propose to risk committee. The visualization makes the value of each control concrete: not just "this rule fires when X" but "toggling this rule reveals these specific signals out of N hidden ones." The naive control's dead-end response ("Your active controls don't match this attack") prepares the analyst to defend their detection-engineering choices against execs who might want to default to customer-facing alerting.

### SOC / Cyber Team

**Material change.** SOC reads the same right column as a detection-rule SLA exercise. Each control is a deployable rule; "reveals N of 5 hidden signals" is a coverage metric. The risk-score recalc as signals surface ("12 → 30 → 48 → 66 → 84") communicates that detection precision is cumulative — multiple controls work together to drive the aggregate score over a freeze threshold, which is how risk operations actually use multi-rule pipelines.

### Risk Manager / Exec

**Material change.** The outcome card is the exec-relevant output: "Bank view now shows enough truth to freeze the account. 5 hidden signals surfaced. Risk score crossed the freeze threshold. The mule pipeline is interrupted before execution day." This is the language of incident retrospective. An exec can present this narrative to a board: "we had the data, we needed the rules to make it visible." The cream→amber row highlights communicate "we had visibility we didn't use" in a way written reports rarely manage.

## Honest reflections on the v25.7.0.3.x design loop

This release came out of three failed prototype iterations (timeline first, then a "cinematic three-zone stage", then a layout rebuild after explicit feedback that it looked crowded). The final design — a banking-CRM-on-the-left / threat-panel-on-the-right disguise reveal — wasn't my first instinct. It came from the user pushing back against premise-level mistakes I was making (treating Positioning as fundamentally about *time* when it's actually fundamentally about *disguise*).

Lessons logged for future visualization work:

> **OBS-027 (new):** Before building any visualization, articulate the ONE pedagogical insight the visualization is trying to make stick. If you can't write it in one sentence, you don't know what you're building. The right metaphor follows from the insight, not from genre conventions ("interactive viz = animation," "fraud topic = SOC dashboard"). For Positioning the insight is: "the truth was always there; detection rules don't generate it, they surface it." For Reconnaissance it might be: "the attacker compiled this dossier from things you publicly published."

> **OBS-028 (new):** When delivering interactive prototypes, validate JS execution in a stubbed-DOM environment (per OBS-026) AND visually inspect the rendered output (open in headless browser or local file) before delivery. The two-views first iteration shipped with a JS escape-character bug that produced a blank page — caught only after the user reported empty containers. JS parses cleanly is necessary but not sufficient.
