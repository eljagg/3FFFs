# v25.7.0.14.1 — Auto-derived control-stage hint mapping (engine fix; no scene changes)

Audit-driven hotfix. After Omar uploaded the deployed project files
for review, a programmatic integrity check found that 4 of 6
animations had `revealsAtStages` metadata on detection controls that
drifted from the actual stage-to-signal mapping in scene data. The
hint shown when a trainee toggled a control at a "wrong" stage
under-reported where the control would have caught signals.

**No animation behavior is broken.** All controls correctly surface
their signals at the stages where signals are actually mapped. The
drift was in the supplementary hint text — telling trainees "Active
at stage 7" when the control was actually active at stages 7 AND 8,
or "[5, 6]" when the actual list was [5, 6, 7, 8].

This release replaces the hand-authored `revealsAtStages` arrays with
engine-side derivation from the actual scene data — eliminating this
class of bug entirely. Any future scene change that adds a signal to
a stage automatically updates the hint without requiring a parallel
metadata update.

## Pedagogical insight (OBS-027)

**Hand-maintained metadata that mirrors program data drifts.** The
`revealsAtStages` array on each control was a parallel description of
which stages the control fires at — but the actual fact was already
in scene data via signal-to-stage mappings. As stages got tweaked
across releases, the parallel metadata fell behind.

The right engineering move is to derive the metadata from the source
of truth, not maintain a copy. This release demonstrates the pattern
for the engine architecture going forward.

## What changed and why

Three engines (ProcessAnimation, TimelineThresholdAnimation,
MultiActorSequenceAnimation) all had the same pattern:
- `ControlToggle` rendered a stage-feedback hint based on `control.revealsAtStages`
- Scene authors (me) maintained `revealsAtStages` by hand
- 4 of 6 animations had drift between hand-authored and actual stage mapping

The fix in each engine:
1. Added a `useMemo` computing `derivedRevealsAtStagesByControl` from
   the actual stages + signals + controls data
2. Passed the derived list to `ControlToggle` as a new prop
3. `ControlToggle` prefers the derived list when provided; falls back
   to scene metadata only if derived isn't available (defensive only)

## Honest accounting

Three notes:

1. **No user-visible animation behavior was broken before this fix.**
   Controls fired at the correct stages; signals appeared correctly.
   The drift was in the supplementary hint that helps trainees
   navigate. Trainees may have come away with a slightly less
   complete picture of when each control fires, but the actual
   pedagogy was preserved. This is a polish fix, not a correctness fix.

2. **Future animations are immune to this drift.** Going forward, when
   I author scene files I will not write `revealsAtStages` arrays at
   all (or if present, they're inert — the engine ignores them and
   derives from data). Future scene changes propagate correctly to
   the hint.

3. **Why I caught this only now:** Omar uploading the deployed project
   for review prompted a systematic audit I hadn't run before. The
   integrity check that found this drift is now part of my standard
   validation cycle for future releases — running it before
   packaging will catch this class of error in real-time. This is
   the kind of audit that should have been routine from v25.7.0.11.2
   onward when the hint feature shipped.
