# v25.2 Deployment Notes — Header collapse + Try-again + Replay-from-any-stage

**Three related quality-of-life improvements to the scenario play loop.**
None require a migration. The graph schema change is implicit — a new
`attempt` property on the existing `:ATTEMPTED_STAGE` edge type — fully
backwards-compatible.

## What changes

### 1. Collapsible scenario header

The title block (scenario title, severity chip, estimated loss, summary
paragraph, progress bar) now collapses into a compact 60px bar once the
learner is mid-scenario. Frees vertical space so the AttackPath circles
stay visible alongside the active stage panel — no more scrolling up to
check progress.

**Auto-collapse trigger:** the first time the learner answers a stage, OR
the first time `currentStageId` changes to anything past Stage 1, whichever
fires first. Once collapsed, stays collapsed across stage navigation.

**Re-expand:** click anywhere on the collapsed bar (the down-chevron
indicates the affordance) or click the "Collapse" toggle that appears in
the top-right of the expanded header.

**Default state:** expanded on first scenario open so the learner reads
the framing.

### 2. Try-again on wrong answers

**Behavior change:** when the learner gets a stage wrong, auto-advance is
now suppressed. The rationale displays as before, the "RIGHT ANSWER" pill
highlights the correct option, and a new "Try again" button appears
alongside "Continue →". The learner can:

- Click **Try again** — the question unlocks, options re-enable, the
  rationale hides. Original confidence rating stays locked. They can
  re-attempt.
- Click **Continue →** — advance to the next stage. Original wrong answer
  is preserved as the record.

**Right answers** keep the existing 1-second auto-advance behavior — there's
nothing to consolidate by re-doing a question you got right.

**Calibration meaning is preserved:** the confidence rating is set on first
attempt only. Retries do NOT change the calibration record. This is
deliberate — if you could re-rate confidence after seeing the answer, the
metric becomes meaningless.

**Each attempt is recorded as a separate `:ATTEMPTED_STAGE` edge** with an
`attempt` property (1, 2, 3...). Coverage queries can filter on
`r.attempt = 1` to see first-attempt-only stats; future analytics can show
"got it on second try" patterns.

### 3. Replay-from-any-stage on completed scenarios

When a scenario is completed, every stage circle in the AttackPath becomes
clickable. Clicking re-enters that stage with a clean slate (local answer
and confidence cleared, scenario un-completes). Original attempts stay in
the graph as historical record.

The CompletionPanel now displays a hint pointing at the AttackPath:
*"Click any stage circle above to replay it from a clean slate. Your
original answers stay on the record."*

**Branched scenarios behave correctly:** clicking Stage 4 on a completed
scenario takes you to the primary Stage 4 — not to whatever consequence
branch you may have hit on your original run. Clean slate means clean slate.

## Edge cases handled

- **Header collapse during AttackPath hover:** if the user hovers an
  upcoming stage circle while the collapse animation is in flight, the
  hover preview tooltip positions correctly (the AttackPath is in the
  same DOM position regardless of header state — only the height above
  it changes).
- **Try-again then wrong-again:** Each re-attempt is a fresh
  ATTEMPTED_STAGE edge with `attempt: N+1`. The graph keeps the full
  history. The Try-again button reappears. No infinite-loop concern —
  the user is just consolidating.
- **Replay-from-any-stage and consequence branches:** Clicking a primary
  stage on a completed scenario re-enters the primary stage (not the
  consequence branch the user originally hit). If they re-pick the wrong
  option that led to the consequence, the consequence opens as on a
  fresh play.
- **Replay on a still-in-progress scenario:** Not exposed — circles only
  become "all clickable" when `completed === true`. During play, only
  visited stages are clickable, same as v25.1.

## Why no migration script

The schema change is a property addition on the existing
`:ATTEMPTED_STAGE` edge type. Neo4j doesn't require a migration to add
properties to relationships — they're schemaless in that respect.
Existing edges without `.attempt` continue to work; new edges carry the
property; queries that didn't previously read it are unaffected.

If a future feature needs to query "first-attempt only" data including
historical pre-v25.2 edges, the query treats absent `.attempt` as 1:

```cypher
MATCH (u:User)-[r:ATTEMPTED_STAGE]->(st:Stage)
WHERE coalesce(r.attempt, 1) = 1
...
```

That pattern is already used elsewhere for backwards compatibility
(e.g., the original `correct` property had the same handling).

## Rollback

`git revert HEAD && git push` reverts both file changes. The server
falls back to the original MERGE-with-overwrite pattern; new edges
created post-rollback simply overwrite the most recent attempt as
before. Edges already created with `.attempt` properties stay in the
graph — they don't break anything, they're just unread by the
rolled-back code.

## What this does NOT include

- **No analytics surfacing of multi-attempt data.** The graph now records
  the history, but the Coverage page and admin debug pages still show
  first-attempt data only. If you want to add a "stages where you
  improved on retry" view, that's a v25.3+ feature.
- **No mid-scenario replay (yet).** Replay-from-any-stage only works
  after completion. During an active play, only visited stages are
  clickable, same as before.
- **No Practice Mode.** A separate practice surface (unlimited retries,
  no calibration scoring, no scenario-completion semantics) was floated
  as Option 4 during planning. Deferred to v26+.
