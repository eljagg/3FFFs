combined with under-budgeted audio scripts. Both required addressing.

## Pedagogical insight (OBS-027)

**Audio that cuts off mid-sentence is worse than no audio at all.**
A trainee who hears "Mister Henry, this is Officer Reid from the
Financial Investigations" and then nothing learns the wrong lesson
about how vishing calls sound — they sound like a clipped audio
sample, not like a real fraudster speaking. The realism objective
fails. This release puts speech reliability at the same correctness
tier as visual diagram accuracy.

## Diagnosis

**Root cause #1 — `speakMessage` cancelled itself on every call.**
Inside `audioNarration.js`, `speakMessage` opened with:

```js
try { window.speechSynthesis.cancel() } catch (e) { /* noop */ }
```

That cancels any in-flight speech before queuing the new utterance.
Combined with MultiActorSequenceAnimation's slot-distribution timer
that calls `speakMessage` every few seconds across a stage's duration,
**each new message kills the prior one mid-sentence**. The 2-3 word
intro of each line played; the rest got cancelled.

**Root cause #2 — slot-distribution scheduling assumed equal per-message
time budget that text doesn't fit into.** Stage 6 of Vishing has 4 audio
messages and 8.5s. Slot-distribution gives each message 2.1s. But
"Yes, I see — that warning is for protection against external scams.
Since I am FID, not JNCB staff, the warning does not apply. Read me
the six digits." needs ~14s at default TTS rate. Every long message
overran its slot, then got cancelled by the next slot's `speakMessage`.

**Root cause #3 — audio scripts were authored without word-budget
calibration.** Caribbean-banking-realistic dialogue is naturally
verbose. Browser TTS speaks at ~2.3 words/second at the `authority`
profile rate. SIM Swap stage 8 had 103 words to fit in 9 seconds —
required 11.4 words/sec, ~5× faster than possible. Even with
infinite slot time, the audio physically can't fit a stage of that
duration.

## What changed and why

### Architectural fix (`audioNarration.js`)

- **Removed the `cancel()` call from `speakMessage`.** The browser's
  speech synthesis queue is designed to handle sequencing — utterances
  queued while another is speaking play in order, no cancel needed.
  Engines call `stopAll()` explicitly on stage change / unmount /
  mute (which unconditionally cancels everything).
- **Added `onStart` and `onEnd` callback hooks** to `speakMessage`
  options. Engines pass these to track which message is currently
  speaking for the visual speaker-icon cue, replacing the
  unreliable timer-based approach.

### Engine fix (`MultiActorSequenceAnimation.jsx`)

- **Replaced slot-distribution scheduling with sequential queue.**
  Before: spaced `speakMessage` calls across stage duration via
  `setTimeout`. After: single loop that calls `speakMessage` for every
  audio-bearing message at stage entry. Browser handles sequencing.
- **Activeness cue now uses `onstart`/`onend` callbacks** — the speaker
  icon renders next to the message that's currently speaking, in
  real time, regardless of how long the audio takes.
- Removed `audioTimersRef` (no longer used).

## Honest accounting

Three notes:

1. **Total runtimes increased modestly** — Vishing from 62.5s to 81s,
   SIM Swap from 60s to 75s, Phishing from 41s to 47s. This is the
   trade-off for keeping the dialogue audible. Trainees can still
   set 2× speed to get through faster — and at 2× speed the speech
   rate doubles, so the audio still fits the stretched stage at 2×
   timing.

2. **The 2.3 w/s target is conservative for typical browser voices.**
   Some macOS Safari voices speak faster (closer to 2.7 w/s). On
   those browsers the audio might finish 0.5-1s before stage end —
   a brief silent tail rather than a cutoff. That's the right
   direction to bias.

3. **The audio architecture is now sound for v25.7.0.16+.** Future
   animations should author audio by writing the line first, counting
   words, then setting `durationMs` to at least `words / 2.3 * 1000 +
   500ms buffer`. I'll apply this discipline when authoring Account
   Takeover: Password Reset (F1018.001) so it ships with audio that
   works first try.

The fix is architectural + content. Engine queues correctly; scripts
fit their durations. Phase 1 of comprehensive coverage continues with
