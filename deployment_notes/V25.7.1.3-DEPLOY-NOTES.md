# v25.7.1.3 — Certificate verification case-sensitivity hotfix

**Release type:** Server-only one-line patch on top of v25.7.1.2. One file. No client changes, no schema, no migration, no new dependencies.
**Driver:** Discovered during v25.7.1 verification testing. The PDF certificate displays the verification hash in UPPERCASE for readability (`208EA088EA10E33D`), but the hash is stored in Neo4j in lowercase (`208ea088ea10e33d`) because `crypto.digest('hex')` returns lowercase. The verify endpoint did an exact string match on the URL parameter, so any auditor typing the hash from the prominently-displayed readable field got `{valid: false, error: "Unknown certificate hash"}` despite the certificate being real.
**Sequence:** v25.7.0.2.2 → v25.7.1 → v25.7.1.1 → v25.7.1.2 → **v25.7.1.3 (this release)**.

## The bug, in one paragraph

The certificate route generates the hash via `crypto.createHmac('sha256', secret).update(input).digest('hex').slice(0, 16)`, which returns lowercase hex. That lowercase hash is what gets stored on the `CERTIFICATE_ISSUED` edge in Neo4j. The PDF rendering code then displays the hash with `hash.toUpperCase()` for visual clarity in the verification field — so the certificate's readable text shows `208EA088EA10E33D` while the database has `208ea088ea10e33d`. The verify endpoint did `WHERE ce.hash = $hash` (exact match), so a URL like `/api/engagement/verify/208EA088EA10E33D` (the case the user would naturally type from the certificate) returned 404. Lowercase URLs worked fine; uppercase URLs failed silently.

## The fix

One line in `engagement.js`, in the `verifyHandler` function:

```js
// before
const { hash } = req.params

// after
const hash = (req.params.hash || '').toLowerCase()
```

Server normalizes the inbound hash to lowercase before the Cypher lookup. The PDF still displays uppercase for readability — that part doesn't change. This fix means both `/verify/208EA088EA10E33D` and `/verify/208ea088ea10e33d` return the same valid result.

Why this approach rather than the alternative (storing uppercase, displaying uppercase): the database write happens once per certificate; the database read happens N times per certificate (every audit). Normalizing once on the read side is cheaper than re-issuing already-stored certificates, and it keeps the hash format consistent with how `crypto.digest('hex')` natively produces it. The PDF's uppercase display is a presentation choice — the underlying hash is the same string regardless of casing.

## Backward compatibility

Existing certificates issued before this fix are unaffected. The stored hashes were always lowercase. The fix changes only how incoming verification URLs are interpreted. A certificate issued yesterday with hash `208EA088EA10E33D` printed on it still verifies — both the lowercase URL (which always worked) and the uppercase URL (which now also works) resolve to the same database row.

## Honest reflection

This bug should have been caught during v25.7.1 design. The decision to display uppercase for readability was correct UX — but I didn't think through the implication that the auditor's natural action (typing what they see) would mismatch the storage format. The right pattern when designing identifiers that are both human-readable and machine-queried is to normalize at every boundary, not just at the storage boundary. Documented as:

> **OBS-024 (new):** Identifiers that are stored in one canonical form but displayed in a presentation-friendly form must normalize on lookup, not just on write. This applies to: hex hashes (case), email addresses (case + plus-addressing), phone numbers (formatting), invite codes (whitespace, case), and any other identifier the user might transcribe by hand.

Adding this to the convention set so future identifier-display code paths (in v25.7.2+ when we extend this pattern to invite codes or whatever else) get this consideration up front.
