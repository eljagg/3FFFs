## What changed and why

Three corrections after live framework verification (curl results from
your codespace):

### Correction 1: Structuring (F1087) under Monetization (FA0002)

The structuring animation shipped in v25.7.0.11 was registered under
placeholder F-code `F1XXX` because I couldn't query your live
framework from my environment. After your curl, the actual values are:

- **F-code: F1087** (was F1XXX placeholder)
- **Tactic: Monetization (FA0002)** (was Defense Evasion (TA0005))
- **Framework name: "Structuring"** (was "Sub-threshold structuring")

The framework's tactic placement under Monetization is correct —
structuring is a monetization-stage technique (laundering already-
stolen funds). My earlier placement under Defense Evasion was wrong.
The animation specifically explores the sub-threshold structuring
sub-pattern of the broader Structuring (F1087) technique.

### Correction 2: Gather Victim Information (F1067) — framework name fix

The animation I've been calling "OSINT Profiling" is correctly attached
to F1067, but F1067 in your live framework is named "Gather Victim
Information" — that's what trainees see in the framework UI when they
click the technique.

Per the refined pair-codes-with-names convention (OBS-029, refined
v25.7.0.11.1), the metadata now reads "Gather Victim Information
(OSINT Profiling)" — framework name first, animation-scope descriptor
in parens. The animation itself didn't change; only the displayed
technique name did.

### Correction 3: Naming convention refined — human name FIRST, F-code in brackets

OBS-029 [pair-codes-with-names] was supposed to address the readability
problem with bare F-codes ("F1067" requires lookup; "Gather Victim
Information" doesn't). But the convention I wrote put the F-code first:
"F1067 Gather Victim Information." That's still F-code-leading. The
correct format puts the human name first: "Gather Victim Information
(F1067)" — the brain reads the meaningful part first, the cross-
reference second.

Updated everywhere: this deploy notes file, the ANIMATION-TRIAGE.md
top section, all scene-file metadata, the placeholder fallback message
in TechniqueDetailSidebar.jsx, and the locked memory entry for OBS-029
itself (so future Claude sessions get the right format).

## Pedagogical insight (OBS-027)

**Project documentation that contradicts the source-of-truth becomes
a debugging trap.** I was carrying F-codes from my training-data cache
(MITRE ATT&CK and an older F3 reference) and assuming they'd match
your live framework. They didn't. Two of the three F-codes I'd been
citing for months were wrong (F1097 ≠ 3DS Bypass; F1067 has a
different framework name than I'd been using). The placeholder F-code
in v25.7.0.11 was a known-wrong-but-explicit marker; the F1067 +
F1097 misnames were unknown-wrong, which is worse.

The lever this release enables: every F-code reference in the project
is now verified against the live framework or explicitly flagged for
later audit. The lower triage sections of ANIMATION-TRIAGE.md still
contain unverified F-codes from cache, flagged with a warning at the
top of that section.
