## Four-lens block (OBS-018)

- **Teller lens:** Tellers don't directly see IVR traffic, but they DO talk to customers post-call. After watching this animation, a teller who hears "I called the IVR yesterday and my balance was checked, but I didn't make that call" knows what kind of attack that was, and routes accordingly. Pattern recognition outside their primary channel — the animation extends their situational awareness beyond the branch counter.
- **Analyst lens:** Direct relevance. The hidden signals (call velocity, navigation speed, balance-only short calls, ANI churn) are exactly the kind of pattern an analyst would chase if surfaced. The animation makes the pattern visceral instead of just an abstract metric — analyst sees the attacker activity AND the silent monitoring view simultaneously, which is the punch line.
- **SOC lens:** Most direct. Each of the four real toggleable controls IS a deployable IVR-monitoring rule. SOC sees not just "what to detect" but the relative coverage of each control. The naive control demonstrates why the obvious rule ("block after 3 failed auths") fails — attacker spaces failures across spoofed ANIs.
- **Executive lens:** Less direct interaction with the animation, but the Stage-7 headline ("0 alerts during 4 hours of attacker reconnaissance, JMD $760K of accounts mapped") is exactly the kind of data exec wants for "are we exposed?" conversations. Exec doesn't need the toggleable controls — the headline alone is the actionable insight.

## Pedagogical insight (OBS-027)

IVR discovery is invisible because it doesn't generate the events fraud
monitoring is built to detect. No transaction. No agent contact. No
failed digital-channel login. The IVR sees only "a customer checked
their balance" — same shape as 10,000 legitimate calls per day.

The animation teaches by showing the LOUD attacker activity (left zone:
notepad filling, calls placed counter ticking up, activity chip
escalating from PLANNING → MINING → INDUSTRIAL SCALE) alongside the
SILENT bank-monitoring view (right zone: alerts counter stays at 0
through all 7 stages by default, "SILENT" badge visible). The defender
zone's emptiness IS the lesson — that's what makes IVR discovery
dangerous.

The lever this animation exists to enable: training the trainee's
reflex to ask "what if monitoring shows nothing because the attacker
chose a channel monitoring doesn't watch?" — broadly applicable beyond
IVR. This is a meta-skill the rest of the framework reinforces.

## Architecture decisions (locked)

1. **Generic engine + bespoke scenes.** The ProcessAnimation engine
   knows nothing about IVRs specifically. It consumes a scene-data
   object that defines stages, controls, signals, and an IVR menu.
   Future animations are added as new scene-data files plus an entry
   in `ANIMATION_MAP` in TechniqueDetailSidebar.jsx — no engine
   changes needed.

2. **Three-zone canvas: attacker / IVR / defender.** Mirrors the
   existing PositioningTwoViews disguise-reveal frame, extended to
   three zones. Left-to-right reading order = cause-to-effect.
   Defender zone gets cream banking palette? — NO. The IVR (middle)
   gets cream because IT'S the bank's customer-facing surface. The
   defender (right) gets the standard project charcoal because it's
   the "what we're learning" surface.

3. **No autoplay.** Animation lands on Stage 1 paused. User explicitly
   clicks ▶ Play to start. Per pedagogical research: autoplay-once
   animations teach LESS than user-controlled. The big primary "▶
   Play" button is unmissable and signals interactivity without
   demanding attention from a user who landed on the sidebar by
   accident.

4. **Sidebar widens for techniques with animations.** 600px → 820px
   when ANIMATION_MAP[techId] exists. Three zones need room. Sidebar
   without animation stays at 600px. Width transitions smoothly
   (200ms) when navigating laterally between techniques.

5. **Focal-zone signaling.** At any given stage, ONE of the three
   zones is "active" — gets a 2px accent border + slight scale-up
   (1.005×). Other zones recede. Trainee's eye is led to the focal
   activity. This is direct application of Mayer's signaling
   principle from cognitive theory of multimedia learning.

6. **Toggleable detection controls (4 real + 1 naive).** Same 4+1
   pattern from Positioning/Defense Evasion two_views. Toggling a
   control reveals matching signals in the previously-silent
   defender zone, with "would have flagged X of Y calls" callouts.
   The naive control ("block after 3 failed auths") shows "no match"
   — teaches WHY obvious rules fail.

7. **No audio (Q4 confirmed).** Pure visual. Audio waveforms (animated
   SVG bars) substitute for actual sound when calls are active.
   Prompt text bubbles substitute for IVR voice prompts.

8. **Caribbean grounding throughout (Q3 confirmed).** Allison Brown
   character (continues from SC007). JMD amounts. JNCB/Scotia JM IVR
   conventions. +1-876 caller IDs. Half-Way Tree / New Kingston
   geography (in stage descriptions where relevant).

## Known limitations / non-goals

- **One animation only.** F1073 IVR Discovery is the ONLY technique with an animation in this release. Every other technique still shows the placeholder. Future releases add more.
- **No deep-link to specific stage.** You can't link directly to "Stage 4 of F1073." Possible future enhancement.
- **No telemetry.** We don't track which controls users toggle, how far they get in the animation, replay counts, etc. Could be useful data for tuning. Future enhancement.
- **No keyboard shortcuts.** Spacebar-to-pause, arrow keys for prev/next would be nice. Future enhancement.
- **No "share this animation" feature.** Future enhancement if useful.
- **Mobile/narrow viewport.** The 3-zone canvas needs ~720px of horizontal room minimum. On narrow viewports this will look cramped. Not in this release. Same backlog item as v25.7.0.6 narrow-viewport storyboard treatment.
- **Sidebar at 820px on narrow desktop.** On a 1280px laptop the sidebar takes 64% of the screen. Some users may want it narrower. If feedback warrants, we could make this configurable or add a "compact mode" toggle.

## Honest accounting

This is a substantial release — about 1600 lines of new client code,
the most significant frontend addition since v25.7.0.5 (storyboard
ship). A few decisions I want to flag for your review:

1. **The cream banking palette in the IVR zone** is a deliberate
   call-back to PositioningTwoViews where the bank's "customer view"
   gets cream and the "actually happening" gets charcoal. Here the
   IVR IS the bank's customer-facing surface. If on first viewing
   the cream looks off in the dark project palette, easy to
   reconsider — but I think it's pedagogically right.

2. **The "SILENT" badge** in the defender zone when alerts=0 is a
   verbal label for what would otherwise be just an absence. Without
   it, the user might not register that "0 alerts" is the point —
   they might just see a blank panel. The badge makes the silence
   speak.

3. **"Would have flagged X of Y" callouts on active controls** —
   these numbers are stage-specific (e.g., ANI velocity catches 38 of
   50 calls AT STAGE 6, where "50" exists). At stage 1 those numbers
   would be misleading. The callouts only appear on control cards,
   not in the defender zone's signals — so if a user toggles ANI
   velocity at stage 1 they see the callout on the card but the
   signal hasn't been "earned" by the stage yet, so the defender zone
   correctly shows nothing for that signal. This is the right
   behavior but it's subtle — flagging in case you find it confusing
   on first view.

4. **Stage 1 has no IVR activity.** The "— No active call —" empty
   state is intentional (it's the prologue, no calls placed yet) but
   visually it makes the IVR zone look "wrong" until the user hits
   Play. Could pre-populate the menu tree visible-but-greyed during
   stage 1 if this looks broken. Easy follow-up.

5. **Animation total runtime 38.5s** is on the LONG end of what's
   pedagogically optimal. Mayer's research suggests segmenting at
   ~30s. The step-through controls let users break it up themselves,
   which is the standard mitigation. If feedback says it feels too
   long, we could split into "Probe phase" (stages 1-4) and
   "Validation phase" (stages 5-7) sub-sequences.

6. **Future technique animations don't all have to follow this
   3-zone shape.** 3DS Bypass (planned v25.7.0.11) is an MITM flow —
   probably wants attacker / customer / merchant / bank zones, OR a
   sequence-diagram shape entirely. The ProcessAnimation engine as
   shipped DOES enforce 3 zones. When v25.7.0.11 lands, the engine
   may need to evolve to support different zone counts / layouts. I
   shipped this version as "good enough for the IVR shape;
   refactorable when the second animation forces the issue" rather
   than over-engineering for hypothetical needs.

If anything in the foundation feels wrong on first ship — particularly
the cream IVR palette, the no-autoplay decision, the focal-zone
border treatment, or the 38.5s total runtime — easy to iterate. The
data layer is unchanged, the engine is self-contained, backouts are
cheap.
