# v25.7.0.2.2 — Visualization polish hotfix (ISS-023)

**Release type:** Polish hotfix on top of v25.7.0.2. Client-only — no server, no schema, no migration.
**Driver:** Live testing of v25.7.0.2 surfaced four issues: (1) mitigated overlay was effectively transparent, leaving cards looking broken rather than disabled; (2) no visible edges between connected nodes — the "kill chain" read as a grid; (3) state changes snapped without motion, making interactions feel unresponsive; (4) tellers viewing TA0043 saw no signal that the asymmetry was intentional.
**Sequence:** Inserted between v25.7.0.2 and v25.7.1. Comments still slips to v25.7.2.

This release fixes execution, not design. No new visualizations, no new kinds, no new attachments. The pattern stays the same; the rendering finally reads as designed.

---

## Summary

**1. Opaque mitigated overlay (real bug).** The shipping overlay used `rgba(0,0,0,0.04)` — almost invisible. Mitigated cards showed the underlying label and badge bleeding through the shield+text overlay. Replaced with `var(--success-bg)` plus a 2px `backdropFilter` blur fallback. Mitigated cards now read as cleanly disabled rather than visually corrupted.

**2. Visible SVG edges between connected nodes.** The data was always there in `config.edges`; v25.7.0.2 never rendered it. v25.7.0.2.2 layers an absolute-positioned SVG behind the column grid, computes pixel coordinates for each edge from real DOM positions (via refs + `getBoundingClientRect`), redraws on resize, and applies state-aware styling:
- Edges touching the selected node draw in `var(--accent)` at 2px stroke
- Edges touching mitigated nodes draw in `var(--success)` with a dashed pattern
- All other edges fall to `var(--rule-strong)` at 1.25px
- Non-active-path edges dim to 0.25 opacity when a node is selected
- Bezier S-curve geometry (control points at 50% horizontal offset) rather than straight lines, for visual flow

The kill-chain now visibly *is* a chain. Toggling MFA breaks the visible paths into Credential Stuffing and Account Takeover, not just the cards. This was supposed to ship in v25.7.1 — bringing it forward because the static feel of v25.7.0.2 was driven primarily by edge absence.

**3. Motion on state transitions.** All node state changes (select, mitigate, dim, hover) now animate via framer-motion (already a dependency, no new package):
- Selected card scales to 1.02 with a 0.22s easeOut transition
- Hover scales to 1.012 (1.025 if already selected)
- Mitigated overlay fades in via `AnimatePresence` over 0.2s
- Description text expands from height 0 → auto on selection, collapses on deselect
- SVG edge stroke-opacity and stroke-width animate to their target values rather than snapping

The motion is deliberately subtle — 200ms, easeOut, no parallax theater. The point is to make state changes feel caused by the user's action, not to animate for its own sake.

**4. Teller affordance card.** When a tactic has visualizations attached but none target the current role, the right column now renders an "Interactive content for other roles" card instead of collapsing to single-column. The card shows the viz title, a brief explanation that a role-tuned version is on the v25.7.1 backlog, and the audience chips for the other roles. Tellers viewing Reconnaissance now see explicit evidence that the asymmetry is intentional, not a missed deploy.

When the tactic has NO visualizations at any role (the case for TA0042, TA0001, etc. for now), the body still falls back cleanly to single-column. The affordance only fires when the asymmetry is real.

## Interactivity Design (OBS-018)

### Frontline / Teller

**Material change.** TA0043 now renders the "Interactive content for other roles" affordance card instead of collapsing to single-column. The teller can see the viz title, can read why they're not seeing the interactive version, and can see which roles do see it. This is the OBS-018 discipline made user-visible: "no change for this lens" is documented in the UI, not just in deploy notes.

The teller-tuned reconnaissance interaction is still on the v25.7.1 backlog. The affordance card is explicit about that — it cites v25.7.1 by name in the body copy.

### Fraud Analyst

**Material change.** The kill-chain now has visible SVG edges between connected nodes, motion on every state change, and a working mitigated overlay. The chain reads as a chain. Selecting Wire/Transfer Fraud shows visible bezier curves to its parents (Spearphishing, Infrastructure Scanning, Account Enumeration via path); toggling MFA breaks Credential Stuffing and Account Takeover with visible dashed-green edges into them. This is what the v25.7.0.2 spike was supposed to feel like.

### SOC / Cyber Team

**Material change.** Same as analyst — the kill-chain is now legible as a chain rather than three columns of cards. The defense toggles produce visibly broken paths in the SVG layer. For SOC users this is the clearest read: a graph with active defenses on it.

### Risk Manager / Exec

**Material change.** Same as analyst — the falsifiability hook (defenses break the chain) is now visually obvious because the chain has visible edges that go from "live" to "dashed/dimmed" when defenses fire. This frames defense investments as causal in a way the v25.7.0.2 layout did not.

## Honest reflections

**What I should have shipped in v25.7.0.2:** edges and motion. Both were already in scope at architectural level — the data model has `edges`, framer-motion is a dependency. I deferred them to v25.7.1 with the rationale that "the first read should be about whether the pattern works at all, and visible edges are aesthetic rather than functional." That was wrong. Without visible edges, the "kill chain" doesn't read as a kill chain — it reads as a grid, and the user can't tell that toggling defenses breaks anything because there are no visible paths to break.

The opaque-overlay bug is more straightforward — a bad CSS value that I should have caught in light-mode visual review. Lesson: at the bare minimum, run the local dev server and look at every state of every component before delivering. Building cleanly via `vite build` is necessary but not sufficient.

**What I'm not committing to until v25.7.1:** the wider rollout. v25.7.0.2.2 is purely about making the one viz we shipped feel right. v25.7.1 still picks up the multi-tactic fan-out, the new viz kinds (phishing-anatomy, fund-flow-sankey, exec-roi), and the teller-tuned reconnaissance interaction.
