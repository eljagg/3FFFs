# v25.7.1 — Engagement layer (ISS-024)

**Release type:** Feature release. Server + client changes, additive Neo4j schema (3 new edge types, 2 new indexes), one new server dependency (`pdfkit`).
**Driver:** The four highest-priority interactivity recommendations from the April 2026 product-recommendations document — the ones the research identified as outsized retention/credibility wins. Bundled together because they share infrastructure (Home-page restructure, new edge types, certificate template) and because the four together close the full learning loop (acquire → recall → retain → certify) in one go.
**Sequence:** Lands on top of v25.7.0.2.2. ISS-018 (Comments) slips one slot to v25.7.2 (was already slipped from v25.7.1 in the v25.7.0.2.2 notes).

This release deliberately does not touch the visualization layer or the multi-tenant Bank work — those continue as v25.7.0.x lineage. v25.7.1 is the engagement-layer drop.

## Summary

**1. Daily Signal — habit-loop micro-learning on Home.** New widget at the top of Home renders one deterministically-picked stage question per UTC day. The pick is `SHA256(dayKey) % candidateCount`, so every user sees the same Signal on the same day — a Slack chat about "did you get today's Signal?" works. After the user answers, the widget reveals all four rationales (the chosen pick highlighted, the correct one highlighted, the rest dimmed). A streak indicator counts consecutive UTC days with an answer; one missed day breaks it. The streak counter has a one-day grace window — if you haven't answered today yet, the streak still counts up to yesterday.

**2. Review queue — wrong-answer spaced-repetition surface.** New widget below the action cards on Home pulls the user's last 12 unresolved wrong-answer stages (most-recent attempt was wrong AND older than the 5-day cooldown). Each card links to the scenario at the specific stage, so the user can re-attempt without restarting from stage 1. Cooldown is configurable via `?cooldownDays=N` for testing — defaults to 5 days, the point at which the Ebbinghaus forgetting curve flattens. Empty state framing for users with no wrong answers in queue (rather than just hiding the section).

**3. Retrieval practice — post-completion recall check.** When a scenario completes, after a 600ms read-the-success-screen delay, a new panel slides in below `CompletionPanel`. Three "did this signal appear in this scenario?" questions, each with one real signal from the just-completed scenario and three foils sourced server-side from other scenarios. Skippable; failure of the API hides the panel rather than blocking the user. Foils are served already-shuffled so a user can't cheat by inspecting the response payload — correctness is verified server-side via a base64-encoded challenge token round-tripped on submit. Tests the testing effect: actively pulling information out of memory creates substantially more durable learning than passive re-reading.

**4. Printable completion certificate.** New "Download certificate" button on `CompletionPanel` triggers a server-rendered PDF download. Embeds learner name + scenario title + completion date + first 6 F3 techniques covered + a 16-character HMAC-SHA256 verification hash. Public verification endpoint at `/api/engagement/verify/:hash` lets a third-party auditor (BOJ, FATF, JBA) confirm the certificate without app credentials — returns scenario title + issuance date, deliberately does not reveal the email (privacy). PDF rendering uses `pdfkit` (single new server dependency) — landscape Letter, terracotta/cream branded, Helvetica family for portability.

## Schema additions (additive, idempotent)

Three new edge types are introduced. None of the existing nodes or edges are altered. Two indexes back the most frequent query patterns.

```
(:User)-[:DAILY_SIGNAL_ANSWERED {dayKey, correct, optionIndex, answeredAt}]->(:Stage)
   - Composite identity: same user can have many edges to the same stage (one per day)
   - dayKey format: "2026-05-05" (UTC ISO date)
   - Indexed on dayKey for the streak query and the "answered today?" guard

(:User)-[:RETRIEVED {scenarioId, correct, total, answeredAt}]->(:Scenario)
   - One edge per retrieval-practice round (creates a new edge each time, not a merge)
   - Allows future analytics on "how many recall checks has this user done?"
   - No index — write-and-aggregate access pattern, no point lookups needed

(:User)-[:CERTIFICATE_ISSUED {hash, firstIssuedAt, lastIssuedAt}]->(:Scenario)
   - Merged on (user, scenario) — re-downloading produces the same hash, updates lastIssuedAt
   - hash is HMAC-SHA256(secret, "userId|scenarioId|completedAt").slice(0, 16)
   - Indexed on hash for the public /verify/:hash lookup
```

Migration is idempotent — `add-engagement-schema.js` uses `CREATE INDEX ... IF NOT EXISTS`, so re-running is safe. The script also prints a preservation check at the end (edge counts by type) for sanity.

## Interactivity Design (OBS-018)

### Frontline / Teller

**Material change.** Daily Signal widget on Home gives tellers a 30-second daily check-in that fits between customer interactions — the pick is from the same scenario library tellers already see, weighted toward stages flagged for `roles: ['teller', ...]`. Review queue surfaces wrong-answer scenarios at the stage level, so a teller who answered SC005-S2 (Smishing — Initial Access) wrong can re-attempt that specific stage 5 days later without restarting the whole scenario. Retrieval practice after completion uses the testing effect — the highest-impact mechanism for retention in the learning-science literature, especially for staff who only get to engage with the platform in short bursts. Certificate download is the BOJ/audit credibility piece: a teller can show their manager a signed PDF with their name on it after completing each scenario.

### Fraud Analyst

**Material change.** The Review queue is most useful for analysts because they encounter the longer scenarios (SC013/SC014 governance/SOC) where wrong answers compound. The 5-day cooldown matches the analyst's typical weekly review cadence — questions wrong on Monday surface for re-attempt the following Monday. Retrieval practice on the post-completion screen pulls signals from across the scenario, testing whether the analyst can distinguish genuine signals from foils sourced from other scenarios — exactly the discrimination skill that separates pattern-matching analysts from genuinely-trained ones. Daily Signal gives them a habit anchor; certificates feed into their regulatory CPE evidence.

### SOC / Cyber Team

**Material change.** Daily Signal cycles through stages drawn from all scenarios including the SOC-targeted SC014, so SOC users see SOC-relevant questions on roughly the same cadence as analysts see analyst-relevant questions (the deterministic pick is uniform across the candidate pool, not weighted by role yet — that's a v25.7.2+ refinement). Retrieval practice helps SOC consolidate the signal-pattern vocabulary that's central to their job. The verification hash on certificates is the piece that matters most for SOC: the public `/verify/:hash` endpoint means a SOC team member's training records can be confirmed by an auditor without app access, which is a discoverable surface SOC teams tend to value.

### Risk Manager / Exec

**Material change.** The certificate is the primary win at this lens — every scenario completion now produces an audit-evidentiary artifact that exec stakeholders can reference in regulatory conversations. The verification URL on the PDF (e.g., `3fffs.app/verify/a1b2c3d4...`) means a BOJ supervisor can confirm a stack of completed certificates without needing app credentials, which removes a friction point in the audit conversation. Daily Signal gives execs a "did the team show up this week" metric (future addition: aggregate streak coverage by team — slated for v25.7.2+ when manager analytics get the engagement-layer treatment). Review queue and retrieval practice are not directly exec-facing, but their effect on retention is what makes the manager dashboard's coverage metrics meaningful.

## How to verify on Railway after deploy

UI walkthrough — there is also one curl test for the public verification endpoint at the bottom.

1. **Sign in as any user, navigate to `/`.**
   Expected: At the top of the Home content, before the stats grid, a "Daily Signal" panel renders with an eyebrow label, a question pulled from one of the existing scenarios, four answer options, and a streak indicator on the right (showing "Build a streak →" for first-time users).

2. **Click any answer option.**
   Expected: Options lock (no further clicks). The correct option border turns green with a checkmark; if your pick was wrong, your pick gets a red border with an ✕. Both rationales fade in below their respective options ("Why this is right…" / "Why this missed…"). Refreshing the page shows the "you already answered today" calm state with the streak ticker incremented.

3. **Sign in as a user with at least one wrong scenario answer older than 5 days.**
   Expected: Below the three main action cards on Home, a "Review queue" panel renders with up to 12 cards. Each card shows the scenario title (truncated if long), days-since-wrong-answer in accent red, the question text (clamped to 2 lines), and a "Try again →" link. Clicking a card navigates to the scenario at that specific stage via `?reviewStage=:stageId`. For users with no wrong answers past cooldown, an empty-state message renders explaining the spaced-repetition concept.

4. **Complete a scenario.**
   Expected: `CompletionPanel` renders as before with the existing celebration treatment. After a ~600ms beat, a new "Quick recall check" panel slides in below `CompletionPanel`. Three questions, each "Which of these signals appeared in this scenario?" with 4 options. Pick one per question; "Check answers" button activates only when all three are answered. Submission shows green/red banner per question with your pick echoed back. "Skip" button dismisses the panel.

5. **In `CompletionPanel`, click "Download certificate".**
   Expected: Button shows "Generating…" briefly, then a PDF downloads named `3fffs-certificate-{scenarioId}.pdf`. Open the PDF: landscape Letter, terracotta wordmark top-left, "CERTIFICATE OF COMPLETION" eyebrow, "This certifies that [your name]" headline in accent terracotta, scenario title quoted, technique list, completion date, 16-character verification hash, and `Verify at 3fffs.app/verify/{hash}` text. The hash is deterministic for the same `(user, scenario, completedAt)` triple — re-downloading produces the same hash.

6. **curl the public verify endpoint with the hash from step 5:**
   ```bash
   curl https://server-production-7882.up.railway.app/api/engagement/verify/{hash}
   ```
   Expected: `{"valid": true, "scenarioTitle": "...", "scenarioId": "SC...", "issuedAt": <timestamp>}`. No auth required. The email is deliberately not in the response. A bogus hash returns `{"valid": false, "error": "Unknown certificate hash"}` with HTTP 404.

7. **Hard-refresh and confirm the Daily Signal panel re-renders correctly.**
   Expected: After hard-refresh (Cmd-Shift-R / Ctrl-Shift-R), the answered-today state persists — the user sees "Nailed it." or "Got it next time." with the streak indicator. The quiz UI does not re-appear; the streak count is loaded from the server.

8. **Switch theme between light and dark.**
   Expected: All four new components (Daily Signal, Review queue, Retrieval practice, Certificate button) render correctly in both modes. The certificate PDF itself is theme-independent (always renders against white).

### Run the migration after deploy

The new edge types don't need any seed data, but the indexes do. After Railway finishes deploying, run the migration locally against the production Neo4j:

```bash
cd server
npm install            # pulls pdfkit
node --env-file-if-exists=../.env src/seed/add-engagement-schema.js
```

Or, if you've added the npm script to `server/package.json` (this release does):

```bash
cd server
npm install
npm run migrate:engagement-schema
```

✚  Index daily_signal_day_key (DAILY_SIGNAL_ANSWERED.dayKey)
✚  Index certificate_hash (CERTIFICATE_ISSUED.hash)

### One-time Railway env var addition

Set `CERTIFICATE_SECRET` on the Railway **server** service (not the client). Use a 32+ character random string — this is the HMAC key for the certificate verification hash. If unset, the server falls back to a development-only secret (`'dev-only-secret-change-in-prod'`) which is logged on startup as a warning. **For production credibility on certificates, set this env var.** Re-using the same secret across deploys keeps existing hashes valid; rotating it invalidates all previously-issued hashes.

```
CERTIFICATE_SECRET=<paste a 32+-char random string here>
```

You can generate one with: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`.

## Honest reflections

**What I'm confident about:**
- The four features are independently validated by research (recommendations doc) and shared infrastructure was the right way to bundle them. Doing them as one release saves engineering churn versus four small hotfixes.
- The architecture stays consistent with v25.7.0.x — `publicRouter` / `authedRouter` split mirrors the visualization route's pattern; new edges follow the additive-merge convention; migration is idempotent with a preservation check.
- PDF generation works end-to-end. I tested with `pdfkit` in isolation and produced a valid 2.3KB PDF with the same code path that's in the engagement route.

**What I'm watching for in the first 24 hours of usage:**
- The `streakRows` query pulls 60 days of `DAILY_SIGNAL_ANSWERED` per request. For a single user that's tiny, but if the candidate-stage population exceeds ~1000 stages later, the deterministic-pick query (`MATCH (st:Stage)... ORDER BY st.id`) will need to be cached. For now (~40 stages across all scenarios) it's fine.
- Retrieval foil generation needs at least 9 distinct foil signals from other scenarios. With 9 scenarios and ~3 signals per stage, the foil pool is comfortably large — but if you ever ship a single-scenario dev environment, the panel just won't render. The `onSkip` fallback handles this gracefully but it's worth knowing.
- The certificate hash is HMAC-SHA256 truncated to 16 hex characters (64 bits). That's strong enough for "casual auditor verification" but not strong enough to be the only auth mechanism on a high-stakes verification path. If a bank later wants the verification page to show user identity, the hash should grow to 32 characters and the secret must be rotated. v25.7.1's certificate is "good faith verification," not "cryptographically binding."

**What's deliberately not in this release:**
- Per-tenant Daily Signal weighting (so banks can configure which stages cycle for their staff). v25.7.2+.
- Certificate template customisation per Bank (with the bank's logo). v25.7.2+.
- Manager visibility into team Daily Signal streaks. Pairs naturally with the manager-dashboard work in v25.7.0.x lineage — slated for v25.7.3+.
- Progressive review-queue cadence (re-test at 5 days, then 21 days, then 60 days like Anki). v25.7.1 uses a single 5-day cooldown; the progressive cadence needs an additional `lastReviewedAt` tracking edge that I deferred to keep this release focused.
