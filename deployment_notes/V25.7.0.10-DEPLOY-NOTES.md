## Four-lens block (OBS-018)

- **Teller lens:** Tellers handle profiled-fraud victims after the
  fact. Recognizing the LANGUAGE of profiled scammer — "they knew my
  mother's name," "they knew my recent vacation" — tells a teller
  this is profiled fraud (different escalation path than card
  skimming). Animation makes the aggregation viscerally clear: 8
  individually-harmless data points combine into a unique attack
  target.
- **Analyst lens:** Most direct. The Hidden Signals surfacing pattern
  here is about RISK POSTURE rather than activity events — different
  in character from IVR's signals but the same lesson: aggregation
  creates exposure no single data source represents.
- **SOC lens:** This animation surfaces the BANK'S OWN OSINT
  contribution. Stage 6 explicitly shows that Sangster Travel's public
  LinkedIn ("JNCB corporate banking partner") provided the inference
  that let the crew identify Allison's likely bank. SOC controls here
  aren't traditional detection rules — they're attack-surface
  reduction. New surface area for SOC discussion.
- **Executive lens:** Stage 7 final headline is exec-actionable: "$0
  paid to Allison, $4 to a broker, 47 minutes invested, complete
  attack profile, bank's own properties contributed ~30% of the
  dossier." That's a "are we exposed?" data point exec wants.

## Pedagogical insight (OBS-027)

**OSINT profiling is invisible because it leaves no log entry on any
system the target controls.** The crew assembles a complete identity
profile of Allison Brown over weeks — name, DOB, employer, address,
family connections, financial behavior, banking institution — without
ever touching her devices, her bank, or any system she'd notice.

Each individual data source is harmless in isolation. The danger is
the AGGREGATION: when 8 harmless data points combine, they form a
complete attack target dossier.

**The lever this animation enables:** training the trainee's reflex
to think about INFORMATION HYGIENE not as hiding any single data
point (impossible), but as recognizing that aggregation creates risk
no single source represents. Customers should be told this. Bank
staff should think about what the bank's OWN public-facing
communications add to attacker dossiers.

## What ships in this release

### Engine refactor (validates architectural hypothesis)

The v25.7.0.9 ProcessAnimation engine had ZoneAttacker / ZoneIVR /
ZoneDefender subcomponents HARDCODED into the engine. To support
OSINT (which has different middle-zone shapes per stage and different
attacker/target zone vocabulary), the engine had to either grow
IVR-AND-OSINT-aware code (bad) OR become source-agnostic (good).

This release does the second:

- **Engine** (`ProcessAnimation.jsx`) now reads zone configs from
  `scenes.zones.{left,middle,right}` and orchestrates layout +
  ZoneFrame focal signaling only. The engine is now genuinely
  source-agnostic.
- **Scene files** declare their own zone renderers as React functions
  receiving `({ state, scenes, revealedSignals, activeControls,
  controlById })` and returning the inner JSX content. Engine wraps
  the result in ZoneFrame.
- **Engine exports** the visual primitives and styles needed by scene
  files (engineStyles, ZoneFrame, AudioWaveform, CascadeAnimation,
  SurgeAnimation) so scenes don't have to reinvent the wheel.

The ProcessAnimation.jsx engine dropped from ~1380 lines to ~1140
lines. The IVR-specific renderers moved to ivrDiscoveryScenes.jsx.
The new osintProfilingScenes.jsx is ~900 lines (scene data + zone
renderers + 5 bespoke source-platform mockups).

|---|---|---|---|
| 1 | Prologue | A name in a crew's shopping list | 3.5s |

### The 5 OSINT detection controls

Different in CHARACTER from IVR's controls. IVR controls were
detection rules deployable inside the bank. OSINT controls are about
reducing the bank's own footprint and helping customers — because
OSINT activity itself happens on third-party systems the bank can't
monitor.

| # | Control | Type | Reveals |
|---|---|---|---|
| 1 | Reduce employee LinkedIn footprint | Real | Stage 6 (employer-bank inference) |
| 2 | Customer info-hygiene training | Real | Stages 3, 7 (data-point exposure, aggregation) |
| 3 | Breach corpus monitoring | Real | Stage 5 (credential reuse) |
| 4 | Public bank-property minimization | Real | Stage 6 (bank's own OSINT contribution) |
| 5 | Make customer LinkedIn private | NAIVE | (none — bank doesn't have this authority) |

The naive control is also different in character from IVR's. IVR's
naive control was a TECHNICAL workaround failure ("block after 3
failed auths" gets defeated by ANI rotation). OSINT's naive control
is a RESPONSIBILITY ASSIGNMENT error ("blame the customer's privacy
settings" — bank doesn't have this authority and the framing pushes
risk onto victims). Worth teaching as a different KIND of bad
control.

## Deploy

⚠️ **One extra step for this release:** the rename from
`ivrDiscoveryScenes.js` → `ivrDiscoveryScenes.jsx` means the old `.js`
must be deleted manually after unzipping. `unzip -o` only writes new
files; it doesn't delete old ones. The deploy workflow gets ONE extra
line for this release only:

```bash
cd /workspaces/3FFFs
unzip -o v25.7.0.10-3FFFs.zip
rm v25.7.0.10-3FFFs.zip
rm -f client/src/components/framework/animations/ivrDiscoveryScenes.js  # one-time cleanup of renamed file
git add -A
git status
git commit -m "v25.7.0.10: F1067 OSINT Profiling animation + engine refactor for scene-driven zone rendering"
git push origin main
```

Note: the rename will appear in `git status` as a delete + add of the
file, OR git may detect the rename automatically. Either way, both
states are correct after the commit.

## Honest accounting

This release is the largest since v25.7.0.9 itself. ~1000 lines net
new code (osintProfilingScenes.jsx) plus ~250 lines moved between
files for the refactor. Several decisions worth flagging:

1. **The engine refactor was scoped tightly.** I considered going
   further (extracting all primitives into a separate module, adding
   a TypeScript-style scene contract, etc.) and chose not to. The
   v25.7.0.10 refactor is just enough to support TWO animations
   cleanly. v25.7.0.11 (different shape) may force more refactoring,
   and that's fine — we'll do it then with concrete information.

2. **The OSINT controls' character is different from IVR's** — flagged
   in the four-lens block. IVR controls = detection rules. OSINT
   controls = posture + customer hygiene. Same UI pattern; different
   meaning. If feedback says "the OSINT controls feel weaker than
   IVR's" that's actually correct — OSINT defense IS weaker because
   the bank doesn't control the channels. The control list reflects
   reality. Worth surfacing in any pedagogical guidance later.

3. **Naive control's politicality.** "Make customer LinkedIn private"
   is more politically loaded than IVR's "block after 3 failed
   auths." It's a real strawman that real bank policies have
   contained (customer-blame language in fraud responses). Surfacing
   it as NAIVE is intentional pedagogy. If it lands wrong on first
   viewing — looks like the framework is being preachy — easy to
   reword to something more technical-sounding while keeping the same
   responsibility-assignment lesson.

4. **The 5 source-platform mockups in the middle zone** are bespoke
   SVG/JSX — each has its own styling (~80-150 lines of styles each).
   This is real authoring work that doesn't generalize as a primitive.
   Future animations with similar source-platform-shape needs would
   either copy this pattern or the patterns get factored into shared
   primitives later. v25.7.0.11 will tell us which.

5. **Two stages have empty middle zone** (stage 1 prologue, stage 7
   handoff). These are deliberate breathing-room moments in the
   narrative. If they look "wrong" or "broken" on first viewing, the
   captions should make their intent clear, but worth flagging.

6. **ANIMATION-TRIAGE.md is a NEW top-level file** at the repo root
   (not in a docs/ subfolder). I added it there because it's a
   project-level methodology record, equal weight to a README. If
   you'd rather it live somewhere else (docs/, .github/, server/),
   easy to move.
