# v25.7.0.3.2 — Preservation-check formula on add-visualizations.js

**Release type:** Server-only one-line hotfix on top of v25.7.0.3.1. One file. Migration script logic fix only.
**Driver:** After v25.7.0.3.1 fixed the OBS-021 Cypher syntax, re-running the migration produced the correct end-state in Neo4j (both visualizations seeded, both edges created — `visualizations=2 edges=2` confirmed in the script output) but then the preservation-check assertion at the end of the script failed. The check expected delta=0 on any re-run, but the seed array had grown between runs, so the legitimate creation of `VIZ-POSITIONING-TIMELINE` was flagged as a "preservation violation."

## Important: the actual migration succeeded

Look at the v25.7.0.3.1 script output: `visualizations=2  edges=2`. Both visualizations are in Neo4j right now. Both `VISUALIZES` edges exist. The migration **worked**. What failed is the post-migration sanity assertion that's overly strict for the seed-array-growth case.

This is a v25.7.0.3 verification path issue, not an actual data integrity issue. The user can navigate to F3 Framework → expand FA0001 Positioning right now and the visualization should already render. The assertion failure just makes the script return exit code 1, which CI/CD might treat as a failed deploy — that's the only practical impact.

## The bug

```js
// Before (wrong — assumes re-runs always have delta 0)
const expectedVizDelta = (vizExisted > 0)
  ? 0
  : VISUALIZATIONS.length

// After (correct — accounts for seed-array growth)
const expectedVizDelta = VISUALIZATIONS.length - vizExisted
```

The corrected formula works for three cases:
- **First run:** `N − 0 = +N` (creates all N visualizations)
- **Idempotent re-run with same seed:** `N − N = 0` (nothing new to create)
- **Re-run after seed-array growth:** `N − M = new` (creates only the new entries)

The previous logic only handled cases 1 and 2.

## Honest reflection

**Two consecutive hotfixes on the same migration script in one afternoon is a bad signal.** v25.7.0.3.1 (OBS-021 syntax) was a real bug. v25.7.0.3.2 (this assertion) is also a real bug, but a more embarrassing one — I should have noticed when shipping v25.7.0.3 that adding a second visualization to the seed array would break the assertion that was written when only one visualization existed.

**Pattern to add to the pre-ship checklist:** Whenever a release adds entries to a seeded data array (visualizations, scenarios, etc.) that has a preservation-check pattern, verify that the assertion's formula accounts for both first-run and seed-growth cases. The simplest correct formula is almost always `total_in_seed − total_pre_existing`, not `0` or `seed.length`.

**Adding to OBS-021 family:**
> **OBS-025 (new):** Preservation checks in idempotent migration scripts must compute `expectedDelta = seedArray.length − preExistingCount`, not assume 0 for re-runs. The "always 0 on re-run" formula breaks the moment the seed array grows.
