# v25.7.0.6.1 — Storyboard stub release labels hotfix

**Cause:** When v25.7.0.5 originally shipped, the stub picker tabs were
labeled with their then-planned release numbers: SC008→v25.7.0.6,
SC011→v25.7.0.7, SC013→v25.7.0.8. v25.7.0.6 ended up shipping the
system-wide layout work + storyboard integration instead, pushing every
scenario authoring release one slot later. The stub labels are now stale.

**What this fixes:** Bumps the stub `plannedRelease` strings by one in
both files where they live:
- `server/src/seed/data/scenario-beats.js` (canonical source)
- `server/src/routes/storyboard.js` (denormalized for cheap rendering)

| Stub | Was | Now |
|---|---|---|
| SC008 ATM Skimming | v25.7.0.6 | v25.7.0.7 |
| SC011 Romance Scam | v25.7.0.7 | v25.7.0.8 |
| SC013 Wire Fraud | v25.7.0.8 | v25.7.0.9 |

## Files

- `server/src/seed/data/scenario-beats.js` — STORYBOARD_STUBS const + comments
- `server/src/routes/storyboard.js` — STUBS const

Client unchanged. Migration NOT needed (stubs are served from the route
file, which reads from the in-memory const, not the DB).

## Deploy

```bash
cd /workspaces/3FFFs
unzip -o v25.7.0.6.1-3FFFs.zip
rm v25.7.0.6.1-3FFFs.zip
git add -A
git status
git commit -m "v25.7.0.6.1: bump storyboard stub release labels (v25.7.0.6→7, 7→8, 8→9)"

```

After Railway redeploys server, hard-refresh and hover over any stub tab
in the storyboard picker. Tooltip should now read "Coming in v25.7.0.7"
(SC008), "v25.7.0.8" (SC011), or "v25.7.0.9" (SC013).
