# v25.5.1.1 Deployment Notes — Hotfix: scenarios /:id regression (ISS-011)

**Production regression fix.** The v25.5.1 query that joined
MitreTechnique inline broke EVERY scenario detail response, not just
SC013. SC001-SC012 also returned `{"error":"Internal server error"}`
when fetched. Users could not play any scenarios while v25.5.1 was
live.

## What was wrong

v25.5.1 added a fourth OPTIONAL MATCH to the existing `/:id` query:

```cypher
OPTIONAL MATCH (st)-[:USES_TECHNIQUE]->(tech:Technique)
OPTIONAL MATCH (st)-[:USES_MITRE_TECHNIQUE]->(mitre:MitreTechnique)
WITH s, st, tech, mitre ORDER BY st.order
RETURN s { .* } AS scenario,
       collect({
         stage: st { .* },
         technique: tech { .id, .name },
         mitreTechnique: mitre { .id, .name }
       }) AS stages
```

The exact failure mode hasn't been confirmed via Railway logs yet, but
the regression affects all scenarios uniformly — meaning the bug is in
the query shape itself, not in any particular scenario's data. Most
likely cause: projecting `.id, .name` on a null MitreTechnique node
(which happens for every stage in SC001-SC012 because none of them
have USES_MITRE_TECHNIQUE edges) is a shape Neo4j or the driver
doesn't tolerate when there are already null projections in the same
collect.

## What's fixed

Reverted to the v25.4.2.1 query shape (which works in production),
then added a SECOND query that fetches MITRE wedges keyed by stage ID.
Two round-trips per scenario detail request, but eliminates the
join-shape risk entirely:

```cypher
-- Query 1 (unchanged from v25.4.2.1):
MATCH (s:Scenario {id: $id})
OPTIONAL MATCH (s)-[:HAS_STAGE]->(st:Stage)
OPTIONAL MATCH (st)-[:USES_TECHNIQUE]->(tech:Technique)
WITH s, st, tech ORDER BY st.order
RETURN s { .* } AS scenario,
       collect({ stage: st { .* }, technique: tech { .id, .name } }) AS stages

-- Query 2 (new, narrow):
MATCH (s:Scenario {id: $id})-[:HAS_STAGE]->(st:Stage)-[:USES_MITRE_TECHNIQUE]->(mt:MitreTechnique)
RETURN st.id AS stageId, mt.id AS mitreId, mt.name AS mitreName
```

The second query only returns rows for stages that actually have a
MITRE edge. The JS side builds a map from stageId → mitreTechnique and
merges it into the parsed stages payload. Stages without a MITRE edge
get `mitreTechnique: null`.

Performance: the second query is small (typically 0-3 rows per
scenario, max 9 for SC013). Cost is one extra round-trip to Aura,
which adds ~15-30ms to scenario detail latency. Acceptable for the
robustness gain.

## Lessons (carrying forward)

- **OBS-014 (new) — multi-OPTIONAL-MATCH plus null-projection plus
  collect is the danger zone.** When extending an existing aggregating
  query to add a new OPTIONAL MATCH branch, prefer a separate query
  over chained joins. The join-shape risk in Cypher aggregation is
  real and not always caught at static-analysis or test time.
- **OBS-015 (new) — verify the curl smoke test against an EXISTING
  scenario, not just the new one.** I had v25.5.1's deploy notes
  testing only SC013 via curl. If I'd included SC010 in the smoke
  tests, this regression would have been caught at curl-test time
  before any user opened the app. Going forward: every change to
  scenario plumbing gets curled against at least one existing
  scenario, not just the new one.
- **v25.5.1's deploy notes will be amended in retrospect** to flag
  this as the canonical example of why "test old as well as new" is
  permanent practice now. SC013 itself was unaffected by the bug
  shape — the bug was in the query that ran for every scenario.

## Rollback

If this hotfix introduces its own problem (it shouldn't — it's reverting
to the known-good shape plus a small narrow query), rolling back
returns to the broken state. Better to roll forward with another
hotfix. But:

```bash

```

reverts to v25.5.1 — the broken state. Don't do this unless v25.5.1.1
itself is broken.
