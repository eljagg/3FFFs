# v25.3.1 Deployment Notes — Role visibility fix (ISS-005)

**Resolves the issue where the role selector at signup appeared to produce
identical experiences across all four job-function roles.** Role context is
now visible in the top nav, on the Scenarios page, and persisted to the
graph for audit. Admins get a "simulate as" mechanism to test what each
role sees without leaving their admin session.

This is the v25.3 follow-up; ISS-005 in the tracker.

## What changes

### 1. Top-nav role pill

A small monospace pill (`ROLE: ANALYST` / `ROLE: TELLER` / `ROLE: ALL` for
admins not simulating) sits between the theme toggle and the user menu.
Click it to open the RoleSwitcher dropdown.

- **Non-admin:** dropdown lets them change their actual role
- **Admin:** dropdown lets them *simulate* a role (sessionStorage-only, doesn't
  affect their actual admin role); a "Stop simulating" entry appears once a
  simulation is active

When simulation is active, the pill turns accent-colored and shows
`ROLE: ANALYST · SIM` so admins always know they're not in normal admin view.

### 2. Banner on the Scenarios page

Above the scenarios grid, a context banner shows three states:

- Non-admin: *"Showing scenarios for **Fraud Analyst**. [Change role ▾]"*
- Admin (not simulating): *"Admin view — showing all 12 scenarios. [Filter by role ▾]"*
- Admin (simulating): *"Simulating as **Frontline / teller** — 6 scenarios visible. [↺ Stop simulating]"*

Both dropdowns let users / admins switch roles directly.

### 3. AASE arc opened to analyst role

SC010, SC011, and SC012 were tagged `roles: ['admin', 'manager']` only —
analysts couldn't see them. They're now `['admin', 'manager', 'analyst']`,
matching where the audience for AASE governance content actually sits.

The seed data files are updated AND a new tiny migration
(`migrate:update-aase-roles`) updates the existing scenario nodes in the
graph.

### 4. Server-side `simulateRole` query parameter

`/api/scenarios?simulateRole=analyst` filters as if the requester were a
non-admin analyst — overriding the admin auto-bypass when present. Used by
the admin "Filter by role" / "Simulate as" UI.

### 5. Role persistence to User node + audit history

Until v25.3.1, the user's job-function role was localStorage-only — invisible
to the server, no audit trail, re-prompted every time the user cleared
cookies or switched device. v25.3.1 adds:

- `POST /api/me/role` — persists role to `User.role` and appends to
  `User.roleHistory` array with `{role, at}` timestamps
- `chooseRole()` in client UserContext now fires this endpoint
- Existing localStorage caching is preserved so first-paint stays fast
- `User.roleUpdatedAt` timestamp on every change

This establishes the entire server-side role layer for the first time.

### 6. Defensive show-all when role is empty

A user (non-admin) with no role property at all now sees the full scenario
list — was previously empty list, which was confusing. This is defensive,
shouldn't trigger in normal flow because the RolePicker wraps protected
pages, but covers edge cases (cleared localStorage, race conditions, etc.).

Open the Debug page (admin route) and inspect your User node. You should

## Edge cases handled

- **Empty role + non-admin** — defaults to show-all instead of empty list
- **simulateRole survives navigation but not browser restart** — sessionStorage
  not localStorage, so admins don't get stuck in simulation mode tomorrow
- **Server persistence is fire-and-forget** — UI doesn't block on the
  `setMyRole` call; if the server is down, localStorage still keeps the
  client working
- **roleHistory entries are JSON-string-encoded** — Neo4j arrays must be
  primitive-typed; we serialize to strings and parse on read if needed

## What this fixes (ISS-005)

The user reported: "After logging in, I selected Frontline/teller, Fraud
analyst, Risk manager/exec. For each of those sections, the layout looks
and feels the same."

Root cause: the user is an admin. Admin role bypassed the role filter, so
all four selections returned all 12 scenarios. v25.3.1:
1. Makes the bypass visible (the role pill always shows current state)
2. Lets admins opt into the filter (simulate-as mechanism)
3. Establishes the audit/persistence layer (User.role + roleHistory)
4. Surfaces the filter context in the UI (banner)

ISS-005 moves to **Shipped** and will move to **Done** once user confirms
in production.

## What this does NOT include

- **Quizzes are still role-agnostic.** Login wizard's promise of "we'll
  tailor scenarios *and quizzes*" is half-true. Quiz tailoring is in
  v25.4-v25.5.
- **Tutor responses are still role-agnostic.** Tutor knows your role (it's
  in stage context) but doesn't tailor language. Also v25.4-v25.5.
- **Home page is still role-agnostic.** No role-tailored "next up for you"
  content. Deferred to v26.
- **Scenario-to-role tags for SC001-SC009** were not retroactively cleaned
  up. OBS-006 in the tracker — they're inconsistent (SC003 includes teller
  but not executive; SC006 includes executive but not teller, etc.). A
  future content-cleanup pass should reconcile.

## Rollback

`git revert HEAD && git push` reverts code. The graph changes are minor:
- The `roles` arrays on SC010-SC012 stay updated (no harm, just unused if
  client reverts to v25.3 logic which would still respect those arrays)
- The `User.role` / `User.roleHistory` properties stay (won't be read by
  reverted client code, just orphaned data)

If you want to clean those up:

```cypher
// Revert AASE arc role arrays
MATCH (sc:Scenario) WHERE sc.id IN ['SC010', 'SC011', 'SC012']
SET sc.roles = ['admin', 'manager'];

// Remove role audit data
MATCH (u:User)
REMOVE u.role, u.roleHistory, u.roleUpdatedAt;
```

Both blocks are safe and idempotent.
