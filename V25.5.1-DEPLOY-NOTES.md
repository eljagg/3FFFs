# v25.5.1 Deployment Notes — SC013 governance scenario (ISS-009 part 2)

**First user-facing CBEST scenario, and first user-facing consumer of
the v25.5 MITRE ATT&CK foundation.**

The analyst plays the security manager at "Northgate Apparel plc" — a
fictional UK retailer — who's just been told by the Bank of England
that Northgate is in CBEST scope. Six primary stages walk through the
four-phase CBEST process: kickoff posture (1.1 Launch), critical-
function selection (1.3 Scoping), CREST-accredited provider procurement
(1.4 Procurement), Targeting Report validation (2.3 Validation), the
Test/Halt decision moment during live execution (3.2 Execution), and
Board remediation prioritisation (4.2 Remediation).

Three consequence branches surface on the most catastrophic wrong
answers — defensive-framing fallout, non-CREST procurement rejection,
and stand-down-on-real-intrusion fallout. (Originally planned for two
consequence branches; the kickoff stage produced a strong third one
worth keeping.)

Two stages — Stage 4 (Targeting Report Validation) and Stage 5
(Test/Halt during Execution) — also carry MITRE technique references
(T1566.004 spearphishing voice and T1219 remote access tools
respectively). These are the FIRST stages in the platform that have
USES_MITRE_TECHNIQUE edges; they activate the MitreTechniqueSidebar
shipped in v25.5 and prove the foundation works end-to-end.

## Source-material caveat (carried forward from planning)

SC013 is authored from the Exabeam CBEST 2.0 solution brief plus
general public CBEST knowledge. The Bank of England's authoritative
CBEST Implementation Guide is the canonical source; v25.6 freshness
pass should verify procedural detail against that document and update
as needed. The four-phase shape, party structure (SCT, Firm/FMI, TI
provider, PT provider, Regulator), and CREST accreditation requirement
are public knowledge and reflected accurately. Specific procedural
details (e.g. the exact wording of an SCT supervisory note) are
plausible but not authoritative. Vintage marker on the scenario data
makes this explicit.

## What's new

### 1 new Concept: CREST Accreditation

`CONCEPT-CREST-ACCREDITATION`. Universal: false (CBEST-specific). Has
APPEARS_IN_FRAMEWORK edges to all four frameworks; CBEST is populated
with a CBEST-2.0-vintage summary, the other three are marked
`pending: true` per the v25.4.2 pattern. CBEST analysts opening this
concept's sidebar see real content; AASE/TIBER-EU/iCAST analysts see
the "v25.6 content pending" placeholder.

## Edge cases handled

- **Cypher post-aggregation scope** (OBS-011 lesson). The new MITRE
  join in scenarios.js `/:id` query keeps the WITH ordering before
  the projection — no trailing ORDER BY past the collect.
- **Stage field name** (OBS-012 lesson). All stage seed entries use
  `heading`, not `title`. The migration validates this implicitly:
  the verification query reads heading.
- **Idempotency.** Re-running the migration sets the same properties
  on the same nodes. Preservation check verifies no unexpected
  collateral changes.
- **Mixed concept seedling.** SC013 stages reference both AASE-vintage
  concepts (Critical Function, Test/Halt Model, Threat Matrix,
  Escalation Path) and the new CBEST-vintage CREST concept. The
  per-framework ConceptSidebar handles both correctly.

## What's NOT in v25.5.1

- **No CBEST FrameworkPhase wiring per stage.** SC010 stages link to
  `:FrameworkPhase` nodes via `[:IN_AASE_PHASE]`. SC013 doesn't,
  because CBEST FrameworkPhase nodes don't exist in the graph yet
  (v25.0 only seeded AASE phases). This is v25.6 freshness pass
  territory. SC013 stages have a `framework` property at the scenario
  level (`'CBEST'`) which is sufficient to identify them; the per-
  stage phase mapping waits for v25.6.
- **No SC014.** That's v25.5.2 — the defensive-tactical SOC-tier
  scenario based on the source documentation. ~5-6 days of
  authoring + migration; coming next per Sequence A.
- **No quizzes for SC013.** Quizzes are role-agnostic (OBS-009)
  and would be a separate authoring effort.

## Rollback

`git revert HEAD && git push` reverts code. Graph cleanup if needed:

```cypher
// Remove SC013 and all its stages + edges
MATCH (sc:Scenario {id: 'SC013'})
OPTIONAL MATCH (sc)-[:HAS_STAGE]->(st:Stage)
DETACH DELETE sc, st;

// Remove the new CREST concept (only if no other scenarios use it —
// SC013 is the only one as of v25.5.1, so safe)
MATCH (k:Concept {id: 'CONCEPT-CREST-ACCREDITATION'})
DETACH DELETE k;
```

Both blocks idempotent. SC001-SC012, MitreTechnique nodes, and
everything else unaffected.
