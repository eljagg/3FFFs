# v25.7.0.6 — System-wide layout consistency + storyboard integration

Two parallel workstreams in one release:

- **Option A (system-wide widening):** Widens 8 pages from the 1100px default
  to the 1500px wide layout that F3 Framework adopted in v25.7.0.4.6. Other
  pages stay narrow on purpose (see "Pages NOT widened" below).
- **Option B (storyboard integration):** Promotes the Design C scenario
  storyboard out of the F3 Framework page where it's been buried since
  v25.7.0.5. Now also accessible from the Scenarios page (as a view toggle)
  and from each individual Scenario walkthrough page (as a collapsible
  "Read the case file" section).

## Four-lens block (OBS-018)

- **Teller lens:** A teller no longer experiences page-width whiplash
  jumping between Scenarios, F3 Framework, and Coverage. All three are
  now wide. The teller can also discover the storyboard directly on the
  Scenarios page (where they go for scenario content) instead of having
  to drill through F3 Framework to find it.
- **Analyst lens:** Same width consistency benefit. Plus, when an analyst
  opens an individual scenario for the decision walkthrough, they now see
  a "Read the case file" affordance that gives them the day-by-day
  narrative as a complement. Different mental modes for different moments
  — sometimes you want to play through the decisions, sometimes you want
  to read the case as a story.
- **SOC lens:** Wider Matrix and Coverage = scan more techniques /
  coverage cells per viewport. Wider Admin pages = data tables stop
  truncating. Storyboard now accessible from any scenario page = SOC
  analysts can pull up the counterfactual (e.g. "if Auto-freeze had been
  active...") without leaving the scenario context.
- **Executive lens:** Wider Coverage = whole program posture in one
  glance. Wider Team = manager dashboard scales to larger teams without
  vertical scroll. Storyboard available on Scenarios.jsx as the default
  exec entry point — exec sees the headline + stats card, doesn't need
  to drill to find it.

## Pedagogical insight (OBS-027) for this release

The visual consistency this release ships isn't decorative — it's
epistemic. A user who navigates Scenarios → F3 Framework → My Coverage
right now experiences three different page widths and gets implicit
messages they don't realize they're receiving: "this is more important /
less important / different kind of content." Standardizing wide-where-
data-rich removes that noise. The user's attention stays on CONTENT
instead of subconscious "wait, why does this page feel different?" friction.

For Option B specifically: putting the storyboard on the Scenarios page
is the same insight at a different scale. The storyboard is content about
scenarios, so it belongs on the Scenarios page, not buried under F3
Framework as the only access point. Same for Scenario.jsx — when a user
is mid-walkthrough on SC007, they shouldn't have to navigate away to
find the day-by-day timeline of the same scenario.

## Storyboard integration (Option B)

### On `Scenarios.jsx` (the gallery page)

New view toggle at the top of the page: **Cards** (default, preserves
existing UX) vs **Storyboard** (renders the Design C component with full
picker).

The toggle is only rendered if at least one scenario has authored beats.
It's not always visible — when no scenarios have beats (e.g. fresh DB
without the v25.7.0.5 migration run), the page looks identical to before.

When user picks **Storyboard**, the cards section unmounts and the
`<ScenarioStoryboard>` component renders in its place. The component
keeps its full picker (SC007 active, SC008/SC011/SC013 stubs). When user
picks **Cards** again, the gallery comes back.

### On `Scenario.jsx` (individual scenario page)

New collapsible section between `<ScenarioArt />` and `<AttackPath />`:

```
[ScenarioArt — existing]

▶ READ THE CASE FILE
  Day-by-day timeline of this scenario
  A narrative complement to the decision walkthrough below.
  ...

[AttackPath — existing]
[StagePanel — existing decision walkthrough]
```

Default state: collapsed. Click to expand and reveal the storyboard for
the current scenario. Picker is hidden (the page is already scoped to a
specific scenario).

Only renders for scenarios where `getStoryboardScenarios()` reports the
current scenario has `hasBeats: true`. Currently that's SC007 only.
SC008/SC011/SC013 will get this affordance automatically as their beats
are authored in v25.7.0.7-9.

### `ScenarioStoryboard.jsx` component change

Added a `hidePicker` boolean prop. When true, the picker tabs at the top
don't render. Used by Scenario.jsx where the picker would be redundant
noise (the page is already scoped). The default `hidePicker=false`
preserves existing behaviour for the F3 Framework page and Scenarios
page Storyboard toggle.

## Known limitations / non-goals

- **Storyboard not integrated on Coverage / Home.** Reasoned out and
  deferred. Coverage as "celebrate completed scenarios with their
  storyboards" is a v1+ feature. Home as "feature this week's case study"
  is a marketing decision, not a data decision.
- **Scenarios.jsx view-toggle state is per-session, not persisted.**
  Refreshing the page returns to Cards view. Could persist to localStorage
  in a future polish release if user feedback wants it.
- **Scenario.jsx storyboard section default-collapsed.** Some users may
  want it default-expanded for SC007 specifically since SC007 is the
  showcase. Easy change if requested.
- **No responsive breakpoints added.** Same project convention as before
  — desktop-first, narrow-viewport handling is backlogged.

## Honest accounting

This release is mostly mechanical (8 pages get the same single-line
change) plus two careful integrations. Two specific moments where I had
to think hard:

1. **Wrapping the cards-view content in Scenarios.jsx** in a fragment
   conditional. The temptation was to wrap each piece (progress summary,
   grid, empty state) in its own `view === 'cards' &&`. I picked the
   single-fragment wrap because it makes the toggle behavior unambiguous:
   one mode renders the storyboard, the other renders everything that
   was there before. Easier to reason about, easier to back out if needed.

2. **Adding `hidePicker` to ScenarioStoryboard** rather than building a
   second component for the embedded case. Two-way street: future contexts
   that embed the storyboard will probably want similar control points.
   Better to add knobs incrementally to one component than fork the
   component each time.
