# v25.6.0.1 — add-sc013.js preservation check hotfix

**Release type:** Hotfix-of-hotfix (single-file migration script change; no schema, no data, no client)
**Driver:** ISS-016 — `add-sc013.js` preservation check is not idempotent-aware
**Trigger:** v25.6.0 deploy of SC013 properties hit the half-implemented preservation logic and exited non-zero. The data writes had already committed; the script just declared FAILED on the post-write count comparison.

## What was wrong

`add-sc013.js` had two preservation-check expected-deltas hardcoded to fresh-seed values:

```js
// Before (v25.5.1 — broken on re-run)
const expectedDeltas = {
  Scenario: sc013Pre[0].c === 0 ? 1 : 0,  // correctly idempotent-aware
  Stage: 9,                                 // hardcoded
  Concept: 1,                               // hardcoded
}
```

On v25.6.0 re-run (refreshing properties on existing nodes), actual deltas were `0` for all three. Expected `9` and `1` for Stage and Concept. Script declared FAILED and exited non-zero.

The data writes (concept ensure, all 9 stage MERGE+SETs, edges) **committed independently before the preservation check ran** — Neo4j `runQuery` is per-statement transaction. The v25.6.0 SC013 properties are already in the production graph; the migration script just couldn't acknowledge it.

## What was fixed

Backported the canonical idempotent-aware preservation pattern from `add-sc014.js` (v25.5.2). The preservation check now:

```js
// After (v25.6.0.1 — idempotent-aware)
const expectedDeltas = sc013Existed
  ? { Scenario: 0, Stage: 0, Concept: 0 }
  : { Scenario: 1, Stage: 9, Concept: 1 }
```

Plus the same three-way validation pattern used in SC014:
1. Labels expected to change → validate the delta matches
2. Labels NOT expected to change → must be unchanged
3. New labels appearing → must not happen

Plus dual success message: "(idempotent re-run; no count changes expected, none observed)" vs the fresh-seed delta listing.

Cosmetic fix included: Step 1 console message "CREST Accreditation concept..." → "CBEST Accreditation concept..." for consistency with the v25.6.0 concept rename.

## Honest reflection

This is exactly the failure mode OBS-017 warns about: "when a hotfix doesn't fix anything, that's evidence the diagnosis was wrong, not a deeper bug — step back, don't escalate." We didn't escalate; we stepped back. The diagnosis was correct (v25.6.0 successfully changed the data); the bug was in adjacent code that v25.6.0 didn't touch.

Two things to learn from:

**1.** The retroactive preservation backport was on the v25.6.x list explicitly — but listed under "future work after v25.6.0 ships." It should have been listed as a v25.6.0 *prerequisite* because v25.6.0 is the first release that re-runs migrate:sc013 against an existing graph. Pure timing accident — the issue was foreseeable and was foreseen, just slotted in the wrong order.

**2.** The pattern of "writes commit before preservation check" is a feature, not a bug — it means a failed preservation check doesn't roll back the data. But it also means a non-zero exit doesn't tell you whether the data is bad. Always confirm via curl + verification query before assuming the migration "didn't work."
