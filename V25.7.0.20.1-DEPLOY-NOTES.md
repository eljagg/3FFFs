# v25.7.0.20.1 — Fix F1018/F1018.001 navigation routing

Hotfix release. v25.7.0.20 shipped the F1018 (Account Takeover parent
canonical) animation correctly, but the F3 framework grid's
collapseToSingle behavior — written when F1018 had no animation of
its own — caused F1018.001's card to display correctly but routed
clicks to the F1018 sidebar instead of F1018.001's. Symptom: clicking
F1018.001 opened Karelle Bryan's triage animation, not Andre Lewis's
password reset kill chain.

This release removes the collapseToSingle override of click target and
expand affordance, making card identity always equal to parent.id and
expandability depend only on having sub-techniques.

## Pedagogical insight (OBS-027 — methodological)

**A UX collapse heuristic written for one assumption breaks silently
when the assumption changes.** The v25.7.0.8 implementation of
collapseToSingle was written when single-child parents in the F3
framework were always pure organizational shells — the parent
existed to organize the sub-technique, the sub-technique was the
"real" thing the user cared about. Collapsing into one card was
correct UX.

When F1018 got its own independent animation in v25.7.0.20, the
assumption broke: F1018 is no longer a shell, it's a first-class
pedagogical artifact (the defender-perspective triage scenario).
The collapse heuristic kept treating it as a shell and routed
clicks accordingly, hiding F1018 from navigation entirely while
displaying F1018.001's identity on a card that didn't actually
open F1018.001's sidebar.

Going forward, any UX heuristic with the shape "if X has only one
Y, treat X as a shell" should be revisited whenever X gains its
own surface area beyond Y. The right replacement is explicit
configuration ("this technique is a shell" as a property), not
heuristic detection ("this technique has one sub").

## Diagnosis

The bug was in `client/src/components/framework/TechniquesTree.jsx`,
in the `TechniqueParentCard` subcomponent. Two coupled problems:

**Problem 1: effectiveTargetId override.** Lines 137-145 (pre-fix)
overrode the displayed-and-clicked technique ID for collapseToSingle
nodes:

```js
const effectiveTargetId = collapseToSingle && subTechniques[0]
  ? subTechniques[0].id    // F1018.001 displayed AND clicked
  : parent.id              // F1018 displayed AND clicked
```

This was correct UX when F1018 was a shell. With F1018 having its
own animation, no path remained to navigate TO F1018 from the grid.
The displayed F1018.001 card was the only entry point, and the
click resolved to one or the other depending on tree shape — your
deployed instance routed to F1018, leaving F1018.001 unreachable.

**Problem 2: expandable depended on collapseToSingle.** Line 133
(pre-fix):

```js
const expandable = !isAtomic && !collapseToSingle
```

This hid the expand chevron on collapseToSingle nodes, so even if
the user wanted to drill into F1018.001 from a hypothetically-fixed
F1018 card, they couldn't — there was no expand affordance.

## The fix

Three small changes in `TechniquesTree.jsx`:

1. **`effectiveTargetId = parent.id` always.** Card identity always
   matches parent.id. No override.
2. **`effectiveTechName = parent.name` always.** Same reasoning.
3. **`effectiveDesc = parent.description` always.** Same reasoning.
4. **`expandable = !isAtomic`** (removed `&& !collapseToSingle`).
   Single-child parents now show the expand chevron and reveal
   their child as a chip — same UX as multi-child parents.
5. **Atomic badge condition simplified** to just `isAtomic`
   (removed the redundant `&& !collapseToSingle`).

Result: F1018 appears as a card displaying "F1018 — Account Takeover"
with an expand affordance. Click expand → F1018.001 chip appears
below. Click F1018.001 chip → opens F1018.001 sidebar. Click parent
card (without expanding) → opens F1018 sidebar. Both navigations
available.

If the server tree shape includes F1018.001 as an orphan-promoted
parent (which the diagnostic suggests was happening in your live
instance), F1018.001 also appears as its own atomic card. Either
way, both techniques are reachable.

## Trade-off

Any future single-child parent that is genuinely a pure organizational
shell will now render as its own card rather than collapsing into the
child. None currently exist in the F3 grid besides F1018, and F1018
is no longer a shell. If a future case arises, the right replacement
is explicit per-technique configuration (a `isShell: true` property
on the Technique node, set in the seed) rather than restoring the
"subs.length === 1" heuristic.

## Honest accounting

Three notes:

1. **I burned cycles on diagnosis.** Two full turns of code-reading
   without conclusive root cause, before settling on a defensive fix
   that addresses the symptom rather than fully proving the cause.
   The symptom-fix approach is the right call given the cost of
   continued investigation, but the methodological lesson is: when
   diagnosis isn't converging within 1-2 turns, a defensive fix that
   makes the failure mode impossible regardless of the underlying
   cause is often a better use of cycles than continued investigation.

2. **The collapse heuristic was load-bearing for the wrong reason.**
   v25.7.0.8 introduced collapseToSingle as a UX nicety. v25.7.0.16
   and v25.7.0.20 both shipped F1018-family content without
   reconsidering whether the heuristic still fit. The methodology
   lock from v25.7.0.18.1 ("verify the technique is in the grid
   before authoring") needs an addition: "AND verify the navigation
   path actually opens the right sidebar." A future Phase 2 release
   could plausibly hit a similar coupled-UX-heuristic issue if a
   parent technique gains its own animation while keeping a sub-
   technique that already had one. Locked methodology going forward.

3. **No content changes in this release.** All animations, scene
   files, audio, and seed migrations from v25.7.0.20 carry forward
   unchanged. This is purely a UX routing fix.
